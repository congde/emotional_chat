# 第 13 章 从个人助手到团队级研发流程

前面几个场景章，讨论的是个人如何把 Codex 用稳：怎样拆任务，怎样写项目说明书，怎样设置边界，怎样把一次有效经验沉淀成可复用能力。到了第 13 章，问题开始变化：当同一套方法进入团队后，第二个人能不能起手，第三个人能不能接力，值班同事能不能在半夜看懂当前状态？

很多团队第一次遇到的瓶颈，并不是 Codex 不够强，而是个人经验没有变成团队默认路径。最开始，只有一两个人高频使用 Codex，他们知道该先看哪里、哪些脚本能跑、哪些目录别碰、什么动作必须等 owner 点头。可一旦更多同事加入，这些“脑内规则”就会开始丢失：review 里反复解释背景，异步接力时重新补上下文，值班时没人说得清哪些动作可以远程做、哪些必须停下来确认。

到这个阶段，问题已经不再是工具熟练度，而是团队有没有共享边界、共享能力和共享交接方式。因此，本章不会把个人开发者叙事突然切换成一套厚重的企业治理手册。它只回答一个更现实的问题：什么时候值得把个人提效升级成团队流程？一旦决定升级，团队应该先共享什么、先治理什么，又该克制什么？

接下来，我们会沿着这条判断往下看：先判断什么时候值得团队化，再讨论哪些规则、权限和能力资产必须共享，最后回到小团队最现实的落地顺序。真正重要的不是把流程铺满，而是让一条任务主线可以被第二个人、第三个人和值班同事稳定接住。

## 13.1 什么时候值得团队化

不是所有团队都应该一上来就把 Codex 从个人助手升级成团队流程。很多时候，单人阶段还没跑出稳定闭环，团队化只会把个人混乱复制成团队混乱。真正值得团队化的信号，从来不是“我们人变多了”或者“这个工具很火”，而是某些重复任务、共享边界和协作摩擦已经开始持续出现。

更现实地说，团队化不是购买一个更贵档位，而是决定要不要把个人经验升级成团队资产。只要这件事还没有明显收益，太早推进只会增加规则维护和协作成本。

### 13.1.1 不看人数，先看三项条件

判断是否值得团队化，最实用的方式不是先看团队规模，而是先看任务、规则和协作是否已经具备共享条件。表 13-1 把这三个维度放在一起，它不是成熟度评分表，而是一个启动前的判断工具。

**表 13-1：判断是否值得团队化的三维**

| 维度 | 你在观察什么 | 值高时意味着什么 |
| --- | --- | --- |
| 任务重复度 | 同类任务是不是反复出现 | 共享模板和能力包开始有价值 |
| 规则稳定性 | 边界、验证、交付方式是否已比较固定 | 规则写下来后不容易过期 |
| 协作密度 | 多人是否经常接力、评审、值守 | 共享流程和 handoff 开始必要 |

如果这三项里有两项长期偏低，团队化就不宜过早启动。团队人数增加，并不天然意味着流程已经具备共享条件；只有当任务类型、验证入口和协作边界逐渐稳定，个人经验才有可能被沉淀为团队资产。在这之前，更重要的仍然是先把个人工作流和项目基础说明打稳。

### 13.1.2 当默认知识不再可靠

个人使用时，很多东西可以存在脑子里：你知道该先看哪里、哪些目录别动、什么测试值得先跑、什么情况要停下来。但团队一旦开始协作，这些默认知识不再可靠。只要每个人都在各自驯化一套 Codex，用法就会越来越分裂。团队化真正要解决的，正是这种分裂。

可以把“值不值得团队化”理解成一个问题：团队是否已经需要共享默认工作方式？表 13-2 列出的不是工具能力，而是协作摩擦。一旦这些信号反复出现，团队化就开始有现实收益，因为它减少的不是某一次操作时间，而是反复解释、反复对齐和反复接力的成本。

**表 13-2：团队已经需要共享默认工作方式的信号**

| 如果团队现在经常出现 | 说明什么 |
| --- | --- |
| 同样的问题每个人都在重复讲一遍 | 共享规则缺失 |
| 新人或其他同事接手时效率骤降 | 共享上下文缺失 |
| review 经常卡在“为什么会这样做” | 共享交付格式缺失 |
| 值班或异步接力总要重新解释背景 | 共享 handoff 缺失 |

示例 13-1 把这个变化过程放进一个小团队的日常里。它要说明的是：团队化通常不是从组织架构变化开始，而是从默认知识开始丢失时发生。

**示例 13-1：从个人提效到共享流程**

一个小组里，最开始只有其中一个人高频用 Codex，帮自己做代码勘察、补测试、整理 PR 摘要，效果不错。第二个人也想沿用这套方式，却发现一半能力来自“你自己知道该怎么起手”。等到第三个人开始接手任务，新的问题又出现了：大家对哪些目录别动、哪些命令先跑、哪些事要等 owner 点头的理解并不一致。

到这个阶段，问题已经不是“工具好不好用”，而是“团队是不是需要共享边界和共享动作”。这就是团队化真正开始发生的地方。

### 13.1.3 先试点，而不是立刻全面升级

很多团队不需要一次性全面升级，只需要先判断是否值得试点。表 13-3 把“可以启动”和“应该暂缓”放在一起，是为了避免把团队化理解成一个只能前进、不能收缩的动作。真正稳的试点，既要看见机会，也要看见前提还不够的地方。

**表 13-3：团队化试点的启动与暂缓信号**

| 观察到的情况 | 更适合怎么判断 |
| --- | --- |
| 同类任务高频重复，且边界逐渐稳定 | 可以开始抽模板和能力包 |
| 仓库已有最小验证入口 | 可以把“完成”的标准共享出来 |
| 已有稳定使用者和明确维护者 | 可以从现成经验里提炼共性 |
| 已出现异步接力或值班需求 | 可以优先试点 handoff 和 runbook |
| 个人闭环还没跑稳 | 先把最小工作流打通 |
| 仓库没有稳定脚本和验证入口 | 先补项目 readiness |
| 任务类型太散，没有高频共性 | 先观察，不急着沉淀 |
| 团队没人愿意维护共享规则 | 先不要把文档和能力包堆起来 |

这张表的重点不是给团队打分，而是帮助团队把“想升级”的冲动拆成更具体的判断：哪些地方已经可以共享，哪些地方还应该先补基础。如果暂缓信号仍然占主导，团队化大概率会变成“文档和能力包建了，但没有人真用”。

### 13.1.4 团队化不是一步到位，而是从一段共享主线开始

很多人一想到团队化，就默认要覆盖需求、开发、测试、发布、值班的全链路。对大多数小团队来说，这样起手太重。更稳的路径通常是：先找一段已经高频、可重复、可验证的主线，比如“接手问题 -> 改代码 -> 跑最小验证 -> 输出交付摘要”。先把这段主线共享起来，再慢慢往两端扩展。

团队化的收益可以很具体。表 13-4 不是在证明团队化必然值得，而是帮助团队把收益落到可观察的协作变化上。只要其中有两项正在明显拖慢团队，就已经足够构成试点理由。

**表 13-4：团队化能换来的具体收益**

| 收益 | 体现在哪里 |
| --- | --- |
| 新人更快起手 | 入口、模板、禁区已经共享 |
| review 更快对齐 | 输出格式和验证标准一致 |
| 异步接力更顺 | handoff、runbook、值班边界明确 |
| 自动化更容易升级 | 因为先有统一规则，再有统一执行 |

因此，本节的核心不是鼓励所有人立刻团队化，而是帮助读者看清一个边界：当个人经验已经开始因为多人协作而反复丢失、重复解释、难以接力时，团队化才真正变成高回报动作。下一节会继续往下落，讨论一旦决定开始，哪些共享规则、权限边界和能力资产必须一起设计，而不是分开补丁式堆上去。

## 13.2 共享规则、权限和能力包

团队化的核心，从来不是“每个人都能用上同一个模型”，而是“每个人和每个 Agent 都沿着同一套边界工作”。只要共享约束没有建立起来，团队里的 Codex 使用方式就会越来越像个人定制：A 同学靠自己的 prompt，B 同学靠自己的脚本，C 同学把经验记在脑子里，最终谁也接不住谁。

所以，团队化真正先要共享的，不是技巧，而是默认约束。

### 13.2.1 共享资产至少有五类

一个小团队想把个人经验升级成团队流程，通常至少要沉淀五类资产。表 13-5 把这些资产和维护角色放在一起，是为了强调：共享资产不能只看“有没有”，还要看“有没有人负责让它继续可信”。

**表 13-5：团队共享的五类基础资产**

| 资产类型 | 作用 | 谁最适合维护 |
| --- | --- | --- |
| 项目说明书 | 入口、禁区、常用命令、交付格式 | 仓库 owner |
| 规则与审批 | 哪些动作默认允许，哪些必须升级 | 模块 owner / 团队维护者 |
| 能力包 | 高频任务的 Skill、模板、脚本 | 高频使用者 + 维护者 |
| 知识库 / runbook | 常见问题、排障路径、值班动作 | 业务 owner / on-call owner |
| handoff / playbook | 接力格式、升级条件、事故流程 | 团队共同约定 |

只要缺其中一类，协作就会开始断层。比如 Skill 有了，但没有 runbook，值班场景仍然接不住；规则有了，但没有 handoff，异步协作仍然不顺。

### 13.2.2 权限和能力包最好一起设计

很多团队在这里会犯一个非常典型的错：先到处接能力，再回头想权限怎么收。结果就是系统会在最早期就暴露过大的动作面。更稳的顺序恰好相反。你应该先问“哪些角色需要哪些能力”，再决定“这些能力以什么形式开放”。也就是说，权限设计要先于能力扩张。

表 13-6 从角色出发给出一张最小能力表。它最重要的价值是：能力不会脱离责任边界单独扩张。

**表 13-6：不同角色默认该拿到的能力**

| 角色 | 默认该拿到什么能力 | 不该默认拿到什么 |
| --- | --- | --- |
| 日常开发 | 代码读取、工作区写入、最小测试、文档更新 | 生产写操作、密钥修改 |
| Reviewer | 读 diff、读验证、读风险说明 | 直接替作者改生产配置 |
| On-call | 日志查看、状态检查、故障复现、只读查询 | 生产删除、权限变更 |
| Owner | 高风险审批、最终放行 | 无需把所有日常操作都集中到 owner |

### 13.2.3 统一默认路径，保留个人偏好

团队化后最容易出现的坏味道，是每个人都说“我这套自己用着挺顺”。从个体角度看，这很自然；从团队角度看，这意味着经验很难迁移。共享规则和共享能力包存在的意义，就是把个人最有效的那部分经验抽成集体可用接口，而不是要求所有人用一模一样的语气。

表 13-7 区分了真正该统一的内容。反过来，像摘要风格、输出语气、个人常用快捷方式，这些通常可以留给个人偏好层，不必强行统一。

**表 13-7：团队最该统一的四类内容**

| 该统一什么 | 为什么 |
| --- | --- |
| 交付格式 | review 和 handoff 才能稳定接住 |
| 最小验证入口 | “做完算完成”才有共同答案 |
| 敏感路径和审批点 | 风险边界不能各人理解各异 |
| 常用 Skill / 模板入口 | 团队资产才能被复用 |

如果要把这些资产落进仓库，最好给它们稳定位置。代码清单 13-1 给出一个够用的目录布局，它不是为了好看，而是为了让“规则、能力、操作手册、自动化入口”有固定归宿。路径一旦稳定，共享就容易很多。

**代码清单 13-1：团队共享资产的目录布局**

```text
.
|-- AGENTS.md
|-- .codex/
|   `-- skills/
|-- docs/
|   |-- runbooks/
|   |-- playbooks/
|   `-- handoff-templates/
|-- scripts/
|-- .github/
|   `-- workflows/
`-- tests/
```

### 13.2.4 给资产写清责任人，也写清成本边界

共享资产最怕一种状态：大家都能改，但没人维护。尤其是 Skill、runbook 和审批规则这三类内容，一旦没有明确 owner，很快就会过期。表 13-8 把最低责任要求列出来，是为了让团队在建立资产时就同步建立维护关系。

**表 13-8：共享资产与最低责任要求**

| 资产 | 最低责任要求 |
| --- | --- |
| AGENTS.md | 谁维护根规则，多久回看一次 |
| Skill | 谁负责更新步骤和输入输出 |
| runbook | 谁负责确认故障路径仍然有效 |
| 审批策略 | 谁可以修改默认门槛 |
| 成本规则 | 谁负责预算、模型档位和长任务使用边界 |

团队级使用和个人使用最大的差别之一，是成本开始变成团队问题。只要多个成员都在跑长任务、并行子任务和自动化，消耗就不再是个人感受，而是预算和治理议题。因此，成本边界也应该成为团队资产的一部分：哪些自动化默认开启，哪些高耗任务要先说明，预算上限如何处理，模型档位如何选择，都需要有团队默认值。

### 13.2.5 知识库、runbook 和 playbook 是能力包的后半段

很多人一谈能力包，只想到 Skill 和工具接入。对团队来说，这还不够。真正让团队持续复用的，往往是后半段资产：知识库告诉你背景，runbook 告诉你怎么排查，playbook 告诉你在特定场景下怎么协作。没有这三类，Skill 再多，也只能解决局部动作，接不住更长的流程。

表 13-9 对这三类资产做一个简要分工。它们一旦和 Skill、规则、审批一起出现，团队级使用才真正从“个人提效扩散”走向“受控协作系统”。

**表 13-9：知识库、runbook 与 playbook 的分工**

| 资产 | 更像回答什么问题 |
| --- | --- |
| 知识库 | 这是什么系统，它怎么运作 |
| runbook | 现在出问题了，该先看哪里 |
| playbook | 某类流程或事件发生时，团队怎么协作 |

本节的结论可以压缩成一句话：团队化的核心不是让每个人都学会同一套 prompt，而是让规则、权限、能力包、知识库和交接协议共同构成一条共享默认路径。只要这条路径存在，个人经验才有机会真正变成团队资产。

## 13.3 Worktrees、云端任务与异步协作

团队一旦开始多人使用 Codex，很快就会遇到一个新问题：同一条任务主线，不再总是由同一个人、同一个时间段、同一个工作目录完成。有人白天定义任务，有人晚上 review，有人离开工位后只在手机上处理审批，有些长任务还需要云端自己继续跑。这时，Worktrees、云端任务和异步协作就不再是可选技巧，而是把团队节奏串起来的基础设施。

这三者分别解决三种不同的问题：Worktrees 解决并行隔离，云端任务解决离开本机后的持续执行，异步协作解决“同一件事如何被不同人和不同 Agent 接力”。理解清楚这一点，团队就不会把手机值守、远程续跑和本地并行混成一种能力。

### 13.3.1 先画清一条团队级任务主线

在讨论具体工具之前，先把一条团队级任务主线画出来会更清楚。图 13-1 的重点不在于它是否覆盖了所有研发环节，而在于它明确了哪些阶段适合同步工作，哪些阶段适合异步推进，哪些阶段必须留下 handoff。

**图 13-1：团队级工作流主线**

[drawio 源文件：图13-1_团队级工作流主线.drawio](figures/图13-1_团队级工作流主线.drawio)

一旦任务主线被画清楚，团队就能讨论每一段该由谁接住、默认验证是什么、哪些地方必须停下来等人确认。没有这条主线，Worktrees、云端任务和值班动作很容易各自变成零散技巧。

### 13.3.2 Worktrees 负责并行隔离，不负责替团队决策

只要一个团队开始并行处理多个问题，或者同一问题开始尝试两种方案，Worktrees 就会变得非常有价值。它的核心不是 Git 技巧，而是任务隔离。你不能一边让 Agent 并行推进，一边让所有改动都挤在同一个工作目录里。

表 13-10 介绍了几类适合使用 worktree 的团队场景。它要表达的是：worktree 把“并行”从概念变成物理隔离，每一组改动都知道自己属于哪条任务线。

**表 13-10：适合用 worktree 的团队场景**

| 场景 | 更适合怎么用 worktree |
| --- | --- |
| 两个人并行试两种修复 | 每种方案一个独立副本 |
| 一个长任务拆成前后端两路 | 各自单独 worktree，最后汇总 |
| review 或 hotfix 插队 | 单开临时副本，不污染主线任务 |
| 试点能力包或规则调整 | 在独立副本验证，再决定是否合入主线 |

不过，worktree 只能解决隔离，不能替团队做判断。它不会自动告诉你哪条方案更值得合并，也不会替你承担审批责任。团队仍然需要在交付格式、验证标准和 owner 放行上保持一致。

### 13.3.3 云端任务适合长跑，值班动作要先扩大观测面

云端任务最有价值的地方，不是“任何事都搬到远程”，而是承接那些已经边界清楚、但需要长时间等待或持续执行的动作。比如长测试、批量分析、夜间 review、远程续跑。这些任务如果强行绑在本机上，不仅浪费本地环境，还会和人的时间表互相打架。

表 13-11 给出一个简单判断：哪些任务更适合放到云端，哪些任务最好先留在本地切清楚。它尤其提醒最后一类情况：只要依赖特殊本机环境或登录态，就不要急着转云端。

**表 13-11：适合放到云端的信号**

| 信号 | 更像适合云端吗 |
| --- | --- |
| 任务目标已锁住 | 是 |
| 本机状态依赖很少 | 是 |
| 需要较长等待时间 | 是 |
| 中途只需少量追问 | 是 |
| 依赖特殊本机环境或登录态 | 否 |

团队级使用时，还要把云端任务和值班动作区分开。第 12 章已经讨论过 PR、CI/CD 自动化和人工审批的边界；到了团队级流程里，还要再多区分一层：值班动作并不等于全面远程执行。表 13-12 用默认建议划出一个更稳的边界：远程和值班最先应该放权的是信息收集和计划准备，而不是直接深度执行。

**表 13-12：值班动作与默认建议**

| 值班动作 | 默认建议 |
| --- | --- |
| 查日志、查状态、查 runbook | 可以远程做 |
| 整理故障摘要、准备修复计划 | 可以远程做 |
| 生产写操作、权限变更、删除数据 | 默认不要直接远程做 |
| 真正修复动作 | 视审批和权限链而定 |

团队化后，很多人会想把 SSH、云 CLI、服务器日志抓取能力也接进来。这些能力确实很有价值，但更稳的方式通常不是开放为默认任意动作，而是以受控能力或 runbook 入口提供。比如允许读取某类日志、允许查询某类状态、允许准备诊断命令，但真正执行生产修复时仍然要走审批链。

换句话说，团队级异步协作里，远程能力最先应该扩大的是“观测面”，不是“破坏面”。

### 13.3.4 异步协作的前提，是 handoff 可读

很多团队以为“有聊天记录”就等于支持异步协作。实际完全不够。异步协作真正依赖的，是 handoff 质量。没有清楚 handoff，下一位接手者就得重读大量历史上下文，效率还不如从头做。

代码清单 13-2 给出一个最小 handoff 模板。它不追求复杂，而是确保下一位接手者能立刻知道目标、状态、变更范围、验证结果和下一步。

**代码清单 13-2：异步 handoff 的最小模板**

```text
当前目标：
- 这条任务主线现在在解决什么

当前状态：
- 已完成什么
- 卡在什么

变更范围：
- 改了哪些文件
- 哪些地方明确没动

验证：
- 跑了什么
- 结果如何

下一步建议：
- 下一位最该先做什么
```

只要这五项齐，异步接力成本会明显下降。对两到五人的小团队来说，这一点尤其重要，因为小团队经常同时承担开发、review、值班和发布，人的时间高度碎片化。只要 handoff 模板清楚、worktree 隔离稳定、长任务能云端续跑，小团队对“谁在什么时间接住这条任务”会明显更从容。

当 Worktrees、云端任务和异步 handoff 都开始稳定工作后，团队就会自然从“有人 review 一次”进入“整条任务主线可以被接力”的状态。也就是说，review 不再只是终点，而是任务在多个阶段都能被重新接住的一部分。一个任务从 ticket 到 release 再到 incident，可以不断留下可供下一阶段读取的上下文，而不是每一段都重来。

这才是异步协作真正值得的地方。它不是让每个人都随时在线，而是让一条任务主线在不同时刻、不同设备、不同角色之间移动时，依然保持可追踪、可解释、可接力。

## 13.4 小团队试点落地顺序

真正把 Codex 从个人提效升级成团队流程时，最重要的不是“一次上全”，而是落地顺序。顺序一错，团队通常会出现两种问题：要么还没打稳共享规则，就急着上自动化和复杂接入；要么为了求稳把流程铺得过厚，结果第一批试点成员很快就绕开它。

对小团队来说，更稳的办法通常是从一个项目、一条流程、一个小组开始。先跑出能被他人接住的共享主线，再逐步补自动化、能力包和更深的治理层。

### 13.4.1 一条够用的试点顺序

表 13-13 把小团队落地压成五个阶段。它的重点是：每一步都建立在前一步已经稳定的前提上。

**表 13-13：小团队试点的五个阶段**

| 阶段 | 最先做什么 | 通过标准 |
| --- | --- | --- |
| S1 | 选一个高频项目和一条高频任务主线 | 任务边界和验证入口说得清 |
| S2 | 写共享说明书、最小规则和交付格式 | 第二个人能按文档起手 |
| S3 | 沉淀一两个高频 Skill / 模板 | 同类任务不再反复口头解释 |
| S4 | 把 review、handoff、异步接力接进流程 | 任务能在多人间稳定交接 |
| S5 | 再考虑自动化、云端续跑、更多能力接入 | 不是靠单个高手维持 |

自动化很诱人，因为它最容易带来“流程看起来升级了”的感觉。但如果共享规则本身还不清楚，自动化只会把混乱自动化。比如团队还没统一交付格式，就开始让 bot 自动 review；或者审批边界还没定好，就开始接远程写操作。这样做往往不是加速，而是把冲突前移。

### 13.4.2 先证明三件事，再升级自动化

试点顺序里真正先要证明的，通常是下面三件事。表 13-14 的作用，是把“先别急着自动化”落到可检查的前提上。只要这三项没稳，后面的自动化大概率都不值得急着上。

**表 13-14：试点前必须先证明的三件事**

| 先证明什么 | 为什么 |
| --- | --- |
| 大家对任务边界理解一致 | 不然自动化只会放大误解 |
| 最小验证入口稳定 | 不然结果无法比较 |
| handoff 模板真的有人在用 | 不然异步协作只是口头说说 |

试点不是宣言，而是局部验证。最适合做第一批试点的，往往是：任务类型重复、仓库边界比较清楚、团队关系紧密、有人愿意维护规则的小组。这样的小组更容易在短期内产出看得见的正反馈，也更容易在问题出现时快速调整。

反过来说，如果第一批试点就铺到多个差异很大的团队，结果往往不是更全面，而是问题来源根本分不清。

### 13.4.3 给试点设退出条件

试点不该只有启动条件，也要有退出条件。没有退出条件，团队很容易把一个效果一般的方案长期拖着不收拾。表 13-15 给出一组够用的退出条件，它不是失败预案，而是防止团队把试点硬撑成长期负担。

**表 13-15：试点退出条件与处理方式**

| 出现什么情况 | 说明什么 | 该怎么处理 |
| --- | --- | --- |
| 第二个使用者仍然无法独立起手 | 共享规则还不够清楚 | 回退到补说明书和模板 |
| review 总在解释基本边界 | 交付格式或规则层有缺口 | 先补规则，不急着加自动化 |
| 自动化误伤频繁 | 规则太粗或时机不对 | 缩范围、降级自动化 |
| 成本明显上升但闭环没更稳 | 升级顺序错了 | 暂停扩展，先做复盘 |

### 13.4.4 用轻量指标判断是否值得扩展

小团队不需要一开始就做很复杂的治理看板，但至少可以跟几项轻量指标。表 13-16 不是为了把试点变成报表项目，而是为了避免只靠“大家主观上觉得好不好”来判断。

**表 13-16：试点期值得跟踪的轻量指标**

| 指标 | 你在观察什么 |
| --- | --- |
| 第二位成员起手成功率 | 共享说明是否有效 |
| 交付摘要是否可复用 | review 和 handoff 是否真的省时间 |
| 最小验证命令是否被稳定使用 | 流程是否落到实处 |
| 自动化误伤率 | 规则是否过粗 |
| 平均接力成本 | 异步协作是否开始起作用 |

当一条共享主线已经稳定后，团队才适合继续往更长链路扩，比如把 ticket、branch、PR、CI、release、incident 串成更深的受控闭环，把知识库和 runbook 更系统地接进来，把高频外部能力做成受控接口，把预算和审批一起纳入日常治理。顺序依然一样：先让每一段可被别人接住，再把多段串成系统。

这也是为什么小团队最需要的，不是一套华丽的大架构，而是一条可逐步扩展的落地路径。只要这条路径正确，团队化就会从一次试点自然长成长期工作方式。

## 本章小结

第 13 章真正想建立的，不是“团队也能用 Codex”这样一句空话，而是“团队什么时候值得开始共享规则、共享能力和共享接力机制”。只要这个前提没成立，团队化往往只会把个人习惯扩大成团队噪音。

如果把本章压缩成三句话，第一句是：值不值得团队化，要看任务重复度、规则稳定性和协作密度，而不是只看团队人数。第二句是：团队化最该共享的不是某个人的 prompt 技巧，而是说明书、规则、权限、能力包、知识库和 handoff。第三句是：小团队最稳的落地方式，是从一个项目、一条主线、一个试点小组开始，再逐步补异步协作、云端续跑和更深治理。

这一章也在提醒读者，团队化的真正门槛并不总在模型，而是在责任和接力。只要任务还只能靠一个人脑内持有，系统就很难升级成团队资产；只有当第二个人能起手、第三个人能接力、值班时也能不慌，团队化才算真正开始生效。

所以，进入最后一章前，最值得带走的不是组织大图，而是一条很实际的判断：先让共享主线成立，再让能力和治理往上长。顺序对了，团队化才会是复利；顺序错了，它就只是额外负担。
