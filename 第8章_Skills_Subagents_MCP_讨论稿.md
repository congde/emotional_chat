# 第 8 章 Skills、Subagents 与 MCP：用 Harness 把能力和上下文装进系统

第 8 章要解决的，不是“如何把 Agent 接得越来越花”，而是“如何把一次成功经验稳定地变成长期可复用能力”。到了这一阶段，`emotional_chat` 已经不只是一个可以让 Codex 修改的仓库：它有后端 Runtime、前端界面、RAG 知识库、插件、记忆系统、危机干预材料和运行数据。你会自然开始想：哪些经验应该沉淀成 Skill，哪些复杂任务应该拆给 Subagent，哪些外部信息值得通过 MCP 接进来。

更准确地说，本章要引入一种 harness engineer 的工作方式。这里的 harness 不是测试框架里狭义的测试夹具，而是一套让 Agent 能够稳定工作的工程环境：它规定输入从哪里来、边界在哪里、可以调用哪些能力、完成后如何验证、结果怎样交接。剧本式指令试图替 Agent 写下每一步动作；harness 式设计则把起点、终点、禁区和反馈回路设计清楚，让 Agent 在边界内自己完成任务。

这三者解决的问题并不相同。Skill 解决经验复用，Subagent 解决复杂任务拆分，MCP 解决外部能力接入。它们不应该被当成同一种“高级能力”，也不应该同时起步。一个成熟系统往往不是三者都很多，而是每一层都只在真正需要时出现。

本章会回答五个问题：

- 为什么能力系统的核心不是写剧本，而是设计 harness。
- 提示模板和 Skill 的真正分界线在哪里。
- 复杂任务什么时候值得拆给 Subagent，什么时候不值得。
- MCP 何时值得接，何时继续用现有脚本或人工导出更合理。
- 为什么一个健康的能力系统必须遵守“先简单、再升级”的装配顺序。

> 注意：能力系统不是按“能接多少”来成熟的，而是按“每多一层，是否真的减少任务摩擦”来成熟的。这个判断会帮你挡掉大量过早抽象和过度接入。

## 8.1 从剧本到 Harness：什么时候值得能力化

很多人第一次尝到 Codex 的甜头，都会马上进入同一个阶段：把一段有效的 prompt 保存下来，准备以后反复用。这一步没有错，但它还只是“把一次表达记下来”，还没有进入“把一次成功能力化”的阶段。

这里最容易走偏的地方，是把能力化理解成“把上次成功的每一步都写成固定剧本”。这会让系统越来越僵：场景稍微变化，剧本就开始失效；任务稍微复杂，剧本就开始漏条件。harness engineer 的思路正好相反：不要先规定每一步，而是先设计让 Agent 能稳定工作的边界条件。

### 8.1.1 Harness 设计的是可运行边界

表 8-1 把剧本式指令、提示模板、Skill、Harness 和临时说明放在一起比较。它的重点是提醒读者：不是所有有效经验都值得升级成 Skill，也不是所有稳定工作都靠写死步骤来完成。

**表 8-1：剧本式指令、提示模板、Skill、Harness 与临时说明的差异**

| 形态 | 主要解决的问题 | 典型内容 | 更适合什么场景 |
| --- | --- | --- | --- |
| 剧本式指令 | 让 Agent 按固定步骤执行 | 第一步做什么、第二步做什么、最后输出什么 | 低风险、路径确定、变化很小的一次性任务 |
| 提示模板 | 下次怎么更快说清任务 | 目标、范围、验证、输出格式 | 重复度中等、仍需要人强介入的任务 |
| Skill | 把一类任务变成长期可调用能力 | 任务说明、步骤、约束、输入输出、依赖 | 重复度高、边界稳定、希望多人或多轮复用 |
| Harness | 让一组能力在边界内稳定运行 | 输入、禁区、工具、权限、验证、反馈、交接 | 多人、多轮、高频、需要持续改进的工作流 |
| 临时说明 | 这次任务有特殊背景 | 一次性上下文和补充判断 | 偶发、变化快、不值得沉淀 |

在 `emotional_chat` 里，“帮我检查一次情绪趋势接口的 diff”可以先是一段提示模板；但如果团队每周都要做“RAG / 记忆 / 危机干预相关变更的风险审查”，而且审查步骤、输入输出和风险边界都稳定下来，它就开始值得沉淀成 Skill。再往前走，如果这个审查还要稳定接入 issue、读取运行手册、触发测试、产出 handoff，并和第 7 章的 Rules、Hooks、Approvals 连起来，它就不再只是一个 Skill，而是在形成一套 harness。

对 harness engineer 来说，真正要设计的不是一句更漂亮的 prompt，而是表 8-1 里那些会影响任务运行的条件：输入是否稳定，禁区是否清楚，工具是否可控，验证是否闭环，交接是否能被下一位人或 Agent 接住。Skill、Subagent 和 MCP 都只是这套 harness 里的部件。

提示模板解决的是“我下次怎么更快说清楚”；Skill 解决的是“系统以后该如何稳定按这套方式工作”。前者主要优化输入表达，后者主要优化可复用能力。两者相关，但不是一回事。

### 8.1.2 一个 Skill 至少要有稳定结构

Skill 最值钱的不是长，而是结构稳定。表 8-2 给出一个最小 Skill 应该明确的字段。只要这些字段还说不清，就说明它可能还只是一个长 prompt。

**表 8-2：一个最小 Skill 应明确的五个字段**

| 字段 | 作用 |
| --- | --- |
| 适用任务 | 这类 Skill 到底在解决什么 |
| 执行步骤 | 推荐按什么顺序做 |
| 约束边界 | 哪些动作别做，哪些情况先停 |
| 输入输出 | 调用时要给什么，结束时要交什么 |
| 验证方式 | 怎么知道这次执行是否合格 |

代码清单 8-1 把这五个字段放进一个 `emotional_chat` 风险审查 Skill 的骨架里。它看起来和 prompt 很像，但差别在于：它开始面向长期调用，而不是只服务一次对话。放在 harness 的视角下看，这个 Skill 只是其中一条稳定工作通道：它负责审查动作本身，至于什么时候触发、能不能读取外部信息、发现高风险后是否需要审批，还要交给更外层的控制和编排来接住。

**代码清单 8-1：`emotional-chat-risk-review-skill` 的最小骨架**

```markdown
# emotional-chat-risk-review-skill

## 适用任务
- 审查 `emotional_chat` 中涉及后端接口、RAG、记忆、危机干预或迁移的改动风险。

## 执行步骤
1. 先看任务目标和 diff。
2. 对照已跑验证和影响范围。
3. 检查是否触碰敏感目录：`knowledge_base/`、`alembic/`、`chroma_db/`、`uploads/`、`config.env`。
4. 先列 findings，再给摘要。

## 约束边界
- 不重写实现。
- 不评价纯风格偏好。
- 发现危机干预、临床建议或运行数据改动时，提高风险等级。

## 输出
- findings
- 风险等级
- 已验证内容
- 剩余风险
- 是否建议继续人工复核
```

### 8.1.3 不要把所有经验都包装成 Skill

比“怎么写 Skill”更重要的问题，是“什么时候值得写”。表 8-3 把适合 Skill 化和不适合 Skill 化的信号放在一起，帮助团队避免过早抽象。

**表 8-3：是否值得沉淀成 Skill 的判断信号**

| 你观察到的信号 | 更适合怎么处理 |
| --- | --- |
| 只是想下次表达更快 | 先做提示模板 |
| 一类任务高频重复，步骤比较固定 | 可以沉淀成 Skill |
| 输入输出和成败标准开始明确 | Skill 会比模板更划算 |
| 任务变化很快，依赖大量临场判断 | 保留为临时说明 |
| 还不确定是否会再用 | 先放进笔记，不急着技能化 |

Skill 一旦进入团队，就要开始考虑验证。个人阶段时，Skill 只要自己觉得好用，可能就够了；团队阶段就不一样了。你至少要能回答：这个 Skill 的成功标准是什么，哪些输出算合格，哪些反馈应该改 Skill 本身，而不是改本次 prompt。也正因为如此，Skill 最终会自然走向 evals。

## 8.2 Subagents：拆复杂任务，不是表演并行

很多人第一次听到 Subagent，会自然把它理解成“让更多 Agent 一起上，所以效率更高”。这句话只说对了一小半。复杂任务值得拆，不是因为多线程本身看起来高级，而是因为有些任务天然由几个相对独立的问题组成。如果这些问题可以并行探索、结果又能回收到主线程整合，那么拆分才会真正降低上下文拥堵和单线程来回切换的成本。

反过来说，如果主任务的下一步就被某个子问题卡死，或者多个子任务会写同一片文件，拆分反而会增加协调负担。Subagent 的本质不是“多叫几个人”，而是“把能并行、能汇总、边界清楚的部分外包出去”。

从 harness engineer 的角度看，Subagent 不是能力炫技，而是给复杂任务设计隔离舱：哪些问题可以独立探索，哪些文件可以独立写入，哪些结论必须回到主线程统一判断。只有这些边界先成立，并行才会变成收益，而不是新的协调成本。

### 8.2.1 适合拆的，不是“大任务”，而是“可分段任务”

任务大，并不自动等于值得拆。表 8-4 给出拆分前的四个判断条件。只要少一条，拆分收益就会明显下降。

**表 8-4：适合拆分成 Subagent 的前提条件**

| 条件 | 为什么重要 |
| --- | --- |
| 子问题边界清楚 | 不然子线程会反复追问主线程 |
| 结果可以独立汇报 | 不然最后无法整合 |
| 写入范围彼此可分离 | 不然容易冲突 |
| 主线程有别的事可继续做 | 不然只是把阻塞转移给别人 |

接手 `emotional_chat` 的陌生模块时，最稳的拆法往往不是让多个 Subagent 同时改代码，而是先让它们并行勘察。表 8-5 展示了一个三路并行勘察的拆法。

**表 8-5：`emotional_chat` 陌生仓库勘察时的三路并行拆分**

| 子任务 | 目标 | 产物 |
| --- | --- | --- |
| A 路 | 找入口文件、关键目录和主调用链 | 项目地图 |
| B 路 | 找复现路径、现有测试和最小验证入口 | 验证入口 |
| C 路 | 找敏感模块、运行数据和禁区 | 风险清单 |

主线程这时并没有停着等结果，而是在准备主任务边界、清理用户需求、规划后续整合动作。等三路结果回来，主线程再把它们压成一份更稳定的计划。

代码清单 8-2 把这个拆法写成一段可直接下发的任务说明。注意它强调“不修改文件”，因为并行最适合先发生在探索和信息收集层。

**代码清单 8-2：三路并行勘察的任务说明**

```markdown
子任务 A：
- 只负责找 `emotional_chat` 的项目入口、关键目录和主线程建议先看的文件。

子任务 B：
- 只负责找复现命令、现有测试和最小验证入口。

子任务 C：
- 只负责找敏感目录、历史脏点和建议不要动的区域。

共同要求：
- 不修改文件。
- 输出清单和判断依据。
- 不重复探索别人的问题。
```

### 8.2.2 子任务必须可合并

复杂任务拆分后，最终一定会回到一个主线程整合问题：这些结果如何合并成后续动作。表 8-6 给出常见交付格式和适用场景。子任务一开始就应该写清楚交付格式，否则主线程还要重新追问和整理。

**表 8-6：子任务常见交付格式与适用场景**

| 交付格式 | 适合什么 |
| --- | --- |
| 文件清单 | 勘察、定位、依赖图 |
| 结论清单 | 风险、问题、候选方案 |
| 局部 patch | 写入范围天然分离的实现工作 |
| 验证结果 | 测试、日志、回归信号 |

如果任务已经进入实现阶段，拆法就要从“按问题拆”转成“按写入边界拆”。表 8-7 用 `emotional_chat` 的功能开发来说明这种拆法：API、前端和验证可以分开，但每个子任务都必须知道自己的写入边界。

**表 8-7：实现阶段按写入边界拆分的示例**

| 子任务 | 负责范围 |
| --- | --- |
| API 子任务 | 接口定义、服务实现、后端测试 |
| 前端子任务 | 视图和交互，尽量不改 API 协议 |
| 验证子任务 | 回归测试、RAG 样例、文档同步、交付摘要 |

这里最关键的不是分工本身，而是让每个子线程都清楚自己拥有的写入边界。没有边界的并行写入，很快就会变成冲突制造机。

### 8.2.3 主线程负责整合和判断

Subagent 越多，主线程越不能退化成转发器。表 8-8 列出主线程必须保留的职责。高噪声、可并行的探索和执行可以下放；全局边界、优先级和责任归属仍然要在主线程。

**表 8-8：主线程必须保留的职责**

| 主线程职责 | 为什么必须留在主线程 |
| --- | --- |
| 锁任务边界 | 这是全局判断，不适合分散 |
| 决定是否采纳子结论 | 子线程只能给材料，不能替你定总方向 |
| 整合交付 | 最终对用户负责的是主线程 |

很多人做子任务拆分时，最容易只关注“怎么发出去”，却忽略“怎么收回来”。代码清单 8-3 给出一个固定回收格式。它能显著降低主线程整合成本。

**代码清单 8-3：子任务结果回收格式**

```markdown
请在最终结果里固定包含：
- 你的结论
- 你依据哪些文件或命令得到这个结论
- 仍然不确定的点
- 是否建议主线程继续深入
```

Subagent 最应该服务的，不是“看起来更强”，而是“让复杂任务的探索、实现和验证在不互相打架的前提下并行发生”。只要记住边界清楚、结果可合并、主线程仍然保留判断权，这套机制就会非常有用。

## 8.3 MCP：先问是否值得接，再问怎么接

MCP 很容易被写成两种极端：要么变成协议百科，全在讲术语；要么被写成“只要能接就都接”，仿佛能力来源越多越好。对工程实践来说，这两个方向都不够有用。真正该先问的问题不是“技术上能不能接”，而是“这类外部能力接进来后，任务真的更稳、更省还是更可控吗”。

在 `emotional_chat` 里，外部能力可能包括 issue、设计稿、运行日志、监控数据、知识库来源、模型服务和部署环境。并不是所有能力都值得立刻接成 MCP。很多时候，一个现有 CLI 脚本、一个只读 API 包装、甚至一份固定导出的文件，就已经够用了。

如果说 Skill 是 harness 里的经验通道，Subagent 是 harness 里的任务隔离通道，那么 MCP 就是 harness 的外部接口层。它把仓库外的信息和能力接进来，也因此必须比普通 prompt 更重视权限、审计和失败回报。

### 8.3.1 接入方式有不同成本

表 8-9 把几种常见能力接入方式放在一起比较。它最大的作用，是防止“为了接入而接入”。

**表 8-9：不同能力接入方式的成本和适用场景**

| 方式 | 更像在解决什么 | 成本 | 适合什么 |
| --- | --- | --- | --- |
| 手动提供文件 / 导出结果 | 一次性上下文补充 | 最低 | 偶发需求 |
| 现有 CLI / 脚本 | 在本地稳定执行一类动作 | 低 | 团队已有命令入口 |
| 轻量 API 包装 | 让外部数据可查询 | 中 | 只读查询较多 |
| MCP | 让外部能力变成长期可调用接口 | 中到高 | 高频、跨任务、需要统一治理 |

如果一个问题每周只发生一次，而且现有脚本已经很好用，直接上 MCP 可能只是增加维护层。只有当能力需要长期复用、上下文交互频繁、并且值得被统一治理时，MCP 才开始明显划算。

### 8.3.2 只读型接入优先，高风险接入后置

谈 MCP 最容易出错的地方，是把所有接入都当成同一种风险。表 8-10 按权限厚度拆开三类接入。这个拆法比“能不能接”更重要。

**表 8-10：按权限厚度区分的 MCP 接入类型**

| 接入类型 | 典型动作 | 风险特点 | 默认建议 |
| --- | --- | --- | --- |
| 只读型 | 查 issue、查文档、读设计稿、读监控数据 | 主要是信息质量和访问边界 | 最容易先落地 |
| 受限写入型 | 发评论、改状态、创建草稿 | 有副作用，但通常可回滚 | 需要清楚审计和范围限制 |
| 高风险写入型 | 改生产配置、执行远程命令、动真实数据 | 一旦误操作，代价高 | 默认非常保守，先别急着接 |

只读型接入通常最值得优先尝试，因为它能明显补强上下文，却不把高风险动作直接暴露给 Agent。很多团队第一次真正从 MCP 获得收益，都是从读信息开始，而不是从代执行开始。

代码清单 8-4 展示的是一个只读型 MCP 接入说明。它强调的不是功能多，而是边界清楚。

**代码清单 8-4：只读型 MCP 接入说明示例**

```markdown
能力：
- 查询 issue 状态
- 读取发布记录
- 检索运行手册

限制：
- 只读
- 不允许改状态
- 不允许触发流程
- 输出时标注信息来源和时间
```

### 8.3.3 MCP 值不值得接，要看它是否减少摩擦

表 8-11 把“值得接”和“不值得接”的信号放在一起。判断 MCP 价值时，不要只看技术可行性，而要看它是否在可控边界内持续减少任务摩擦。

**表 8-11：MCP 是否值得接入的判断信号**

| 信号 | 更适合的处理 |
| --- | --- |
| 同类查询反复出现，手工搬运上下文已经开始浪费 | 可以考虑只读型 MCP |
| 团队希望统一入口，并且权限边界清楚 | MCP 的治理价值开始显现 |
| 能明显减少切换成本 | 接入可能提升闭环 |
| 只是偶尔查一次 | 手动导出或临时文件更合适 |
| 现有 CLI 已经稳定好用 | 继续用 CLI，不急着再包一层 |
| 外部系统权限极乱 | 先整理权限，不急着开放给 Agent |

一旦 MCP 接入开始变多，复杂度会从能力端反噬回工作流端。表 8-12 列出接入后必须回答的治理问题。

**表 8-12：MCP 接入后必须回答的治理问题**

| 问题 | 不回答会怎样 |
| --- | --- |
| 哪些能力默认可见 | Agent 可能在过多工具里乱试 |
| 哪些能力谁能用 | 权限边界会失真 |
| 工具失败后怎么回报 | 主线程难以判断是真失败还是调用错 |
| 哪些调用需要审计 | 事后追责和复盘困难 |

本节的结论可以压缩成一句话：MCP 值不值得接，不取决于技术上能不能接，而取决于它是否在可控边界内持续减少任务摩擦。先从只读、高频、边界清楚的能力开始，通常是最稳的路径。

## 8.4 三者如何协同，而不是堆叠复杂度

到这里，读者已经分别看过 Skill、Subagent 和 MCP。真正难的地方现在才开始出现：三者一起上时，怎么避免系统越来越厚、入口越来越多、维护越来越乱。很多能力系统失败，并不是单个组件没用，而是所有组件都被当成“反正以后可能需要”的通用升级项，最后堆成了一个谁也说不清为什么存在的复杂层。

一个更稳的思路，是始终记住三者各自解决的问题。harness engineer 的工作，不是把 Skill、Subagent 和 MCP 全部装上，而是判断哪一层真的能降低摩擦，并把它们装到正确的位置。表 8-13 是本章最重要的职责分工表。

**表 8-13：Skill、Subagent 与 MCP 的职责分工**

| 组件 | 主要解决什么 |
| --- | --- |
| Skill | 把一类经验固化成可复用执行套路 |
| Subagent | 把复杂任务里的可并行部分拆出去 |
| MCP | 把外部能力接成长期可调用接口 |

只要这三个职责不混，装配路径就会很清楚。很多复杂度，都是从角色混淆开始的。

### 8.4.1 能少一层就少一层

最重要的装配原则其实非常保守：能少一层就少一层。表 8-14 把能力升级顺序写成一个判断表。每升一层，都必须回答“它解决了上一层解决不了的什么问题”。

**表 8-14：能力升级的推荐顺序**

| 当前问题 | 最先考虑什么 | 不要急着上什么 |
| --- | --- | --- |
| 只是一次任务没说清 | 普通任务说明 | Skill、MCP |
| 同类任务总在重复表达 | 提示模板或 Skill | Subagent |
| 单线程信息量太大 | Subagent 勘察 | 高风险 MCP |
| 外部数据反复手工搬运 | 只读型 MCP | 复杂编排层 |

图 8-1 把这条升级路线画成一条从简单到复杂的路径。它强调的是先把任务边界设计成 harness，再逐步加入具体能力，而不是让所有组件同时出现。

**图 8-1：从一次成功任务到 Harness 式能力系统的升级路线**

```text
普通任务说明
  ↓
Harness 式任务边界
目标 / 输入 / 禁区 / 验证 / 交接
  ↓
提示模板
  ↓
Skill（只在经验稳定、边界清楚时）
  ↓
Subagent（只在可并行、可回收时）
  ↓
只读型 MCP（只在外部上下文高频搬运时）
  ↓
evals / traces / handoff / 编排层
```

evals、traces、handoff 不应该在系统一开始就变成重治理负担，而应该在能力开始复用、开始共享、开始多组件协同时再逐步进入。

### 8.4.2 能力系统需要反馈、评测和交接

当 Skill 真正被多人调用，Subagent 开始参与复杂任务，MCP 也接进来后，系统会自然需要四类配套资产。表 8-15 解释它们分别解决什么问题。

**表 8-15：能力系统的四类配套资产**

| 配套资产 | 解决什么 |
| --- | --- |
| 反馈 | 这次哪里好，哪里误判 |
| traces | 这次到底怎么执行到这个结果 |
| evals | 这类能力是否稳定有效 |
| handoff | 结果如何被下一位人或 Agent 接住 |

这四类资产不是额外装饰，而是能力系统开始长期运转后必然出现的维护面。没有它们，能力很难持续变好，只会越积越多。

比如一个代码审查 Skill，最开始可能只是大家觉得好用；再往后，你会开始想知道它是不是总在遗漏同一类风险；然后你会希望把几次好的 review 结果沉淀成评测样例；再后来，你会要求它输出固定 handoff，方便下一位 reviewer 接手。这就是一个自然的升级过程。

### 8.4.3 编排层不要重写执行器

有些团队在接入 Skill、Subagent、MCP 之后，会进一步搭一个更上层的编排系统。这里最容易犯的错，是上层系统什么都想做，最后把 Codex 现成擅长的执行能力又重写一遍。更稳的分工通常是：让 Codex 负责读仓库、改文件、跑命令、整理结果；让上层编排层负责渠道接入、任务路由、审批管理、状态同步和多会话组织。

表 8-16 把执行层、编排层和规则层的职责拆开。这个表的作用是防止团队把所有能力都堆到同一层。

**表 8-16：执行层、编排层与规则层的职责**

| 层 | 更适合负责什么 |
| --- | --- |
| Codex / 执行层 | 理解代码、执行任务、产出结果 |
| 编排层 | 收入口、做审批、管状态、转发消息 |
| 规则层 | 定义哪些任务怎么走、哪些动作需升级 |

能力系统长期可用，最怕三种复杂度：能力过多、路由过多、权限过多。表 8-17 给出这些复杂度的典型症状。

**表 8-17：能力系统三种常见复杂度**

| 复杂度来源 | 典型症状 |
| --- | --- |
| 能力过多 | 同类 Skill 很多，但没人知道差别 |
| 路由过多 | 一个任务到底该走哪条路径说不清 |
| 权限过多 | 外部能力接了很多，但边界模糊 |

如果开始出现这些症状，系统通常需要做的不是再加一层，而是回头删。删掉低频 Skill，收缩不必要的 MCP，减少并非真正并行的子任务。能力系统的成熟，不是层越来越多，而是层数和任务复杂度保持匹配。

## 本章小结

第 8 章真正想建立的，不是一个看起来很全的能力栈，而是一种 harness engineer 的工作方式。提示模板、Skill、Subagent、MCP、评测和编排层都不是越早越好；它们只有在前一层已经开始显出瓶颈时，才值得被引入。

如果把本章压缩成三句话，第一句是：不要替 Agent 写死剧本，要设计输入、边界、工具、验证和交接。第二句是：模板复用表达，Skill 复用能力，Subagent 只拆可并行、可合并的问题。第三句是：MCP 值不值得接，取决于它是否在清楚边界内持续减少上下文搬运和任务摩擦。

本章最后其实是在强调一种工程克制。越是进入能力系统阶段，越要警惕层数失控、权限失控和路由失控。真正成熟的系统，不是组件越来越多，而是每个组件都有非常清楚的职责，并且只在必要时才出现。

所以，进入后面的团队化和长期方法论之前，最值得留下的判断是：先简单，能少一层就少一层；每次升级都要回答“它到底解决了上一层解决不了的什么问题”。只要这个问题答得稳，harness 就会把能力组织起来，而不是把复杂度堆起来。
