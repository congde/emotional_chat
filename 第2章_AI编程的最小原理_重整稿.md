# 第 2 章 AI 编程的最小原理：知道边界在哪里

Codex 最容易让人误判的地方，不是它写不出代码，而是它写得太像一个已经理解了问题的工程师。它能顺着需求进入代码库，找到相关文件，补出实现，运行测试，再根据报错继续修改。很多时候，结果来得又快又完整，于是人会自然地产生一种错觉：既然它已经能把任务做成这样，剩下的判断似乎也可以一并交给它。

但 AI 编程真正的分水岭，恰恰在这里。会生成代码，不等于看清了整个项目；能修复报错，不等于理解了所有业务约束；局部修改看起来正确，也不等于工程决策就是合适的。要把 Codex 用成一个可靠的协作工具，工程师首先要知道它为什么能写代码，它完成任务时依赖什么信息，又会在哪些边界上犯错。

## 2.1 大语言模型为什么能处理代码

代码看起来和自然语言很不一样。自然语言允许含糊、跳跃和省略，一句话即使语法不完整，人也常常能猜出意思；代码则严格得多，少一个括号、错一个缩进、传错一个参数，都可能让程序无法运行。也正因为如此，很多工程师第一次看到大语言模型写代码时，会本能地觉得奇怪：一个靠生成文本工作的模型，为什么能处理如此精确的东西？

答案恰恰藏在代码的这种“精确”里。

对大语言模型来说，代码首先也是一种由符号组成的序列。函数名、关键字、变量、括号、运算符、缩进关系，都会成为它判断后续内容的线索。模型并不是先像编译器那样建立一套完整的语法树，再像工程师那样逐步设计实现；它是在已有上下文中，根据训练中学到的大量结构和模式，生成接下来更可能成立的内容。

代码比自然语言更适合这种生成机制。因为它的约束更强，后续写法的空间反而更窄。看到一个函数定义，接下来通常是函数体；看到一次数据库查询，后面往往会出现结果处理、异常处理或返回值封装；看到一个测试文件中的既有写法，新测试大概率会沿用相同的 fixture、断言风格和命名方式。代码的结构性，让模型不必在无限开放的表达空间里寻找答案，而是在一组更明确的候选路径中继续往前写。

更重要的是，工程代码并不是一堆孤立的符号。人类为了让代码可维护，会不断在代码里留下意图：`calculate_order_total` 暗示它要处理订单金额，`validate_email` 暗示它要做合法性校验，`is_authenticated` 暗示它返回的是一个判断结果。目录结构、类型注解、测试名称、接口字段，都会把“这段代码要做什么”暴露出来。好的工程实践本来是为了帮助人协作，到了 AI 编程时代，它们也在帮助模型更准确地获得上下文。

这也是为什么同样一句“帮我补一下逻辑”，放进不同代码库里，Codex 的表现会差很多。在命名清楚、模块边界明确、已有实现可参考的项目中，它很容易顺着现有模式继续工作；在职责混乱、文件庞大、约定全靠口头传递的项目中，它就更容易误读意图，写出局部像样但整体不合适的代码。

除了结构和语义，代码还有一个天然优势：大量日常工程任务本身就具有很强的重复性。新增接口、补表单校验、处理分页查询、封装错误响应、补日志、写回归测试，这些工作并不等于“简单”，但它们确实在不同项目中反复出现。模型在大规模代码和技术文本中见过大量相似模式，因此在这些高频任务上，往往能很快生成一版看起来相当成熟的实现。

这里的关键，不是把 Codex 理解成一个“背过很多代码答案”的机器，也不是把它理解成一个已经拥有完整工程经验的程序员。更准确的说法是：它能利用训练中形成的模式能力，再结合眼前的代码上下文，快速生成一个很像工程解法的候选结果。

这也解释了 AI 编程中最重要的一条经验：越接近通用工程模式的任务，Codex 越容易发挥；越依赖具体业务规则、历史包袱和隐性约束的任务，它越需要工程师把上下文讲清楚。参考现有接口新增一个相邻功能，和独立决定一套复杂促销系统的定价规则，并不是同一种难度。

所以，理解 Codex 为什么能写代码，并不是为了神化它，而是为了把它放回正确的位置。它之所以有用，是因为代码本身足够结构化，工程上下文中充满语义线索，而软件开发又存在大量可复用模式。它的能力从这里长出来，它的边界也从这里开始显现。

## 2.2 Codex 完成任务时依赖什么

Codex 的能力不只是“模型会生成代码”。在实际工程中，它之所以比一段孤立的聊天回复更有用，是因为它能围绕任务建立上下文，读取项目文件，执行命令，观察反馈，再继续修改。

换句话说，AI 编程不是一次回答，而是一段工作过程。

### 2.2.1 上下文决定它此刻能看到什么

大语言模型每次处理任务，都受上下文窗口限制。你可以把它理解为模型在当前推理中能同时参考的信息容量。

这个窗口里通常要容纳：

- 你的任务描述；
- 项目中的相关代码和配置；
- 约束条件和已有文档；
- 工具调用结果；
- 测试日志、报错信息和执行历史；
- 当前会话中已经形成的结论。

窗口再大，也不等于整个代码库都在眼前。一个真实项目往往有大量模块、测试、配置、脚本、依赖说明和历史约定。Codex 必须在有限空间里选择当前最相关的信息。

这带来一个直接后果：

> 它不是基于“整个项目的全知视角”工作，而是基于“当前可见上下文”工作。

上下文足够聚焦时，它会显得很懂项目。上下文缺失、噪声过多或关键约束没有进入窗口时，它就可能写出局部合理、全局不合适的代码。

### 2.2.2 文件读取和检索帮助它补足项目背景

真实代码库通常远大于一次上下文能容纳的范围。AI 编程工具因此需要不断“找资料”。

它可能通过文件名、目录结构、关键词、符号引用、已有测试和调用关系，定位与当前任务有关的内容。你可以把这理解成一种面向项目的查阅能力：先找可能相关的材料，再把关键片段放进当前工作上下文中。

这也是为什么下面这些工程习惯会直接影响 AI 协作质量：

- `user_auth_service.py` 比 `service.py` 更容易定位；
- `src/payment/` 比一堆堆在 `utils/` 中的支付逻辑更容易理解；
- 清晰的测试名比含糊的测试名更能表达预期；
- 模块文档、接口注释和迁移说明会补充代码本身没有说清的约束。

检索和文件读取能扩展视野，但不能保证全知。它仍然可能找错文件、漏掉隐式依赖，或者把两个名称相近的模块混在一起。

### 2.2.3 执行反馈把“生成”推进到“交付”

代码和自然语言最大的不同之一，是代码可以被运行。

语法错误、导入错误、类型错误、测试失败、构建失败，这些都能形成反馈。Codex 如果能看到这些反馈，就能从“写一版”进入“改到更接近可交付”的循环：

1. 理解任务；
2. 读取相关上下文；
3. 修改代码；
4. 运行测试、构建或检查命令；
5. 根据反馈继续修复；
6. 汇报结果和剩余风险。

这个闭环非常重要。没有执行反馈，AI 代码更像一份草稿。加入验证之后，它才更接近工程协作。

因此，项目里已有的测试、lint、类型检查、构建脚本和最小可运行示例，不只是给人类开发者准备的。它们也是 AI 编程的护栏。

### 2.2.4 模型强，不等于上下文一定对

很多 AI 编程失败，并不是因为模型完全不会写，而是因为它在错误的上下文中写得太像回事。

例如：

- 项目里同时存在 `v1` 和 `v2` 两套接口，它改到了旧版本；
- 某个模块存在隐式注册关系，搜索调用方时漏掉了动态分发路径；
- 任务描述只说“加缓存”，却没有说明项目禁止新增外部依赖；
- 长会话积累了大量历史输出，当前任务的关键约束被淹没。

这也是为什么工程师不能只问：

> AI 会不会写？

还要问：

> 它此刻看到了什么？漏掉了什么？我怎样让错误尽快暴露？

## 2.3 第一条边界：信息永远不完整

AI 编程最基础的边界，不是“它聪不聪明”，而是“它能不能拿到足够的信息”。

人类工程师也会在信息不足时犯错。不同之处在于，Codex 生成速度很快，局部完成度又很高，所以信息缺口更容易被漂亮的输出掩盖。

### 2.3.1 它看不到整个项目

一个中型项目可能有几十个模块，上百个配置点，以及长期演化出来的历史约定。Codex 在执行某个任务时，通常只能围绕当前目标读取一部分内容。

如果任务范围很小，这不是问题。比如参考现有风格给一个已有函数补测试，所需上下文也许只包括实现文件、测试文件和少量配置。

但如果任务跨越模型、接口、序列化、迁移、文档和前端展示，信息缺口就会迅速扩大。只看见其中三处，往往只能完成一个不完整的改动。

### 2.3.2 隐式关系最容易被漏掉

代码中的显式依赖相对容易发现：

```python
from serializers.order import serialize_order
```

但项目中经常存在更隐蔽的关联，例如注册表、配置驱动、反射、字符串协议、事件监听和跨服务契约。

```python
order_data = serializer_registry["order"](order)
```

如果 `Order` 字段发生变化，真正需要同步修改的序列化逻辑可能藏在注册表背后。只靠局部阅读，很容易漏掉。

因此，越是依赖动态机制、弱约束协议和隐式约定的系统，越需要通过文档、测试和任务说明补充上下文。

### 2.3.3 长任务会放大上下文噪声

长任务中，代码片段、日志、报错、尝试过的修复方案和中间结论会不断累积。它们都在争夺当前工作上下文。

这会带来两种风险：

1. 早期关键约束被后续输出稀释；
2. 已经过时的中间状态继续影响后续判断。

当一个任务明显变长时，工程师要开始主动管理上下文：

- 把目标重新收束；
- 明确当前已完成和未完成的部分；
- 把关键约束写进任务描述或项目文档；
- 必要时拆成更聚焦的新任务。

### 2.3.4 项目结构就是上下文工程

很多人一提上下文工程，就只想到提示词。事实上，代码库本身就是最重要的上下文载体。

下面这些实践，既提升人类可维护性，也提升 AI 协作质量：

1. 保持模块小而聚焦，避免几千行的“上帝文件”。
2. 文件名和目录名表达业务语义，而不是把大量逻辑塞进 `utils`、`helpers` 和 `misc`。
3. 给关键接口写清输入、输出、依赖和兼容性约束。
4. 让测试表达真实预期，尤其覆盖边界条件和回归风险。
5. 把团队约定写下来，不要只依赖口头共识。

AI 编程不是让项目结构变得不重要。恰恰相反，项目越清楚，AI 越容易成为有效协作者。

## 2.4 第二条边界：它会犯三类工程错误

知道信息边界之后，再看 Codex 的错误，就会更容易分类。

在实际编程中，AI 常见的失败模式可以归纳为三类：幻觉、漏改和误判。它们不是同一种错误，检测方式也不同。

### 2.4.1 幻觉：不知道，却写得很像

幻觉最容易被注意到，因为它常常表现得很直接：

1. 调用了不存在的 API；
2. 混用了不同语言或不同版本的语法；
3. 写出了结构合理但事实不成立的参数、配置项或返回值处理方式。

例如：

```python
import requests

response = requests.fetch(url)
```

`requests` 并没有 `fetch` 这个常规调用方式。这个错误对熟悉 Python 的工程师很显眼，但如果它藏在一个陌生库或小众 SDK 里，就未必那么容易发现。

幻觉还有一种更隐蔽的形态：代码在某个版本中可能成立，但在你项目锁定的版本中不成立。模型“记得”一个常见写法，不等于它知道你当前环境的真实依赖版本。

#### 如何缓解幻觉

幻觉的应对方式，核心是让事实尽快接受检查：

1. 让代码跑起来。
2. 运行测试、构建、lint 和类型检查。
3. 对陌生库和关键 API 查项目实际版本与官方用法。
4. 审查新增 import、依赖和配置项。
5. 对小众库、内部 SDK 和版本敏感接口，给出文档或现有示例。

幻觉不是靠“相信它更认真一点”解决的，而是靠验证暴露的。

### 2.4.2 漏改：局部正确，整体不完整

漏改比幻觉更常见，也更像真实工程问题。

你让 Codex 把 `User.name` 拆成 `first_name` 和 `last_name`。它可能改了模型和接口，却忘了序列化器、迁移脚本、前端展示、测试和文档。

一次典型漏改，往往长这样：

```text
已修改：
  src/models/user.py
  src/api/users.py

可能遗漏：
  src/serializers/user_serializer.py
  migrations/
  tests/
  docs/api/users.md
```

漏改的特点是，每个已经修改的文件都可能写得没问题，但改动集合不完整。

#### 如何缓解漏改

漏改的应对方式，是主动扩大“需要检查的范围”：

1. 在任务开始时明确变更面，尤其是跨模型、接口和数据层的改动。
2. 搜索待修改字段、函数名、接口路径和配置键的引用。
3. 检查测试、迁移、文档、序列化和调用方是否同步更新。
4. 用 `git diff --stat` 和完整 diff 检查变更范围是否符合预期。
5. 运行覆盖相关模块的测试，必要时再跑更大范围的回归验证。

测试能发现很多漏改，但它只能覆盖已有测试能表达的风险。没有测试的地方，仍然需要人类审查和结构化搜索。

### 2.4.3 误判：代码没错，决策不对

误判最值得警惕，因为它往往不会立刻报错。

例如，项目原本统一使用标准库缓存方案。你只说“给这个函数加缓存”，Codex 可能引入一个新依赖：

```python
from cachetools import TTLCache, cached

@cached(cache=TTLCache(maxsize=100, ttl=300))
def get_user(user_id: int) -> User:
    ...
```

这段代码可能语法正确，行为也能满足局部需求。但它带来了新的工程问题：

- 增加外部依赖；
- 偏离项目已有缓存策略；
- 增加维护和安全评估成本；
- 让同类实现出现不一致。

误判不一定来自“不会写代码”，而是来自缺少项目的隐性背景：团队约定、历史决策、兼容性承诺、业务优先级和维护成本。

#### 如何缓解误判

误判最依赖工程师参与：

1. 把关键约束显性化，例如“不要新增依赖”“保持 API 向后兼容”“沿用现有缓存方案”。
2. 在项目文档中记录技术栈约束、风格约定和关键决策。
3. 审查依赖变更、接口变化和架构层调整。
4. 区分实现问题与决策问题。函数怎么写可以交给验证闭环检验，方案该不该这样设计仍需人把关。

幻觉、漏改和误判，暴露问题的方式并不相同。幻觉往往要靠运行和测试尽快揭穿；漏改需要搜索引用、检查变更面和回归验证；误判则更依赖工程师审查约束、依赖和架构决策。理解这一区别，后面的协作方法才有针对性。

## 2.5 从边界意识到协作方法

理解这些边界之后，AI 编程就不该再被看作一次“提问与回答”。它更接近一次受约束的工程协作：工程师提出目标，提供必要背景，保留关键判断，再用验证结果检验修改是否成立。Codex 能加快其中许多环节，但它不能替代这些环节本身。

这也解释了为什么同一个任务，换一种交代方式，结果会出现明显差异。“优化一下用户模块”几乎没有给出可执行边界；“为注册接口补充邮箱重复校验，保持现有响应格式，并补充回归测试”则同时说明了目标、约束和验收方向。前者把大量判断留给模型猜测，后者把任务放回工程语境中。AI 协作并不是把提示词写得花哨，而是尽量减少那些本不该由模型猜测的部分。

上下文也是同样道理。陌生 API、内部协议、特殊业务规则、历史兼容性要求，这些信息如果不进入任务过程，Codex 就只能依据眼前代码和既有模式作出推断。推断有时会很接近答案，有时却会把局部经验误当成项目事实。工程师真正需要补充的，不是越多越好的材料，而是能改变判断结果的材料：相关实现、关键文档、错误日志、数据样例，以及当前变更不能突破的边界。

对于跨文件修改，边界意识还意味着先确认影响面，再追求生成速度。字段重命名、接口调整、数据结构变化，看起来只是改一个定义，实际上常常牵动调用方、序列化层、迁移脚本、测试和文档。让 Codex 先沿着引用关系和现有实现查清范围，再进入修改，通常比直接要求它“完成改造”更接近真实工程流程。这样做不是降低效率，而是在更早的位置拦住漏改。

最后，验证必须回到协作中心。代码生成得再流畅，也不能把“看起来完成了”当成“已经成立”。运行、测试、构建、静态检查和 diff 审阅，分别在不同层面暴露问题：有些揭穿虚假的 API 和语法错误，有些发现关联修改没有同步，有些提醒工程师这次变更已经越过了原本的设计边界。到了依赖引入、兼容性、安全和业务规则这些问题上，最后的判断仍然属于人。

所以，本章讨论边界，并不是为了给 AI 编程泼冷水。恰恰相反，只有知道它依赖什么、会在哪些地方失真，工程师才可能把它纳入稳定的交付过程。下一章要展开的工作流，正是建立在这个前提之上。

## 本章小结

本章没有试图把大语言模型讲成一门完整课程，而是给出了 AI 编程最需要先建立的心智模型。

Codex 之所以能写代码，是因为代码高度结构化，自带大量语义线索，工程任务中又存在大量可复用模式。它能在上下文中生成很强的初稿，也能借助文件读取、命令执行和验证反馈推进任务。

但它不是站在整个项目之外的全知工程师。它依赖当前可见的上下文，依赖项目结构、文档、任务约束和验证闭环。信息不完整时，它会出现三类典型错误：

- 幻觉：不知道，却写得很像；
- 漏改：局部正确，整体不完整；
- 误判：代码没错，工程决策不合适。

工程师要做的，不是盲信，也不是拒绝，而是把这些边界纳入协作过程：让目标更明确，让上下文更可靠，让验证更及时，并在关键工程判断上保持责任。

理解这些之后，下一步不是继续停留在“AI 到底会不会写代码”的讨论里，而是把上下文、约束和验证组织成一个可复用的协作工作流。
