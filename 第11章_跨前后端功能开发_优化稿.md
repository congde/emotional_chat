# 第 11 章 场景三：跨前后端功能开发

跨前后端开发的核心挑战不是“写代码”，而是“对齐”。后端改了字段名，前端不知道；API 响应增加了新字段，前端没有处理；数据库迁移已经执行，ORM 模型却仍停留在旧版本；前端入口已经打开，后端数据回填还没有完成。很多联调问题表面上是代码 bug，本质上是接口、数据、发布顺序和验证方式没有对齐。

Codex 的价值，正好体现在这一类跨模块、跨文件、跨流程的任务中。它不是只在单个文件里补几行代码，而是可以围绕一个明确任务，读取并理解仓库中的 API 定义、数据库 schema、前端组件、测试用例和项目规则；在 CLI、IDE、Web 或 Cloud 等入口之间衔接任务；必要时并行处理子任务，再通过测试、代码审查和 PR diff 把结果交还给开发者确认。

因此，在跨前后端功能开发中，Codex 最适合承担的不是替人拍脑袋做技术决策，而是把需求拆成可执行的工程任务，把接口、数据、前端、测试和迁移统一到同一条上下文链路里，再用自动化验证和人工确认形成闭环。这样，“对齐”就不再只依赖人工沟通和经验记忆，而可以沉淀为一套可复用、可审查、可验证的工程流程。

## 11.1 需求的结构化拆解：从模糊到精确

产品经理给出的需求，往往天然是“业务语言”，而不是“工程语言”。比如一句“用户可以对对话进行评价”，听起来很简单，但真正落到代码里，背后会牵出一串技术决策：评分范围是多少，评论长度如何限制，评价后是否允许修改，标签是固定枚举还是后台配置，管理员能否回复，评价数据展示在哪里，是否涉及权限、审核、分页和数据迁移。

如果直接让 Codex “开始开发”，等于把这些尚未确认的技术决策默认交给 AI。Codex 虽然能读取仓库、理解上下文、修改代码、补充测试，但它并不天然知道业务方真正想要的边界条件。它会基于当前代码结构和常见工程惯例做出合理推断，但这些推断未必符合项目的真实约束。

更稳妥的做法，是先把 Codex 用作“需求结构化分析器”，而不是马上把它当作“代码生成器”。也就是说，先让 Codex 基于现有仓库上下文，分析这个需求会影响哪些数据模型、API 契约、前端组件、权限逻辑、测试用例和数据库迁移，再把其中需要人确认的技术决策显式列出来。

示例 11-1 展示了一个适合跨前后端需求拆解的指令。它的重点不是要求 Codex 直接开工，而是先把工程影响面和待确认问题摊开。

**示例 11-1：把产品需求拆成工程任务的 Codex 指令**

```text
分析以下产品需求，并结合当前代码仓库，从技术角度进行结构化拆解：

【原始需求】
用户可以对对话进行评价：1-5 星评分，可选文本评论（最多 500 字），
可选标签，评价后不可修改，管理员可回复，用户可在历史页查看。

请输出：
1. 需要新增或修改哪些数据模型？字段包括什么？
2. 需要设计哪些 API 接口？请求和响应格式是什么？
3. 前端需要新增或修改哪些页面、组件和状态逻辑？
4. 是否涉及数据库迁移？会影响哪些表和索引？
5. 哪些技术决策点必须由人确认？
6. 哪些测试用例需要补充？
7. 按依赖关系排列开发任务，并标出每一步的验证方式。
```

Codex 的输出通常会覆盖数据模型、API 端点、前端组件、迁移方案和测试建议，但其中最有价值的部分，往往不是“它准备写哪些代码”，而是“它暴露出了哪些还没想清楚的问题”。表 11-1 列出几类常见的确认点。它们越早被确认，后面的实现和联调越少返工。

**表 11-1：跨前后端需求拆解中的关键确认点**

| 确认点 | 需要回答的问题 | 如果跳过会怎样 |
| --- | --- | --- |
| 数据模型 | 字段、约束、索引、关联关系是什么 | 迁移后发现数据结构不支持业务规则 |
| API 契约 | 请求、响应、错误码和分页规则是什么 | 前后端各写一套理解，联调时漂移 |
| 权限边界 | 谁能评价、谁能查看、谁能回复 | 上线后出现越权或入口不可用 |
| 前端状态 | loading、empty、error、submitted 如何展示 | 页面能跑但体验不完整 |
| 测试策略 | 哪些单元、集成、契约和回归测试必须补 | 修复问题时无法判断是否破坏旧链路 |
| 发布顺序 | 数据迁移、后端、前端、灰度如何排序 | 某一步回滚时牵连其他模块 |

因此，Codex 在需求拆解阶段的最佳用法，不是让它替人做最终决策，而是让它把隐藏在一句业务需求背后的工程影响面全部摊开。人确认关键决策后，再让 Codex 进入实现阶段，细化任务、修改代码、补充测试并生成 PR diff。这个“需求分析 -> 决策确认 -> 实现执行 -> 测试验证”的闭环，才更适合真实团队中的跨前后端开发。

## 11.2 API 契约的演进：从手动同步到自动生成

跨前后端开发中，最容易出问题的并不是某一段代码写错了，而是 API 契约在多个地方发生了漂移。后端的 Pydantic 模型、FastAPI 路由、OpenAPI 文档、前端 TypeScript 类型、API Client、表单校验和测试用例，本来应该描述同一件事，但在真实项目中，它们经常各自演进：后端新增了字段，前端没有处理；后端把字段从必填改成可选，前端校验仍然停留在旧逻辑；接口返回了新的错误码，前端却没有对应的展示分支。

这类问题靠人工同步很难彻底避免。Codex 的价值不只是帮你补几段接口代码，而是能够把仓库中分散的契约信息放到同一个上下文里检查。它可以同时阅读后端模型、路由实现、OpenAPI 输出、前端请求封装、组件调用点和测试文件，判断这些地方是否仍然指向同一套接口约定。

### 11.2.1 先发现契约漂移，再决定事实来源

最原始的做法，是后端写 Pydantic 模型，前端照着写 TypeScript interface 或 JSDoc 类型定义，然后靠人工检查一致性。这种方式在小项目中勉强可行，但只要需求稍微复杂，就很容易出问题。比如后端新增了 `rating_tags` 字段，前端忘了传；后端把 `comment` 限制为 500 字，前端仍然允许输入 1000 字；后端返回 `error_code`，前端只处理 `message`。

示例 11-2 是一次契约一致性检查指令。它要求 Codex 先检查、再输出漂移点，而不是马上修改代码。

**示例 11-2：检查前后端 API 契约是否漂移**

```text
检查以下几处定义是否一致：

1. 后端 FeedbackRequest / FeedbackResponse 的 Pydantic 模型
2. FastAPI 路由中的请求参数和响应结构
3. 自动生成的 OpenAPI schema
4. 前端 feedback API Client 的请求参数和返回值类型
5. FeedbackForm 组件中的表单校验逻辑
6. 相关单元测试和集成测试

请输出：
- 已经一致的部分
- 存在漂移的字段、类型、必填规则或错误码
- 哪些地方应该以谁为单一事实来源
- 最小修复方案
- 修复后需要运行哪些验证命令
```

这类任务非常适合 Codex，因为它不是凭空猜接口，而是基于当前仓库里的真实文件做对比。它的输出重点也不应该是马上修改代码，而是先把“契约漂移点”列清楚，让人确认哪一份定义才是事实来源。

### 11.2.2 用自动生成消除重复维护

更稳妥的做法，是从“人工同步”升级到“自动生成”。如果后端使用 FastAPI，Pydantic 模型可以进入 OpenAPI schema；前端再通过 Hey API、OpenAPI Generator 等工具生成 TypeScript 类型和 API Client。FastAPI 官方文档也把这种路径作为生成 SDK 的典型方式：后端通过 OpenAPI 描述接口，再由工具生成前端可用的客户端代码。

图 11-1 展示了这条自动化链路。它的核心是减少手写副本，让前后端围绕同一份契约演进。

**图 11-1：API 契约的自动化链路**

[drawio 源文件：图11-1_API契约的自动化链路.drawio](figures/图11-1_API契约的自动化链路.drawio)

代码清单 11-1 给出一个最小生成脚本。这里的命令只是示意，真实项目还需要根据目录结构、鉴权方式和生成器配置调整。

**代码清单 11-1：从 OpenAPI 生成前端类型和 API Client**

```json
{
  "scripts": {
    "openapi:generate": "openapi-ts --input http://localhost:8000/openapi.json --output frontend/src/generated --client axios",
    "openapi:check": "npm run openapi:generate && npm run typecheck"
  }
}
```

在这条链路中，Codex 的角色不是替代工具链，而是把工具链串起来。它可以检查 FastAPI 是否正确暴露 OpenAPI；补充生成脚本；修改前端调用方式；把手写类型替换为生成类型；补充契约测试；最后运行测试并生成 PR diff。也就是说，Codex 最适合承担的是“从发现问题到整理修复路径”的工程编排，而不是让它长期维护两套手写接口定义。

对 Python 后端 + React 前端这类异构技术栈来说，REST + OpenAPI + TypeScript Client 生成通常是最实用的路径。如果是 TypeScript 全栈内部应用，也可以考虑 tRPC，通过类型推断实现端到端类型安全。但 tRPC 的前提是前后端共享同一套类型系统，对 Python、Java、Go 等后端并不天然适合。

### 11.2.3 规范优先、代码优先和运行时校验要分清边界

代码生成还可以进一步前置。如果核心 API 对业务非常关键，可以先写 OpenAPI 规范，再让 Codex 基于规范生成后端接口、前端 Client 和测试用例。这就是规范优先的思路：先把接口契约讨论清楚，再进入实现阶段。

规范优先的优势是约束更早。API 的字段、类型、错误码、鉴权方式、分页规则、兼容性要求，都可以在编码前被显式讨论。Codex 处理这类结构化输入也更稳定，因为 OpenAPI 规范比自然语言需求更明确。但规范优先也有成本。如果规范更新不及时，规范本身也会变成新的漂移源。因此实践中更推荐混合策略。表 11-2 把几种策略放在一起，方便团队判断。

**表 11-2：API 契约策略的适用边界**

| 策略 | 更适合的场景 | 主要风险 | Codex 可以做什么 |
| --- | --- | --- | --- |
| 手动同步 | 很小的内部接口、短期原型 | 类型、字段和错误码漂移 | 做一致性检查，列出漂移点 |
| 代码优先 | FastAPI 等框架快速迭代 | 生成链路缺失，前端仍手写类型 | 补生成脚本、替换手写类型、加契约测试 |
| 规范优先 | 核心业务 API、对外 API、多团队协作 | 规范更新不及时 | 根据规范生成实现和测试，检查破坏性变更 |
| 运行时校验 | 前端需要防御异常响应、表单规则复杂 | 与后端事实来源分裂 | 检查 Zod schema 与 OpenAPI / Pydantic 是否一致 |

TypeScript 类型只能在编译阶段生效，真实 API 返回的数据不会自动经过校验。如果后端返回了错误结构，TypeScript 本身并不会在运行时阻止问题进入页面逻辑。Zod 的价值在于把“类型定义”和“运行时验证”结合起来：前端既可以用 schema 推导类型，也可以在接口返回后做实际校验。

如果项目使用 Zod，还可以通过 zod-openapi 等方式把 schema 转成 OpenAPI 描述，形成另一种事实来源。但这里要注意边界：Zod 更适合作为前端运行时安全和表单校验的核心工具；如果后端是 FastAPI / Pydantic，通常不建议再让前端 Zod 反向主导整个 API 规范，否则前后端事实来源会再次分裂。更合理的做法是：后端契约以 Pydantic / OpenAPI 为主，前端运行时校验以 Zod 为补充。Codex 可以负责检查两者是否一致，并在不一致时提示开发者确认应该以哪一边为准。

示例 11-3 把这一节的做法收束成一条可执行指令。它要求 Codex 设计契约自动化方案，而不是只修某个字段。

**示例 11-3：设计 API 契约自动化方案的 Codex 指令**

```text
请为当前项目设计一套 API 契约自动化方案：

1. 以后端 Pydantic / FastAPI 为事实来源
2. 自动生成 OpenAPI schema
3. 自动生成前端 TypeScript 类型和 API Client
4. 在 CI 中检查 OpenAPI 是否发生破坏性变更
5. 补充一个最小契约测试
6. 输出需要新增的脚本、配置文件和验证命令
7. 所有修改以最小 PR diff 的形式完成
```

这样，Codex 的能力就不只是“帮开发者写接口”，而是把跨前后端协作中最容易出错的契约同步问题，转化成一条可检查、可生成、可验证、可审查的工程流程。

## 11.3 数据库迁移：最高风险的一步

在跨前后端开发中，数据库 schema 变更往往是风险最高的一步。前端代码写错了可以回滚，后端接口写错了可以重新发布，但如果迁移脚本误删字段、错误转换数据，或者在大表上执行了阻塞性变更，影响的就不只是代码，而是线上数据和业务连续性。

Codex 可以帮助开发者生成 Alembic、Flyway、Liquibase 或其他迁移文件，也可以根据模型变化推断应该新增哪些字段、索引和约束。但这里必须明确一个原则：数据库迁移不是“让 Codex 写完就执行”的任务，而是“让 Codex 生成草案，人类重点审核，系统自动验证”的高风险流程。生成只是起点，审核和验证才是关键。

### 11.3.1 迁移风险不只在 migration 文件里

普通代码变更主要影响程序逻辑，数据库迁移影响的是历史数据。代码 diff 里看到的只是一行 `add_column` 或 `drop_column`，但线上可能对应几百万、几千万甚至更多历史记录。一次看似简单的字段新增，可能引发锁表；一次字段类型修改，可能导致历史数据转换失败；一次删除表或删除列，可能让回滚失去基础。

Codex 生成的迁移文件，通常可以在字段名称、字段类型、基础索引上给出合理方案，但它不一定知道线上数据量、数据库版本、业务访问峰值、是否允许停机、是否存在历史脏数据，也不一定知道团队内部的发布窗口和回滚预案。因此，数据库迁移必须比普通代码变更多一层审查。

表 11-3 把原文中的五类风险合并到同一张审查表里。它们的共同点是：单看迁移文件往往发现不了，必须把数据规模、应用逻辑、发布顺序和跨层契约一起看。

**表 11-3：数据库迁移的五类高风险点**

| 风险 | 典型表现 | 应该让 Codex 检查什么 |
| --- | --- | --- |
| 逻辑关联下的孤儿数据 | 没有物理外键，删除主记录后关联数据残留 | Service 层级联、软删除、孤儿数据检测 SQL |
| 大表变更的停机风险 | 加列、改类型、建索引导致锁表或复制延迟 | 数据量、数据库版本、在线 DDL、低峰执行 |
| 回填与发布顺序错误 | 代码先读新字段，数据还没有回填 | schema、后端、回填、前端、约束收紧的顺序 |
| 降级脚本不完整 | `downgrade` 只是形式存在，无法真正回滚 | 索引、约束、枚举、不可逆数据变更 |
| 跨层契约不一致 | 迁移已改，ORM / Pydantic / OpenAPI / 前端类型没同步 | 模型、接口、生成类型、组件和测试是否一致 |

示例 11-4 展示了一个数据库迁移审查指令。它继承原文的思路：不要只要求 Codex 生成迁移，而要让它带着线上规模和执行约束做风险判断。

**示例 11-4：带线上约束审查数据库迁移**

```text
请基于以下前提审查迁移方案：

- chat_sessions 表约有 800 万行数据
- 当前数据库为 MySQL
- 业务不允许长时间锁表
- 迁移需要在不停机条件下完成

请判断：
1. 当前 Alembic 迁移是否可能触发长时间锁表？
2. 新增字段是否可以先设置为 nullable？
3. 是否需要分阶段执行：加字段 -> 回填数据 -> 加约束 -> 切换代码？
4. 是否需要使用在线 DDL 工具或低峰期执行？
5. 是否需要增加批量回填脚本？
6. 每一步失败后如何回滚？
```

这样，Codex 的角色就不再是简单生成 migration 文件，而是帮助开发者把“数据库变更”拆解成一组更安全的上线步骤。

### 11.3.2 把迁移拆成向前兼容的发布计划

跨前后端功能开发里，数据库迁移经常不是孤立动作，而是和后端代码、前端页面、API 契约一起发布。问题在于：数据库变更、代码发布、数据回填之间存在严格顺序。如果顺序错了，代码可能读取不到字段，前端可能收到异常响应，或者旧版本代码无法兼容新 schema。

更稳妥的方式，是把一次迁移拆成“向前兼容”的多阶段发布。图 11-2 展示了五阶段安全发布计划：先做兼容性 schema 变更，再发布兼容代码，随后回填数据、打开前端入口，最后在确认稳定后收紧约束。

**图 11-2：数据库迁移的五阶段安全发布计划**

[drawio 源文件：图11-2_数据库迁移五阶段安全发布计划.drawio](figures/图11-2_数据库迁移五阶段安全发布计划.drawio)

Codex 很适合帮助开发者生成这类分阶段计划。示例 11-5 的重点是明确要求它不要直接给出单次迁移脚本，而是先拆成安全发布步骤。

**示例 11-5：把数据库变更拆成安全发布计划**

```text
请不要直接给出单次迁移脚本，而是把本次数据库变更拆成安全发布计划：

1. 哪些 schema 变更可以先做？
2. 哪些代码需要兼容新旧字段？
3. 是否需要数据回填脚本？
4. 前端入口应该在哪个阶段打开？
5. 哪些变更必须延后到确认稳定之后？
6. 每个阶段的验证命令是什么？
7. 每个阶段失败后如何回滚？
```

这比“生成一个 migration 文件”更符合真实工程交付，也更符合 Codex 作为工程执行助手的定位。

### 11.3.3 迁移审查清单要进入团队默认规则

迁移完成不代表开发完成。一个常见问题是：迁移文件已经新增了字段，但 ORM 模型没有同步更新；ORM 模型更新了，但 Pydantic response model 没有更新；后端返回结构变了，但前端 TypeScript 类型和 API Client 没有同步生成。最后表现出来的不是“数据库迁移失败”，而是接口报错、字段缺失或前端渲染异常。

图 11-3 展示了迁移后的跨层一致性检查。它把数据库迁移、ORM 模型、API schema、前端类型和测试用例串起来看，避免某一层单独“看起来正确”。

**图 11-3：跨层一致性检查链路**

[drawio 源文件：图11-3_跨层一致性检查链路.drawio](figures/图11-3_跨层一致性检查链路.drawio)

对于 Codex 生成的数据库迁移，至少应该按表 11-4 审查。这张表适合写进 `AGENTS.md` 或团队迁移规范中，让 Codex 每次处理数据库变更时自动遵循。

**表 11-4：数据库迁移审查清单**

| 检查项 | 要确认什么 |
| --- | --- |
| 不可逆操作 | 是否存在删除表、删除列、清空数据等不可逆操作 |
| 大表变更 | 是否涉及可能锁表的加列、改类型、建索引 |
| 默认值与约束 | 新字段是否需要默认值，默认值是否会触发全表重写 |
| 分阶段迁移 | 是否需要先 nullable，再回填，再加 not null 约束 |
| 逻辑关联 | 是否存在逻辑外键和孤儿数据风险 |
| 索引成本 | 索引是否覆盖主要查询路径，是否增加额外写入成本 |
| 降级脚本 | downgrade 是否完整，是否明确不可逆风险 |
| 跨层一致性 | ORM、Pydantic、OpenAPI、前端类型是否一致 |
| 回滚预案 | 是否有迁移前备份、迁移后校验和失败回滚方案 |
| 启用节奏 | 是否需要 Feature Flag 控制新字段或新功能的启用 |

这类检查本质上是机械性、系统性的，非常适合交给 Codex 做第一轮扫描。但最终是否允许执行迁移，仍然应由工程师确认。

## 11.4 渐进式交付：从写完就发到灰度发布

代码写完只是开发工作的结束，交付工作的开始。在跨前后端场景中，上线风险尤其高：前后端必须同时兼容，API 契约必须稳定，数据迁移必须先于代码生效。渐进式交付通过 Feature Flag、灰度发布和监控回滚降低风险，但它不能替代正确的发布顺序。

### 11.4.1 Feature Flag 控制的是入口，不是发布纪律

对评价功能来说，Feature Flag 的集成点通常有三个：前端组件根据 Flag 状态决定是否渲染；后端 API 端点根据 Flag 返回 404 或正常响应；配置系统负责灰度策略，比如先对 5% 用户开放，观察后再扩大到 50% 和全量。

代码清单 11-2 是一个简化的 FastAPI + Unleash 示例。它保留原文中的核心做法：后端路由在执行前检查功能开关，避免前端入口之外的直接调用绕过灰度。

**代码清单 11-2：在 FastAPI 路由中接入 Feature Flag**

```python
import os

from fastapi import Depends, FastAPI, HTTPException
from UnleashClient import UnleashClient

app = FastAPI()

unleash = UnleashClient(
    url="https://unleash.example.com/api",
    app_name="emotional-chat",
    custom_headers={"Authorization": os.environ["UNLEASH_TOKEN"]},
)


@app.on_event("startup")
def init_flags():
    unleash.initialize_client()


def feedback_enabled(user_id: str) -> bool:
    return unleash.is_enabled(
        "feedback.v1",
        context={"userId": user_id},
        fallback_function=lambda *_: False,
    )


@app.post("/api/feedback")
def submit_feedback(
    req: FeedbackRequest,
    enabled: bool = Depends(lambda: feedback_enabled(req.user_id)),
):
    if not enabled:
        raise HTTPException(status_code=404)
    return svc.create_feedback(req)
```

前端侧把同样的 flag 名挂在 React 组件上，灰度策略统一在 Feature Flag 平台中调整，后端代码不需要重新发布。Codex 可以帮助识别哪些入口应该挂 flag、补齐前后端调用点、生成验证测试，但“创建、启停和扩大灰度比例”应该保留明确的人类审批。原因很简单：Feature Flag 改变的是生产流量的可见范围，不只是代码分支。

如果发布的功能本身涉及大语言模型调用，还需要双指标体系。除了传统工程指标，如错误率、延迟和资源消耗，还要观察业务质量指标，如回答准确性、用户满意度、人工接管率或投诉率。否则可能出现“工程指标全绿，但用户体验变差”的情况。

### 11.4.2 设计系统也需要进入上下文

跨前后端交付中还有一个常被忽视的问题：AI 生成的 UI 代码是否符合设计系统规范。如果 Codex 不知道项目的设计 token、组件 API、交互状态和 Storybook stories，它可能生成“看起来能用但不符合设计系统”的代码，导致视觉一致性问题和返工。

Storybook 的 MCP 能力已经开始把组件文档、props、stories 和交互测试暴露给 AI 代理。对 Codex 这类工具来说，关键不是“让 AI 自己设计界面”，而是让它在生成 UI 前先查询项目已有组件和使用规范。示例 11-6 给出一段适合写入 `AGENTS.md` 的规则。

**示例 11-6：让 Codex 遵循设计系统的项目规则**

```text
生成或修改前端 UI 时：
1. 先查询 Storybook 中已有组件、props 和 stories。
2. 优先复用现有组件，不新增平行组件。
3. 遵循项目设计 token，包括颜色、间距、字体和状态样式。
4. 新增交互状态必须包含 loading、empty、error、disabled。
5. 需要补充对应 story，便于人工预览和交互测试。
```

这样，设计系统就不再只是设计师和前端工程师脑中的约定，而成为 Codex 可以读取、遵守和验证的工程上下文。

### 11.4.3 前后端协调发布要建立向前兼容窗口

Feature Flag 和设计系统解决的是“谁看到新功能”和“界面是否符合规范”，但跨前后端发布还有更基础的一步：什么时候发什么。如果数据库迁移、后端代码、前端代码、数据回填的顺序错了，再完美的灰度策略也救不回来。用户可能看到 500、看到空字段，或者旧 UI 调用了新 API。

图 11-4 展示了跨前后端发布的安全时序。核心思想是建立一段“向前兼容窗口”：在这段时间里，数据库 schema、后端代码、前端代码都同时支持新旧两个版本，任何一步出问题都能独立回滚而不连锁。

**图 11-4：跨前后端发布的安全时序**

[drawio 源文件：图11-4_跨前后端发布安全时序.drawio](figures/图11-4_跨前后端发布安全时序.drawio)

代码清单 11-3 把这条纪律写成 `AGENTS.md` 片段。它比任何单次代码评审都更值得沉淀，因为它把“上线翻车”从一个偶发事件变成一个有结构、有兜底的过程。

**代码清单 11-3：跨前后端发布纪律的 AGENTS.md 片段**

```md
## 跨前后端发布纪律

- 任何涉及 DB schema 的功能必须按“6 步发布时序”拆 PR：
  schema 兼容 -> 后端兼容 -> 数据回填 -> 前端 + Flag -> 灰度监控 -> 收紧约束
- 禁止把 schema 收紧（NOT NULL / DROP）与前端打开 Flag 放在同一次发布。
- 任何回填脚本必须分批执行，单批不超过 5 万行，且可幂等。
- Feature Flag 灰度至少经过 5% -> 25% -> 50% -> 100% 四档。
- 每一档扩大前必须确认错误率、p99 延迟和关键业务指标。
```

多数生产事故的根因不是 AI 写错代码，而是“该按顺序发布的东西被合在了一起”。Codex 可以帮助生成计划、拆 PR、检查顺序和补验证，但它不能替代团队对发布风险的判断。

## 11.5 一次通过率：信息对齐是核心

“一次通过率”指的是代码从提交到合并不需要返工修改的比例。在跨前后端开发中，它面临的最大威胁是上下文碎片化：后端工程师不知道前端组件结构，前端工程师不知道后端数据校验规则，接口对接时频繁翻车。

Codex 的独特优势是它能在同一个上下文窗口中同时持有前后端信息。当你让它实现“管理员回复评价”功能时，它可以同时检查后端返回的错误码前端是否有对应处理、前端 loading 状态是否覆盖后端异步操作、前端表单校验是否与后端 Pydantic 校验一致。但 Codex 不是万能的。提高一次通过率的核心策略，是把信息对齐做在开发过程里，而不是等 PR 打开后再补。

表 11-5 汇总了三种最值得保留的做法。它们都来自原文，只是从分散表述收回到同一条主线：减少返工，靠的是持续对齐。

**表 11-5：提高一次通过率的三种对齐机制**

| 机制 | 做法 | Codex 的作用 |
| --- | --- | --- |
| 统一技术规格文档 | 开发前生成并维护一份技术规格，记录接口、数据、状态和发布顺序 | 起草规格、更新决策、发现遗漏 |
| 增量验证 | API 完成后立刻 curl / 集成测试，前端完成后用 mock 数据验证，联调时跑完整流程 | 生成验证命令、补测试、整理失败原因 |
| 交叉验证清单 | 合并前检查 API 参数、错误码、ORM、迁移、OpenAPI、前端类型是否一致 | 做机械性一致性扫描，输出 PR 风险摘要 |

示例 11-7 给出一个适合在 PR 前执行的交叉验证指令。它把“是否对齐”变成一项明确任务，而不是靠 reviewer 临时想起。

**示例 11-7：跨前后端功能的 PR 前交叉验证指令**

```text
请对当前 PR 做跨前后端交叉验证：

1. API 请求参数、响应字段和错误码是否前后端一致？
2. ORM 模型、数据库迁移和 Pydantic 模型是否一致？
3. OpenAPI schema 是否已更新，前端类型和 API Client 是否已重新生成？
4. 前端组件是否覆盖 loading、empty、error、disabled 状态？
5. 新功能是否受 Feature Flag 控制，灰度入口是否明确？
6. 是否有必要的单元测试、集成测试、契约测试和回归测试？
7. 请输出：已通过项、风险项、建议补充的最小修改和验证命令。
```

这种机械性的对齐工作正是 Codex 最擅长的，而人类工程师应该把精力集中在真正需要判断的部分：业务规则是否合理、发布风险是否可接受、用户体验是否符合预期。

## 本章小结

跨前后端开发的挑战不是写代码，而是对齐。本章从需求拆解开始，先把一句业务语言转成数据模型、API 契约、前端状态、迁移和测试任务；再讨论 API 契约如何从手动同步走向自动生成和契约验证；随后进入风险最高的数据库迁移，把迁移文件、线上数据、发布顺序和跨层一致性放在一起审查；最后回到渐进式交付和一次通过率，说明为什么 Feature Flag、设计系统、向前兼容窗口和交叉验证都应该成为默认流程。

Codex 在这一章里的定位始终没有变化：它不是替人决定业务边界，也不是把跨前后端开发变成一次性代码生成，而是帮助团队把分散在仓库里的事实对齐起来，把容易遗漏的检查显式化，把一次成功的交付沉淀成下一次也能复用的默认路径。

下一章会继续沿着交付链路往后走：当开发者离开工位，任务还在云端运行，PR 还在等待审批，手机上又收到新的反馈时，Codex 应该怎样支撑远程续跑、异步审批和离线后的回看。
