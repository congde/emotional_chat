# 第 6 章 AGENTS.md：给 Agent 的项目说明书

Codex 进入一个仓库时，最缺的往往不是代码，而是项目里的默认知识：先看哪里、哪些别动、改完怎么验证。以本书一直使用的 `emotional_chat` 项目为例，仓库里同时有 FastAPI 后端、React 前端、RAG 知识库、向量数据库、Alembic 迁移、监控配置和情感/危机识别逻辑。人类同事可以靠追问补上下文，Agent 如果一开始猜错方向，就很容易把小问题放大成误改、漏测和越权操作。

AGENTS.md 要解决的就是这件事。它不是 README 的重复版，也不是团队文化宣言，而是一份给 Agent 的项目说明书：这个项目怎么分层，关键入口在哪，哪些区域高风险，完成任务后该如何交付。本章所有示例都会尽量落回 `emotional_chat`，因为一份说明书只有贴着真实仓库，才会从“写得对”变成“用得上”。

本章会把 AGENTS.md 拆成五个可落地的问题：

- 怎么写项目地图、关键入口和禁区，让 Agent 上来先找对地方。
- 为什么复杂项目需要按根目录与子目录分层写说明书。
- 团队硬规则和个人偏好该怎么拆，哪些该留给后面的控制层。
- 多人协作时，如何把角色分工、交接协议和值班边界写进项目说明书。
- 为什么新项目、遗留项目、团队项目不能共用同一份模板。

> 注意：AGENTS.md 负责说明项目和交付边界；Rules、Hooks、Approvals 负责把硬约束变成持续生效的控制层。第 6 章先把说明书写对，第 7 章再把护栏接上。

## 6.1 项目地图、关键入口、禁区

很多人写 AGENTS.md，第一段就从“本项目致力于……”开始。对 Agent 来说，这类背景介绍通常价值不高。真正决定任务成败的，是它能否在前几分钟内回答三个问题：项目主要由哪些部分组成，应该先读哪些入口文件，哪些地方碰了就可能出事故。没有这张地图，Agent 往往会用遍历搜索来补信息，成本高，误触风险也高。

一份能工作的项目地图，至少要包含六个字段。第一，系统概览，用三到五行讲清项目做什么、主要技术栈是什么、运行边界在哪里。第二，关键目录，用列表标出 `backend/`、`frontend/`、`knowledge_base/`、`alembic/`、`monitoring/` 这类高频目录的职责。第三，主要入口文件，比如 `README.md`、`Makefile`、`pyproject.toml`、`frontend/package.json`、`backend/main.py`、`docker-compose.yml`。第四，核心命令，包括安装、启动、测试、构建、RAG 初始化和数据库迁移。第五，验证方式，明确“改前端至少跑什么，改 RAG 至少测什么，改数据库至少检查什么”。第六，禁区，也就是绝不能靠猜去改的区域。

这里最容易写空的是“禁区”。很多仓库只写一句“请谨慎修改生产相关文件”，这等于没写。禁区必须是可执行约束，例如：“涉及 `knowledge_base/clinical_guidelines/` 和危机干预话术时，先给出风险说明，不直接改结论”；“涉及 `alembic/versions/` 和数据迁移时，先给迁移计划，再等待人工确认”；“不要直接删除或重建 `chroma_db/`、`context_storage/`、`uploads/` 这类运行数据目录”；“没有明确要求时，不要批量升级依赖或改格式化配置”。这种写法的价值在于，Agent 不需要猜“谨慎”是什么意思。

不过，AGENTS.md 在这里承担的是“说明边界”，不是“强制执法”。如果某条禁区规则必须稳定生效，后面还要交给 Rules、Hooks 或 Approvals 去阻断和确认。

如果直接写进 AGENTS.md，一个最小片段更像代码清单 6-1。它把目标、目录、入口、命令、验证和禁区压缩到同一个最小说明块里，适合成为 `emotional_chat` 第一版根目录说明书的起点。

**代码清单 6-1：`AGENTS.md` 的最小项目说明片段**

```markdown
## 项目目标
`emotional_chat` 是一个情感陪伴聊天机器人，核心链路是用户输入、情感/意图识别、记忆与 RAG 检索、共情回复生成。

## 关键目录
- `backend/`：FastAPI 后端、Agent Runtime、RAG、记忆、插件和接口
- `frontend/`：React 前端界面、聊天组件、上传和流式展示
- `knowledge_base/`：心理健康、危机干预、自助工具等知识材料
- `alembic/`：数据库迁移
- `monitoring/`：Prometheus 和 Grafana 配置

## 先看这些文件
- `README.md`
- `Makefile`
- `pyproject.toml`
- `backend/main.py`
- `backend/runtime/README.md`
- `frontend/package.json`

## 常用命令
- `make install-uv`
- `make run`
- `make rag-init`
- `make db-current`
- `pytest backend/tests`
- `cd frontend && npm test`

## 验证要求
- 后端改动至少运行相关 pytest
- 前端改动至少运行对应测试或给出手动验证路径
- RAG 或知识库改动至少说明影响范围，并运行最小检索验证
- 数据库迁移改动必须说明 upgrade/downgrade 路径

## 禁区
- 不要直接删除 `chroma_db/`、`context_storage/`、`uploads/` 中的运行数据
- 不要随意改写危机干预和临床建议内容
- 不要改历史迁移文件
- 不要在未说明原因时批量升级依赖
```

代码清单 6-1 看起来简单，但每一项其实都在回答一个不同的问题。表 6-1 把这些字段和它们对应的执行含义拆开，方便你判断哪些内容不能省。

**表 6-1：`AGENTS.md` 中各字段的执行含义**

| 字段 | 回答什么 | 为什么不能省 |
| --- | --- | --- |
| 项目目标 | 这是个什么系统，当前主链路是什么 | 没有目标，Agent 不知道什么是“相关” |
| 关键目录 | 主要代码和高风险区域分别在哪 | 没有地图，就只能全仓搜索 |
| 先看这些文件 | 初次进入时从哪里起步 | 没有入口，会浪费很多探索轮次 |
| 常用命令 | 改完后跑什么 | 没有验证入口，闭环就断了 |
| 验证要求 | 什么叫“算完成” | 否则只会得到“我觉得可以” |
| 禁区 | 哪些地方不能靠猜 | 没有禁区，主动性会越界 |

这也是为什么 AGENTS.md 最怕的不是写得短，而是写得抽象。只要字段能直接回答执行问题，文档很短也能有效；反过来，哪怕写了三页愿景，如果不能告诉 Agent 去哪看、跑什么、别碰什么，它依然会在仓库里盲走。

为了看清“介绍项目”和“指导执行”的差异，可以看表 6-2。它把愿景式写法和执行式写法放在一起对比。

**表 6-2：愿景宣言与执行说明的写法对比**

| 写法 | 看起来像什么 | 对执行有没有帮助 |
| --- | --- | --- |
| “本项目致力于打造稳定、灵活、面向未来的平台能力。” | 愿景宣言 | 几乎没有，Agent 仍不知道先看哪里 |
| “先看 `backend/main.py`、`backend/routers/` 和 `backend/services/`，不要直接改 `chroma_db/`，接口改动跑相关 pytest。” | 执行说明 | 直接有帮助，任务可以起步 |

前一种写法不是完全没价值，但更适合 README 或项目介绍。后一种写法虽然朴素，却能直接减少无效搜索、误改和错测。AGENTS.md 要优先服务后者。

如果想把 6.1 节进一步落到可操作层，可以直接使用表 6-3 这样的“首次进入仓库 60 秒检查表”。它的作用不是替代完整说明书，而是把第一次阅读顺序固定下来。

**表 6-3：首次进入仓库的 60 秒检查表**

| 60 秒内先看什么 | 目的 |
| --- | --- |
| README.md 或项目概览 | 确认系统是什么 |
| 一个主入口文件 | 确认代码从哪里开始 |
| 一个测试或脚本入口 | 确认验证从哪里开始 |
| 禁区列表 | 确认哪些地方不能靠猜 |

很多说明书之所以没用，不是字段不存在，而是字段排布没有优先级。把最影响方向判断的信息放到最前面，本身就是一种执行优化。

失败案例通常也很典型。比如 `emotional_chat` 没有把 `alembic/versions/` 标成高风险区域，Agent 为了修一个字段缺失问题，直接改写了旧迁移文件，本地测试通过，但新老环境状态可能因此不一致。再比如说明书没有告诉 Agent 前端测试入口在 `frontend/package.json`，它只在根目录跑了后端测试，以为验证完成，实际上聊天界面的上传、流式输出或 Markdown 渲染根本没被检查。问题不在模型“笨”，而在说明书没有把路线和禁区写清楚。

还有一个常见失误，是把 AGENTS.md 写成 README 的重复副本。README 面向人类读者，强调如何理解和使用项目；AGENTS.md 面向执行者，强调先后顺序、风险边界和交付标准。前者可以多讲背景，后者必须优先讲动作。判断一份 AGENTS.md 有没有写对，可以看一个标准：陌生人第一次接任务时，是否能在不追问的情况下知道“先看哪里、别碰哪里、改完怎么证明确实有效”。

所以 6.1 的核心不是“介绍项目”，而是降低方向性错误。项目地图负责让 Agent 快速定位，关键入口负责减少无效搜索，禁区负责把主动性拦在事故之前。只要这三块写扎实，后面的规则分层和协作边界才有落点。

## 6.2 根目录与子目录分层

当项目只有一个服务、一个测试命令时，单份 AGENTS.md 通常够用。但 `emotional_chat` 这样的仓库已经不是单一服务：它有后端运行时、前端界面、知识库、数据库迁移、监控配置和本地运行数据。所有规则都堆在根目录就会迅速失控。最常见的症状是两种：要么文档越来越长，Agent 找不到重点；要么根目录写得太粗，局部模块的特殊约束根本落不进去。

因此，复杂项目通常需要分层，尤其当局部目录有独立命令、风险边界或交付要求时。根目录 AGENTS.md 只管全局共识，回答“这个仓库整体怎么工作”；子目录 AGENTS.md 只管局部差异，回答“进入这一块以后，额外要注意什么”。根目录适合放项目目标、统一命令、统一提交流程、全局禁区、跨模块验证要求。子目录适合放模块职责、局部入口、局部脚本、依赖限制、特殊验收标准。这样写，Agent 在根目录先建立大图，再在进入具体目录时加载局部补充，不需要在一份大文档里来回找。

如果把这种分层关系画出来，可以得到图 6-1。它强调的是“根目录先给全局共识，子目录再补局部差异”，而不是要求每个目录都必须放一份文件。

**图 6-1：根目录与子目录 AGENTS.md 的分层关系**

```text
emotional_chat/
  AGENTS.md                  # 全局共识
  backend/
    AGENTS.md                # 后端、Agent Runtime、RAG 局部差异
  frontend/
    AGENTS.md                # 前端局部差异
  knowledge_base/
    AGENTS.md                # 心理健康知识库和危机干预内容
  monitoring/
    AGENTS.md                # 监控配置局部差异
```

一个实用的直觉规则是“继承优先，覆盖从简”。也就是：子目录默认继承根目录规则，只有在本地差异确实存在时才补充或覆盖，而且要明确写出覆盖点。比如根目录规定“后端改动至少跑相关 pytest”，而 `frontend/AGENTS.md` 可以补一句“前端 UI 改动至少运行 `npm test` 或给出手动验证路径”；`knowledge_base/AGENTS.md` 则可能覆盖成“心理健康和危机干预内容只允许提出修改建议，不直接替换专业结论”。不要把根目录内容整段复制到子目录，这样一旦全局规则更新，就会出现多个副本互相打架。

图 6-1 说明的是抽象层次。落到 `emotional_chat`，可以进一步写成代码清单 6-2 这样的目录示意，让读者看到说明书会具体放在哪些位置。

**代码清单 6-2：`emotional_chat` 中的 AGENTS.md 分层示意**

```text
emotional_chat/
  AGENTS.md
  backend/
    AGENTS.md
    runtime/
    services/
    routers/
    tests/
  frontend/
    AGENTS.md
    src/
  knowledge_base/
    AGENTS.md
    clinical_guidelines/
    organization_policy/
  monitoring/
    AGENTS.md
```

代码清单 6-2 里，根目录说明的是“整个仓库怎样工作”，而各个子目录说明的是“进入这一块以后，额外要注意什么”。只要把这两个层次混在一起，Agent 就会不知道哪些规则全局生效，哪些只在局部生效。

下面这个结构就比较典型：

**根目录 AGENTS.md**

- 仓库概览：情感陪伴机器人、FastAPI、React、RAG、向量数据库、Agent Runtime
- 顶层目录职责：`backend/`、`frontend/`、`knowledge_base/`、`alembic/`、`monitoring/`
- 统一命令：`make install-uv`、`make run`、`make rag-init`、`pytest backend/tests`
- 全局禁区：运行数据、历史迁移、危机干预内容、依赖批量升级
- 交付摘要格式

**frontend/AGENTS.md**

- 聊天容器、侧边栏、上传组件、Markdown 渲染和流式输出入口
- `npm start`、`npm test`、`npm run build`
- 不要顺手改主题、全局样式和构建配置

**backend/AGENTS.md**

- FastAPI 入口、routers、services、runtime、agent、plugins、schemas
- pytest、RAG 初始化、数据库迁移检查
- 涉及记忆、危机检测、插件调用、迁移脚本时必须提示风险

**knowledge_base/AGENTS.md**

- 心理健康知识、危机干预协议、自助工具说明
- 内容修改必须说明来源、影响范围和人工复核点
- 不把模型生成内容直接当作专业结论落地

**monitoring/AGENTS.md**

- Prometheus、Grafana 配置入口
- 修改监控指标或面板时说明影响范围
- 不顺手改告警阈值，除非任务明确要求

表 6-4 进一步把“继承”和“覆盖”的分工讲清楚。它适合放在团队讨论 AGENTS.md 分层时使用，避免所有规则都挤进根目录。

**表 6-4：根目录与子目录的规则分工**

| 规则类型 | 根目录更适合写什么 | 子目录更适合补什么 |
| --- | --- | --- |
| 项目目标 | 仓库整体目标、统一交付原则 | 模块局部职责 |
| 命令入口 | 全局安装、后端启动、统一测试入口 | 前端测试、RAG 初始化、迁移检查、监控验证 |
| 风险边界 | 全局禁区、统一审批点 | 知识库、记忆、迁移、UI、监控各自的危险区域 |
| 交付格式 | 团队统一摘要字段 | 前端补截图，RAG 补检索样例，迁移补 upgrade/downgrade |

这套分层还有一个容易忽略的好处：它天然减少规则冲突时的争论成本。比如前端想补视觉回归要求，后端想强调迁移风险，只要各自写在模块说明里，就不必挤进根目录抢优先级。根目录只保留真正全局的东西，局部差异则留在局部解决。

这种分层还有一个好处：它天然适合多人协作。前端 owner 只需维护 `frontend/AGENTS.md`，后端 owner 维护 `backend/AGENTS.md`，熟悉心理健康内容的人维护 `knowledge_base/AGENTS.md`，根目录由仓库 owner 或项目维护人维护统一规则。这样既减少中心化瓶颈，也避免某个不了解模块的人往全局文档里写错局部约束。

失败案例往往出在两个极端。第一种是完全不分层，导致根目录文档里塞进“前端颜色 token 不可改”“数据迁移需先 review”“定时任务不要在 9 点跑”这类互不相干的细节，Agent 读完仍不知道当前任务最相关的是什么。第二种是过度分层，每个小目录都放一份 AGENTS.md，内容高度重复，最后没人知道以哪份为准。工程上更合理的做法是，只有当局部差异真的会影响执行路径时，才增加子目录说明。

所以不是目录一多就必须加文件，而是“局部差异是否大到足以改变执行方式”。如果只是多了几个普通子目录，根目录说明通常还够；只有当某个子目录拥有独立命令、独立风险边界、独立交付要求时，子目录 AGENTS.md 才真的有存在价值。

真遇到根目录规则和子目录规则冲突时，也最好直接写明优先级。例如：根目录要求“后端改动提交前跑相关 pytest”；但 `knowledge_base/AGENTS.md` 可以覆盖成“危机干预内容默认只提出建议，不直接改正文，除非有人类复核结论”。这种冲突不是坏事，坏的是冲突存在却没人说明。只要优先级写出来，Agent 就知道什么时候该继承，什么时候该局部覆盖。

这一节最需要读者建立的判断，不是“是否支持分层”这种形式问题，而是“哪些知识属于全局，哪些知识只在本模块成立”。把作用域写对，规则才不会互相打架；把继承和覆盖讲清楚，Agent 才能在复杂仓库里保持稳定的执行方向。

## 6.3 团队规则与个人偏好

团队刚开始整理 AGENTS.md 时，经常会出现一种微妙失真：谁先动笔，谁就容易把自己的工作习惯写成“项目标准”。比如“提交摘要必须先写背景再写方案”“回答时尽量分三点”“代码里多写注释”“优先用我熟悉的测试框架”。这些偏好未必错，但它们不一定都值得变成团队硬规则。一旦文档里硬规则过多、过细，大家很快就会把它当成参考而不是约束。

实用的拆法是把内容分成三层。第一层是团队硬规则，必须遵守，违反就会造成风险、返工或协作失败。第二层是团队建议，能提高一致性，但不必每次都强制。第三层是个人偏好，只在个人工作流里有价值，不该绑架整个项目。AGENTS.md 主要承载第一层，少量承载第二层；第三层更适合留给个人配置、私有模板或后面会讲到的记忆系统，而不是直接写进公共说明书。

什么算硬规则？通常有四类。第一类是质量门槛，例如必须跑哪些测试、哪些目录改动必须附验证结果。第二类是风险边界，例如禁止修改哪些文件、遇到哪些操作必须停下等待审批。第三类是协作协议，例如提交说明要包含变更范围、风险和未决问题。第四类是命名或接口兼容性这类真正影响多人协作的约束。什么不算硬规则？诸如“更喜欢短句摘要”“变量名尽量简短”“思考过程展示得详细一点”这类更像写作与沟通风格，属于建议或个人偏好。

这里还要提前和第 7 章的控制栈划清边界。AGENTS.md 适合写“规则说明”，不适合承担“自动执法”。例如“提交前必须跑格式化和单测”可以写进 AGENTS.md，但真正自动执行 prettier、阻断危险命令、扫描敏感文件，应交给 Hooks。再比如“涉及生产配置必须人工确认”可以在 AGENTS.md 写明流程，但真正让某个高风险操作无法静默继续，应交给 Approvals。判断标准很简单：如果一条规则必须稳定执行、不能靠自觉，那它最终要下沉为系统约束。

把三层规则拆成表，会比只讲原则更容易落地。表 6-5 给出的是一个最小分类法：哪些必须遵守，哪些只是建议，哪些应该留在个人偏好里。

**表 6-5：团队规则与个人偏好的分层**

| 层级 | 典型语气 | 例子 | 最适合放哪里 |
| --- | --- | --- | --- |
| 必须 | 必须、禁止、需要审批 | “改危机干预逻辑必须说明风险和验证结果” | AGENTS.md + 后续系统约束 |
| 建议 | 建议、优先、尽量 | “优先沿用现有模块风格” | AGENTS.md |
| 个人偏好 | 我习惯、我更喜欢 | “我喜欢三段式摘要” | 个人配置、私有模板 |

一个常见的失败案例是，团队把“优先写测试”“尽量少改无关代码”“回答简洁一些”“提交信息用中文”“注释遵循某某风格”全部写成同一等级的 must。结果 Agent 无法区分哪些是事故级要求，哪些只是表达偏好。到最后，真正重要的边界反而被淹没。更糟的是，一旦现实任务需要取舍，比如为了快速止血先修 bug 再补重构，就会出现“文档全都写了，但没人真按它执行”的情况。

比较稳妥的写法，是直接在文档里标注语气和优先级。示例 6-1 展示了这种写法在 `emotional_chat` 中可以怎样落地。

**示例 6-1：`emotional_chat` 的规则分级写法**

- 必须：修改 `backend/routers/`、`backend/services/` 或 `alembic/` 时，附上执行过的测试或检查命令。
- 必须：涉及危机干预、临床建议、密钥、数据库迁移时，先停在方案说明，不直接落地修改。
- 建议：变更摘要按“做了什么 / 怎么验证 / 还有什么风险”三段写。
- 建议：优先沿用现有模块风格，不引入新框架。
- 个人偏好，不进公共 AGENTS.md：回复字数、注释颗粒度、个人常用命令别名。

这样写的好处是，团队能看懂哪些内容会影响安全和交付，哪些只是为了提升协作体验。AGENTS.md 不是道德守则，而是执行说明书。说明书最重要的品质不是面面俱到，而是优先级清楚。只有把团队规则和个人偏好拆开，后面的 Rules、Hooks、Approvals 才有明确的落点，不会和文档层互相抢职责。

很多团队第一次写坏 AGENTS.md，都不是因为技术不够，而是因为一个很常见的小故事：最先动笔的人把自己平时觉得“顺手”的做法都写了进去，后面的人又往上继续叠。结果文档里同时出现“涉及生产必须审批”和“回复尽量简洁一些”这种完全不同等级的要求。真正危险的，不是规则多，而是事故级规则被风格偏好稀释掉。

更具体地说，团队里最容易打架的通常不是“要不要写规则”，而是“谁有资格把自己的习惯升级成公共规则”。一个够用的判断方法是反问三次：不这么做会不会造成风险，会不会造成返工，会不会造成多人协作失败。只有至少命中其中一项，才值得进入公共 AGENTS.md。否则，它大概率更适合停留在建议层或个人层。

这类冲突如果不提前拆开，后面就会出现一种很隐蔽的执行偏差：Agent 会非常认真地服从所有文字，却分不出哪些要求真的不能违反，哪些只是“更理想的表达方式”。从工程角度看，这比没有规则更糟，因为它会让表面合规掩盖真正的优先级错误。

## 6.4 角色分工、交接协议与值班边界

这一节主要面向已经进入团队协作、云端任务或值班场景的项目；个人项目可以先只保留最小交接格式，不必一开始就写完整的角色和值班规则。

> 讨论点：这一句是新补的缓冲。原稿直接进入夜间告警和值班边界，逻辑没错，但从第 5 章的“最小闭环”跳过来会稍重。

只要任务开始跨人、跨时段、跨环境流转，AGENTS.md 就不再只是“项目地图”，它还会变成一份交接协议。很多团队白天看起来协作顺畅，是因为 owner 都在线，大家默认知道该找谁、哪些事谁能拍板。到了夜间值班、异步 review、云端长任务这些场景，这些默认知识一旦没写下来，Agent 最容易卡在两个极端：要么过于保守，什么都不敢继续；要么过于积极，把本该升级处理的风险也一并执行了。

最典型的小故事是夜里两点的告警。值班同事不在线，自动化任务发现错误率升高，Agent 能收集日志，也能定位最近一次改动，但没人写清楚它接下来有没有权回滚、有没有权重建向量库、遇到危机干预链路该找谁确认。真正让事故放大的，往往不是技术复杂度，而是交接协议和放权边界根本没写出来。

因此，第 6.4 节要解决的是“谁负责、谁接手、谁兜底”。一份靠谱的 AGENTS.md 应该至少把角色分成三类。第一类是 owner，对某个模块、系统或流程有最终拍板权。第二类是 reviewer，可以审核常规改动，确认技术方向和验证结果。第三类是 oncall 或值班角色，负责在特定时段处理告警、止血和升级。不是每个团队都需要用这几个英文名，但职责边界必须清楚。

可以先用表 6-6 这样的角色矩阵把边界压实。它不追求组织结构完整，而是回答“谁能拍板，谁必须升级”。

**表 6-6：角色职责与升级边界**

| 角色 | 主要职责 | 可以直接拍板什么 | 必须升级什么 |
| --- | --- | --- | --- |
| Owner | 模块或系统最终责任 | 常规方向、模块内取舍 | 生产高风险、跨模块重大变更 |
| Reviewer | 审核常规改动 | 范围、验证、说明是否合格 | 发现风险超出常规边界时 |
| On-call | 值班止血和升级 | 日志收集、状态检查、失败复现 | 生产写操作、数据删除、权限变更 |

除了角色名，更关键的是交接格式。Agent 在完成阶段性任务后，不应该只丢一句“已经改好了”。更有用的交接块通常包括五项：结果摘要、变更范围、验证结果、未决风险、需要人工确认的点。如果任务跨接手者分工，还应补一项“建议下一位先看哪些文件和上下文”。这种格式的价值不在好看，而在防止下一位接手的人再做一轮无效搜索。很多多人协作的低效，不是因为不会写代码，而是因为交接内容只说结论，不说边界。

> 讨论点：原稿里这里写“跨 worker 分工”。先改成“跨接手者分工”，更贴近中文正文。

如果你想让这件事立即可执行，交接块可以先写成代码清单 6-3。这个例子对应的是 `emotional_chat` 里一个常规后端修复任务。

**代码清单 6-3：常规任务的交接块示例**

```markdown
## 任务交接
- 结果摘要：修复了情绪趋势接口在空会话时返回 500 的问题，补了 1 条回归测试。
- 变更范围：`backend/routers/emotion_analysis.py`、`backend/tests/unit/test_core.py`
- 已跑验证：相关 pytest 通过，手动检查接口空数据返回结构
- 未决风险：没有覆盖真实用户长会话的趋势聚合表现
- 需要人工确认：是否要补一条集成测试覆盖前端趋势面板
- 下一位先看：接口返回 schema、本次测试 diff、前端调用位置
```

如果把场景换成值班中的线上问题，交接块还应该更偏故障处理。代码清单 6-4 展示的是聊天接口和 RAG 链路出问题时的一种交接写法。

**代码清单 6-4：值班故障交接块示例**

```markdown
## 值班交接
- 现象：01:35 后聊天接口错误率升高，主要集中在 RAG 检索后的回复生成
- 当前动作：已收集最近 30 分钟日志，已确认基础健康检查正常
- 已验证：数据库连接正常，向量库目录可读，普通非 RAG 对话可返回
- 未做动作：未删除 `chroma_db/`，未重建知识库，未修改危机干预内容
- 风险判断：疑似与最近一次知识库初始化或依赖变更有关，但证据不足
- 必须升级：涉及用户情感支持链路和知识库内容，需 owner 确认是否回滚或重建索引
```

这样的交接块有一个关键价值：它明确区分“已经做了什么”和“故意没有做什么”。后者特别重要，因为很多接手失败，不是因为信息太少，而是因为下一位不知道前一位为了守边界刻意停在了哪里。

这一点在值班和事故处理中尤其关键。白天的开发任务里，“没做某件事”也许只是待办；但在夜间告警里，“没做某件事”往往是在主动守边界。比如“没有回滚”“没有改配置”“没有清队列”，这些都不是缺动作，而是为了把最终决定留给有拍板权的人。交接块把这层意图写出来，下一位就不会误把克制当成遗漏。

值班边界更要写得直白。比如：“允许在夜间自动跑只读诊断和测试，不允许自动修改生产配置”；“遇到线上故障时，Agent 可以收集日志、定位最近改动、生成回滚建议，但执行回滚前必须等待 oncall 或 owner 确认”；“如果验证失败且原因不明，停止继续尝试，提交失败摘要和可疑范围”。这些句子看起来保守，但它们实际上是在给自动化留出安全工作区。没有这个边界，“先止血再说”很容易演变成“先把现场改乱再说”。

同样要记住：AGENTS.md 能把值班边界写清楚，但不能替代真正的权限控制。凡是涉及生产写操作、数据删除、危机干预内容修改的动作，最终都应该在第 7 章讲的控制层里落实为审批或阻断。

把升级条件再写成表 6-7，会更容易真正执行。它把“遇到风险要升级”从口号变成了触发条件、可做动作和禁止动作。

**表 6-7：值班与高风险动作的升级条件**

| 触发条件 | Agent 应做什么 | 不能做什么 |
| --- | --- | --- |
| 涉及数据删除 | 生成影响说明和回滚建议 | 不直接执行删除 |
| 涉及生产配置 | 收集上下文、输出计划 | 不静默落地改动 |
| 无法解释的测试失败 | 提交失败摘要、可疑范围 | 不无限重试 |
| 涉及危机干预、用户记忆、权限、密钥 | 停在方案说明和风险点 | 不替代 owner 做最终判断 |

一个很常见的失败案例是，团队允许 Agent 自动处理告警，却没说明它有权做到哪一步。结果某次夜间任务里，Agent 发现某个部署后错误率升高，自动把配置和依赖都回退了，第二天团队根本说不清是服务本身故障，还是自动化动作引入了新变量。问题不在于“自动化不能做事”，而在于交接和升级边界没写清楚。

除了表格，也可以把这部分压缩成示例 6-2 这样的模板化文字，直接放进团队项目的 AGENTS.md。

**示例 6-2：`emotional_chat` 的协作边界写法**

- Owner：危机干预、临床建议、数据迁移和生产配置变更必须由 owner 最终确认。
- Reviewer：常规业务改动可由 reviewer 审核后合并，但需检查验证结果是否完整。
- On-call：允许执行日志收集、状态检查、失败复现；不允许静默执行生产写操作。
- 交接格式：摘要、涉及文件、已跑命令、结果截图或日志、剩余风险、阻塞点。
- 升级条件：涉及数据删除、权限变更、生产配置、成本激增、无法解释的测试失败时立即升级。

这一节的底层逻辑很简单：多人协作不是靠默契维持，而是靠协议维持。AGENTS.md 不需要把组织结构图全抄进去，但必须把会影响执行权限和交接效率的边界写明。这样，Codex 参与的长任务、云端任务和异步值班，才不会因为“谁该接手、谁能拍板、出了事谁兜底”这些问题失控。

真正成熟的交接协议不是为了写得漂亮，而是为了降低凌晨接手成本。下一位不需要认识你，也不需要参与过这个任务，只要看交接块就能知道：事情做到哪一步了，哪些已验证，哪些不能碰，为什么现在必须升级给某个人。这就是多人协作里最实用的文档价值。

这也是为什么 AGENTS.md 里应该尽量避免“有问题及时沟通”这类口号式表达。真正能在夜间、异步、跨班次场景里发挥作用的，不是态度宣言，而是明确的接手格式、升级条件和停止线。

## 6.5 三类模板：新项目 / 遗留项目 / 团队项目

到这里，读者已经知道 AGENTS.md 应该写哪些字段，也知道要和后续 Rules、Hooks、Approvals 分层。但还有最后一个现实问题：到底该从哪份模板起步？如果所有项目都用同一份说明书，很快就会发现它要么太轻，要么太重。真正影响模板结构的，不是技术栈，而是项目成熟度和协作复杂度。

第一类是新项目模板。它的目标不是穷举未来，而是给 Agent 一个稳定起步面。字段应当尽量轻，只保留：项目目标、目录说明、核心命令、验收方式、禁区、交付格式。新项目最常见的错误不是“信息太少”，而是“还没形成事实就写了太多伪规则”。比如架构尚未定型，就把未来可能会改的目录约束写死；测试体系还没稳定，就强制一套没人真正执行的流程。新项目模板要短，方便随着项目演进快速改。

第二类是遗留项目模板。它比新项目更重，重点不是“怎么开始”，而是“哪些历史包袱千万别踩”。遗留项目通常会有旧脚本、半废弃目录、迁移遗留、人工运营流程、无法完全自动验证的模块。这里最值钱的字段不是愿景，而是风险清单：哪些目录看起来能改其实别动，哪些测试不稳定，哪些功能依赖历史数据，哪些模块只有特定同事能判断。遗留项目里，AGENTS.md 更像一份排雷图。

第三类是团队项目模板。它在前两者基础上，还必须补上角色、审批、值班、权限入口和共享能力包位置。因为团队项目的主要风险，已经不是单次误改，而是“多个人和多个 Agent 同时工作时怎么不打架”。这类模板里，除了项目地图和禁区，还应出现：谁维护根目录说明书，子目录由谁负责，什么操作必须 reviewer 或 owner 介入，交接摘要怎么写，共享脚本和技能包放在哪里。

先把三类模板放到同一张表里，会更容易选。表 6-8 的重点不是分类本身，而是提醒你：模板要跟项目成熟度和协作复杂度匹配。

**表 6-8：新项目、遗留项目与团队项目模板的选择**

| 模板类型 | 主要问题 | 最该写清的字段 | 最常见误用 |
| --- | --- | --- | --- |
| 新项目 | 怎么稳定起步 | 目标、目录、命令、禁区、交付格式 | 未来事实未定就写死很多伪规则 |
| 遗留项目 | 哪些历史包袱别踩 | 风险区域、脏点、人工流程、现有验证 | 只讲愿景，不讲排雷信息 |
| 团队项目 | 多人多 Agent 如何协作 | 分层说明、角色、审批、值班、共享入口 | 把协作协议写成代码规范清单 |

如果只讲分类，读者还是很难落手。更有用的方式，是给出三份最小骨架。代码清单 6-5、代码清单 6-6 和代码清单 6-7 分别对应新项目、遗留项目和团队项目，它们不是完整答案，但足够让你复制后按项目改。

### 6.5.1 新项目最小骨架

新项目阶段的 AGENTS.md 不应该急着写成“大而全”的制度文档。以 `emotional_chat` 的早期版本为例，此时最重要的是让 Agent 知道项目目标、主要目录、启动入口和最小禁区，而不是提前规定还没稳定下来的架构细节。代码清单 6-5 给出的就是这种轻量版本：它只保留第一次进仓库最容易用到的信息，方便后续随着项目演进继续补充。

**代码清单 6-5：新项目的最小 `AGENTS.md` 骨架**

```markdown
# AGENTS.md

## 项目目标
- 这是 `emotional_chat` 的早期版本，当前最核心的用户路径是聊天输入、情感识别、回复生成。

## 关键目录
- `backend/`：后端接口和服务
- `frontend/`：前端聊天界面
- `knowledge_base/`：RAG 知识材料
- `docs/`：运行说明和章节草稿

## 先看这些文件
- `README.md`
- `Makefile`
- `pyproject.toml`
- `backend/main.py`
- `frontend/package.json`

## 常用命令
- 安装后端依赖：`make install-uv`
- 启动后端：`make run`
- 初始化 RAG：`make rag-init`
- 前端启动：`cd frontend && npm start`

## 禁区
- 未明确要求时，不要批量升级依赖、改数据库迁移历史、删除本地运行数据。

## 交付格式
- 说明改了什么、跑了什么验证、还有什么风险。
```

这一版的目标只有一个：让 Agent 第一次进仓库时不至于迷路。它故意写得轻，因为新项目最怕的不是约束太少，而是事实还没稳定，规则已经写死。

### 6.5.2 遗留项目最小骨架

当 `emotional_chat` 经过多轮功能演进后，AGENTS.md 的重点就会从“帮 Agent 起步”变成“防止 Agent 踩雷”。这时仓库里可能同时存在旧版 agent 目录、新版 runtime 架构、历史迁移文件、本地向量库和专业知识材料。代码清单 6-6 的作用，就是把这些容易误判的地方提前写出来，让 Agent 在动手前先知道哪些区域需要确认、哪些内容只能提出方案。

**代码清单 6-6：遗留项目的最小 AGENTS.md 骨架**

```markdown
# AGENTS.md

## 项目现状
- 这是一个已经多次演进的 `emotional_chat` 仓库，既有旧版 `backend/agent/`，也有新版 `backend/runtime/` 和 `backend/modules/`，优先保持现有行为稳定。

## 风险区域
- `alembic/versions/`：不要改旧迁移文件
- `backend/runtime/`：涉及对话生命周期、预算、策略、hooks，改动前先说明影响范围
- `backend/services/` 与 `backend/modules/`：存在新旧服务并存，先确认调用链再动
- `knowledge_base/clinical_guidelines/`：涉及专业建议，先提出方案和风险，不直接替换结论
- `chroma_db/`、`context_storage/`、`uploads/`：运行数据目录，不直接删除或重建

## 现有验证路径
- 后端最小测试：`pytest backend/tests`
- 运行检查：`make run`
- RAG 初始化：`make rag-init`
- 前端测试：`cd frontend && npm test`
- 某些依赖模型、音频、图像或外部服务的测试可能需要人工确认环境

## 人工流程
- 涉及数据库迁移、危机干预、临床建议、用户记忆、密钥和生产配置时，先给方案，不直接提交修改

## 已知脏点
- README 信息较多，可能与部分新架构细节不同步
- `backend/agent/` 与 `backend/modules/agent/` 可能存在历史并存关系
- 本地向量库和上传目录可能包含运行态数据
```

遗留项目模板的重点不是“介绍项目多伟大”，而是提前暴露真实脏点。Agent 只要知道哪些地方脆弱、哪些说明已经过期、哪些流程不能自动化，误判率就会显著下降。

### 6.5.3 团队项目最小骨架

当 `emotional_chat` 不再只是个人实验，而是由多人分别维护后端、前端、知识库、监控和部署流程时，AGENTS.md 就不能只回答“从哪里开始看”。它还要回答“谁能拍板、谁负责复核、哪些动作必须停下来、交接时要留下什么”。代码清单 6-7 展示的是团队项目的最小骨架：它把分层说明、角色审批、交接格式和值班边界放在同一份说明书里，让多个 Agent 或多位开发者并行时不至于互相踩线。

**代码清单 6-7：团队项目的最小 AGENTS.md 骨架**

```markdown
# AGENTS.md

## 全局目标
- 本仓库由后端 Agent Runtime、React 前端、RAG 知识库、数据库迁移和监控配置组成，默认优先最小变更。

## 分层说明
- 根目录负责全局规则
- `backend/AGENTS.md`、`frontend/AGENTS.md`、`knowledge_base/AGENTS.md`、`monitoring/AGENTS.md` 负责模块差异

## 角色与审批
- 常规业务改动可由 reviewer 审核
- 涉及危机干预、临床建议、数据删除、数据库迁移、生产配置时必须 owner 确认

## 交接格式
- 结果摘要
- 变更范围
- 已跑验证
- 未决风险
- 是否需要人工继续接手

## 值班边界
- 允许日志收集、失败复现、只读检查
- 不允许静默删除运行数据、重建向量库、修改危机干预内容或执行生产写操作

## 共享入口
- 后端脚本在 `Makefile`、`backend/scripts/` 和根目录启动脚本
- 前端入口在 `frontend/package.json`
- 运行文档和章节草稿在 `docs/`
- 心理健康知识材料在 `knowledge_base/`
```

团队项目模板的重点不是代码规范，而是协作协议。只要把角色、审批、交接和值班边界写清楚，多人和多 Agent 才有机会稳定并行。

### 6.5.4 模板怎么选，怎么升级

模板真正的用法，不是原样照抄，而是先选场景，再删到最小可执行版本。新项目先保证能跑通第一轮任务；遗留项目先把事故高发区标出来；团队项目先把权限、角色和交接协议补齐。很多团队失败，是因为一上来就追求“大而全模板”，结果文档没人维护，Agent 也读不出优先级。

如果必须给一个落地建议，我会这样排顺序：个人或小团队从新项目模板起步；接手老系统时优先套遗留项目模板，把风险点写出来；当仓库开始多人协作、需要值班或审批时，再升级到团队项目模板。模板升级的触发信号，也可以直接写进项目维护清单里，例如“开始出现跨模块误改”“开始需要夜间值守”“开始依赖共享技能包”。AGENTS.md 最有价值的状态，从来不是最完整，而是“当前团队真的会维护、Agent 真的会用、出问题时真的能拦住”的那个版本。

> 讨论点：这里把“主线程维护清单”改成了“项目维护清单”，同样是为了减少术语跳跃。

如果你还拿不准该从哪一份起步，可以按这个顺序自问：

- 现在最主要的问题，是“没法起步”、还是“容易踩雷”、还是“多人会打架”？
- 如果明天换一个人来接手，最容易丢失的是入口信息、风险信息，还是协作边界？
- 目前仓库最大的损失，来自误改、误判，还是交接失败？

这三个问题通常已经足够帮你选出第一版模板，而不必把三份模板揉成一份“大而全说明书”。

模板升级信号也可以直接写成表 6-9 这样的维护表。它帮助团队判断什么时候该从轻量说明书升级到遗留项目模板或团队协作模板。

**表 6-9：AGENTS.md 模板的升级信号**

| 出现了什么信号 | 应该怎么升级模板 |
| --- | --- |
| 新人或 Agent 经常不知道从哪里起手 | 给新项目模板补入口文件和命令 |
| 经常踩到历史脏点或过时脚本 | 升到遗留项目模板，补风险区和脏点 |
| 开始出现多人并行和跨模块接力 | 升到团队模板，补角色和交接协议 |
| 夜间值守或自动化参与越来越多 | 团队模板里补值班边界和升级条件 |
| 共享脚本、共享技能包开始增多 | 团队模板里补共享入口位置和维护人 |

换句话说，模板不是按“公司规模”升级的，而是按“协作复杂度”升级的。一个十个人的小团队，如果已经有夜间值守、共享技能包和多模块接力，就比一个五十人的单服务仓库更需要团队模板。

你甚至可以把模板迁移理解成一个小故事：项目刚开始时，大家只需要知道从哪起手；接着某些历史脏点开始频繁绊人，于是模板变成排雷图；再后来，多人并行、值班和审批都进来了，模板才真正升级成协作协议。模板本身并不神秘，它只是项目复杂度在文档层留下的轨迹。

## 本章小结

第 6 章真正讲清楚的是：AGENTS.md 不是项目介绍文案，而是给 Agent 的执行说明书。它要解决的是非常具体的工程问题：从哪里看、哪些别动、改完跑什么、遇到什么情况必须停下来等人确认。

本章还把几个容易混淆的层次拆开了。项目地图、关键入口和禁区解决的是“初次进入仓库如何不迷路”；根目录与子目录分层解决的是“复杂项目里的规则作用域”；团队规则、交接协议和值班边界解决的是“多人协作时怎样不靠默契硬撑”；三类模板则解决“不同成熟度的项目该从哪一份说明书起步”。

所以第 6 章最值得带走的，不是一份可以原样照抄的万能模板，而是一种写法判断：只写会影响执行方向、风险边界和交接效率的信息；把全局规则和局部差异分开；把硬规则和个人偏好分开；把说明层和自动执法层分开。这样写出来的 AGENTS.md，才会既有人维护，也真能被 Agent 用上。

如果要用一句很实际的话收尾，那就是：每写完一版 AGENTS.md，都问自己一次，陌生人第一次进仓库时，能不能据此知道先看哪里、哪些别动、改完跑什么、出了风险找谁。如果这四个问题还答不完整，说明书就还没有真正完成它的职责。

反过来，只要这四个问题已经能答清，很多后续规则和自动化才会有稳定落点。

说明书写对，后面的护栏才有意义。

这一步不能省。
