# 心语前端 Codex 风格升级计划

## 1. 升级目标

将当前单页聊天界面升级为现代桌面工作台。视觉参考 Codex 的克制、低噪声、三栏布局与浮动输入器，但信息架构服务于“情感陪伴”，不照搬开发工具概念。

核心结果：

- 从“聊天网页”升级为“情感陪伴工作台”。
- 统一颜色、间距、圆角、阴影、字体和交互状态。
- 让历史、记忆、技能、个性化配置成为一等入口。
- 保留现有后端 API 和流式 SSE，降低迁移风险。
- 桌面端优先，同时具备可用的平板与移动端降级布局。

## 2. 目标信息架构

```text
┌────────────左侧导航────────────┬──────────────主工作区──────────────┬────────右侧上下文栏────────┐
│ 心语 / 搜索                    │ 会话标题 / 会话操作                │ 当前情绪                    │
│ + 新对话                       │                                  │ 相关记忆                    │
│ 历史对话                       │ 消息流 / 空状态                    │ 对话建议                    │
│ 记忆                           │                                  │ 附件与链接                  │
│ 技能                           │ 浮动输入器                         │                            │
│ 个性化                         │ 附件 / 深度思考 / 发送              │ 可折叠                      │
│ 用户与主题                     │                                  │                            │
└───────────────────────────────┴──────────────────────────────────┴────────────────────────────┘
```

Codex 结构到心语业务的映射：

| Codex 式入口 | 心语入口 | 说明 |
|---|---|---|
| 新建任务 | 新对话 | 创建独立会话 |
| 任务列表 | 历史对话 | 按日期分组、搜索和管理 |
| 插件 | 技能 | 搜索记忆、情绪趋势、资源推荐等 |
| 项目上下文 | 用户记忆 | 当前用户的长期记忆与画像 |
| 右侧工具栏 | 情绪上下文 | 情绪、记忆、建议和附件 |
| 底部 Composer | 对话输入器 | 输入、附件、模式和发送 |

## 3. 实施原则

1. 先修状态，再换布局：先解决 session 同步、历史刷新和错误态，再迁移 UI。
2. 设计令牌优先：禁止继续在组件中散落临时颜色与透明度。
3. 渐进替换：保留现有 API，按区域替换组件，始终保持主分支可运行。
4. 关键路径优先：新建对话、发送、流式响应、历史恢复必须先稳定。
5. 可访问性内建：键盘操作、焦点、对比度和减少动画不是收尾项。

## 4. 分阶段计划

### Phase 0：基线与风险收敛（1–2 天）

- 建立当前关键流程清单和截图基线。
- 修复流式完成后历史列表不刷新。
- 统一 `useChat` 与 `useSession` 的 session ID 来源。
- 确认空会话、失败响应和中断请求不产生脏历史。
- 为聊天、历史、技能、个性化建立最小冒烟测试。

验收：首条消息完成后立即出现在历史列表，刷新后能恢复同一会话。

### Phase 1：设计系统与应用骨架（2–3 天）

- 新增 `tokens.js` 或全局 CSS variables：颜色、字号、间距、圆角、阴影、层级。
- 新增 `AppShell`、`NavigationRail`、`Workspace`、`ContextRail`。
- 支持左右栏折叠、窗口宽度断点和主题切换。
- 将 Modal、Button、IconButton、Input、Tooltip 抽为基础组件。

验收：三栏骨架在 1440、1024、768 三档宽度不溢出；浅色、深色均有完整背景。

### Phase 2：左侧导航与历史会话（3–4 天）

- 重做品牌区、新对话、搜索、一级导航。
- 历史会话按“今天 / 昨天 / 近 7 天 / 更早”分组。
- 支持搜索、选中态、重命名、删除和管理入口。
- 用户与主题入口固定在底部。

验收：历史加载、选择、搜索、删除、新建均可键盘操作；当前会话状态一致。

### Phase 3：聊天主工作区与输入器（4–5 天）

- 新增空状态欢迎区，降低首屏信息噪声。
- 消息区域使用适合长文本的最大宽度和阅读排版。
- 底部输入器改为浮动卡片，整合附件、深度思考、语音和发送。
- 完善流式光标、停止生成、重试、复制、反馈与错误提示。
- Markdown、代码块、附件和 URL 内容使用统一卡片规范。

验收：发送到首 token 有明确反馈；生成可停止；失败可重试且不重复用户消息。

### Phase 4：右侧情绪上下文栏（3–4 天）

- 当前情绪卡：标签、强度、趋势。
- 相关记忆卡：展示本轮引用的记忆，支持查看来源。
- 对话建议卡：快捷填入但不自动发送。
- 附件与链接卡：展示当前会话上下文。
- 无数据时使用轻量空状态，整栏可折叠。

验收：右栏内容与当前会话绑定，切换历史会话后不串数据。

### Phase 5：技能、记忆与个性化工作区（4–5 天）

- 将技能中心从大型弹窗升级为工作区页面或抽屉。
- 记忆支持搜索、类型筛选、重要性修改和删除确认。
- 个性化配置使用分组表单，提供保存状态和撤销反馈。
- 历史管理复用统一列表与批量操作组件。

验收：所有现有功能在新导航中可达，不再依赖多层透明弹窗。

### Phase 6：响应式、可访问性与发布（3–4 天）

- 1024px 以下默认收起右栏；768px 以下左栏变抽屉。
- 完成焦点环、ARIA、Tab 顺序、Esc 关闭和屏幕阅读器标签。
- 检查对比度、文本缩放、减少动画模式。
- 执行构建、单元测试、关键流程 E2E 和回归截图。
- 增加新旧 UI 开关，灰度观察后移除旧版。

验收：关键路径无阻断缺陷，生产构建无 error，主要页面满足 WCAG AA 对比度。

## 5. 推荐目录调整

```text
frontend/src/
  app/
    AppShell.js
    routes.js
  components/
    ui/                 # Button、IconButton、Dialog、Tooltip、EmptyState
    navigation/         # 左侧导航和历史列表
    conversation/       # 消息流、消息、输入器
    context/            # 情绪、记忆、建议右栏
  features/
    chat/
    history/
    memory/
    skills/
    personalization/
  hooks/
  services/
  styles/
    tokens.css
    globals.css
```

## 6. 技术策略

- 保留 React 18、styled-components、Framer Motion、Lucide React。
- 第一阶段不引入大型 UI 框架，避免视觉被框架默认样式绑架。
- 将服务端状态与纯 UI 状态分开；短期继续使用 hooks，复杂后再评估 TanStack Query。
- SSE 封装为明确状态机：`idle → connecting → streaming → completed/error/cancelled`。
- CSS 变量必须提供默认值，并在浅色、深色主题分别声明完整集合。

## 7. 交付节奏与风险

建议 4–6 周完成一个可发布版本，每个 Phase 独立提交并可验收。

主要风险：

- 现有 `App.js` 同时管理多个 session 状态，迁移时容易出现会话串线。
- 后端历史记录更新时间与流式保存时机需要同步调整。
- 当前部分组件使用未定义 CSS 变量，需要在设计系统阶段集中清理。
- 移动端现状仅隐藏侧栏，新架构必须补充抽屉导航。

## 8. 完成定义

- PRD 中 P0、P1 功能全部验收。
- 新旧关键能力无回退：聊天、历史、记忆、技能、个性化、附件、反馈。
- 1440/1024/768/390 四档布局通过视觉回归。
- 浅色和深色主题无透明面板、不可读文字或焦点丢失。
- 前端生产构建通过，关键 E2E 流程通过。
