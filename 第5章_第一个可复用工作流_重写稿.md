# 第 5 章 第一个可复用工作流：以 emotional_chat 为例

前四章已经完成了必要铺垫：认识 Codex 的工作方式，理解智能体执行机制，并看清不同入口适合承担什么任务。从这一章开始，我们进入更实际的问题：怎样让 Codex 在一个真实仓库里完成一条可验证、可审查、可复用的工作闭环。

本章会以 `emotional_chat` 项目为例。它不是一个玩具仓库，而是一个典型的应用型工程：后端是 FastAPI/Python，前端是 React，项目里有 RAG 知识库、Agent 模块、记忆系统、数据库迁移、Docker Compose、Makefile、根目录脚本测试，以及 `backend/tests` 下的 pytest 测试。这样的仓库足够真实，也足够容易暴露第一次使用 Codex 时最常见的问题：入口多、模块多、验证路径多，稍不收束就会变成全仓漫游。

第一次闭环不应该从大任务开始。更合适的起点，是一个边界清楚的小修改：先说明目标和范围，再让 Codex 收集上下文、完成最小改动、运行验证，最后给出可以交接的结果说明。本章不重复安装步骤，也不写成模型推荐清单；入口确认、仓库脚本、最小变更、团队复用、成本和模型选择，都会围绕同一个目标展开：让“我会用 Codex”变成“我能让 Codex 稳定完成一次工程任务”。

本章会回答五个问题：

- 第一次进入 `emotional_chat` 时，怎样确认入口、目录和仓库状态。
- 怎样把仓库准备到至少有一条局部可验证路径。
- 怎样选择一个足够小、可回退、可审查的第一次任务。
- 怎样把个人闭环升级成团队可以复用的轻量流水线。
- 怎样理解成本、模型、速度与任务设计之间的关系。

> 注意：第 5 章的重点不是“第一次就做大”，而是“第一次就走完整”。只有一个小闭环能被稳定重复，后面的 `AGENTS.md`、Rules、Hooks、Skills 和团队流程才有东西可以沉淀。

## 5.1 先把起点收住：入口、目录与默认边界

第三章已经讲过如何安装和启动 Codex。到了第五章，问题不再是“工具怎么装起来”，而是“它能不能在这个仓库里开始一项真实任务”。`emotional_chat` 这种仓库尤其需要先做确认：它既有 `backend/`，也有 `frontend/`；既有根目录测试脚本，也有 `backend/tests/unit/` 和 `backend/tests/integration/`；既有本地运行脚本，也有 Docker Compose 和数据库迁移。入口一多，第一次任务就不能靠感觉起手。

这一节只做最小确认：当前入口是否可用，工作目录是否正确，仓库状态是否清楚，第一次任务需要的默认边界是否已经说清。只要这几件事没确认，就不要急着让 Codex 进入修改阶段。

### 5.1.1 先确认主入口和工作目录

同一个 Codex，可以从 CLI、IDE、Web 或其他入口进入。但第一次闭环不适合同时切换多个入口。入口越多，变量越多；变量越多，越难判断问题到底出在任务定义、仓库环境，还是工具使用方式上。

开始真实任务前，先确认三件事：本轮任务主要从哪个入口发起；Codex 当前看到的是不是 `emotional_chat` 项目根目录；任务完成后，你准备在哪里审查 diff、验证结果和交付摘要。确认主入口以后，不要急着让 Codex 修改代码。先做一次只读自检：确认自己站在正确目录、仓库状态可控，并且能看到项目的主要文件。

代码清单 5-1 给出首次只读验证的基础自检命令。它的作用不是分析业务，而是确认后续所有搜索、验证和 diff 审查都发生在正确工作面上。

**代码清单 5-1：首次只读验证的基础自检命令**

```bash
pwd
git status --short
git ls-files | sed -n '1,60p'
```

这几条命令分别回答三个问题：`pwd` 确认当前目录；`git status --short` 确认是否已有未提交改动；`git ls-files` 列出 Git 正在跟踪的文件，帮助你快速判断项目结构。第一次闭环里，这些信息比“马上开始改”更重要。

如果你看到根目录里有 `README.md`、`Makefile`、`pyproject.toml`、`requirements.txt`、`backend/`、`frontend/`、`docker-compose.yml`，说明当前工作面大概率正确。如果你只看到 `frontend/src` 或某个子目录，就要先回到项目根目录，否则后面的搜索、测试和 diff 审查都会变得混乱。

如果 `git ls-files` 没有输出，通常说明当前目录不是 Git 仓库，或者文件还没有被 Git 跟踪。这时不要卡在 Git 命令上，可以先用代码清单 5-2 做一次浅层目录观察。它是代码清单 5-1 的备用版本，用来在非 Git 环境里完成同样的结构确认。

**代码清单 5-2：非 Git 环境下的目录结构自检命令**

```bash
find . -maxdepth 2 -type f | sed -n '1,60p'
```

### 5.1.2 先只读跑通，再允许修改

第三章的安装完成，只说明工具已经可启动；第五章的起手检查，要确认它已经能在当前仓库里安全工作。代码清单 5-1 已经完成第一层确认：当前目录对不对、仓库状态是否干净、项目大致有哪些文件。接下来才进入第二层确认：这个仓库里，和本次任务有关的代码入口、测试入口和脚本入口分别在哪里。

对 `emotional_chat`，只读跑通可以分三步。第一步承接上一小节，确认位置没有错；第二步收窄结构，找出后端、前端和测试入口；第三步确认最小验证命令。表 5-1 把这三步压成一个起手检查表。它不是完整开发流程，而是第一次交给 Codex 修改前的防偏动作：先把位置、结构和验证三个变量压住，再让 Agent 进入修改阶段。

**表 5-1：`emotional_chat` 首次只读跑通检查表**

| 阶段 | 要确认的事 | 在 `emotional_chat` 中看什么 | 不要急着做什么 |
| --- | --- | --- | --- |
| 位置 | 当前目录和仓库状态 | `pwd`、`git status --short` | 直接让 Agent 修改 |
| 结构 | 前后端和测试入口 | `backend/`、`frontend/`、`backend/tests/`、根目录 `test_*.py` | 全仓漫游式搜索 |
| 验证 | 最小验证命令 | `python -m pytest backend/tests/unit/test_core.py` 或目标脚本 | 默认整仓回归 |

表 5-1 说明了要看什么，代码清单 5-3 则把它落成一组可以直接执行的只读命令。注意，这组命令不再重复 `pwd` 和 `git status --short`，因为它们已经在代码清单 5-1 中完成；这里的重点是继续定位项目结构、脚本入口和测试入口。

**代码清单 5-3：定位任务入口和验证入口的只读命令**

```bash
git ls-files backend frontend Makefile pyproject.toml README.md frontend/package.json | sed -n '1,100p'
grep -nE "pytest|test|uvicorn|react-scripts|make" README.md Makefile pyproject.toml frontend/package.json 2>/dev/null | sed -n '1,40p'
find backend/tests -type f -name 'test_*.py' 2>/dev/null | sed -n '1,40p'
find . -maxdepth 1 -type f -name 'test_*.py' | sed -n '1,40p'
```

这组命令的价值不在于“全面”，而在于让第一次任务的工作面变窄。第一行列出本轮最可能相关的工程入口；第二行从说明文件和脚本文件里找测试、启动、构建相关命令；后两行分别列出 `backend/tests` 下的 pytest 文件和根目录里的脚本式测试。读完这几行输出，你应该能回答：后端代码在哪里，前端代码在哪里，测试在哪里，最小验证命令可能是什么。

如果当前目录不是 Git 仓库，或者项目文件没有被 Git 跟踪，可以把第一行的 `git ls-files` 换成代码清单 5-4。它只列出后端和前端两个主要目录，避免第一次勘察时把数据库、日志、上传文件和生成物一并卷进来。

**代码清单 5-4：非 Git 环境下定位前后端文件**

```bash
find backend frontend -type f | sed -n '1,80p'
```

### 5.1.3 把默认信息分成三层

只读跑通以后，还要回答一个更容易被忽略的问题：哪些信息是你的个人偏好，哪些是这个项目的默认工作方式，哪些已经属于团队共享规则。把这三层排清楚以后，很多“到底该把这条规则写在哪里”的争论会自然消失。

个人默认解决的是你自己的工作习惯，比如喜欢哪种输出风格、是否倾向先给计划、默认使用哪个入口做主工作台。项目默认解决的是“到了这个仓库里，Agent 应该怎样工作”，例如启动命令、测试最小路径、禁改目录和最小变更标准。团队默认则比项目默认再高一层，关心提交前至少跑哪些验证、哪些危险操作必须审批、变更摘要需要包含哪些字段、失败时如何回退。

表 5-2 展示了三层配置在 `emotional_chat` 里分别对应什么。读这张表时，关键不是把所有规则一次写完，而是先别把个人口味误写成项目规则，也别把项目说明误当成自动执法系统。

**表 5-2：个人默认、项目默认与团队默认的分工**

| 默认层 | 典型内容 | 在 `emotional_chat` 中的例子 | 常见误用 |
| --- | --- | --- | --- |
| 个人默认 | 输出偏好、是否先给计划、摘要格式 | 你要求 Codex 每次先列相关文件和验证命令 | 把个人口味写成项目规则 |
| 项目默认 | 目录职责、测试入口、禁改区域 | `backend/core/` 是基础能力，`frontend/` 是 React UI，`chroma_db/` 和数据库文件不随手改 | 只口头说明，不落文档 |
| 团队默认 | 审批点、PR 模板、危险操作边界 | 数据库迁移、依赖升级、删除数据文件必须先确认 | 把文档当成自动执法系统 |

第一次闭环里，最应该优先明确的是项目默认：本次任务在哪个目录，改完跑什么测试，哪些路径不要碰。比如第一次任务如果选在 `backend/core/utils`，就不要只说“帮我看一下后端工具函数”。更稳的写法，是把范围、优先阅读文件、禁改路径和验证命令一次性交代清楚。代码清单 5-5 给出的就是一个最小任务边界块，它可以直接作为交给 Codex 的开场说明。

**代码清单 5-5：后端核心工具函数的最小任务边界块**

```markdown
本轮只处理后端核心工具函数。
优先查看：
- backend/core/utils/validators.py
- backend/tests/unit/test_core.py

不要改：
- frontend/
- chroma_db/
- emotional_chat.db
- alembic/
- docker-compose.yml

验证：
- python -m pytest backend/tests/unit/test_core.py
```

这段说明看起来保守，但它正是第一次闭环需要的东西。范围越清楚，Codex 越容易把能力用在正确位置。

## 5.2 让仓库支撑 Agent：readiness、脚本与局部验证

如果说上一节解决的是“本轮任务从哪里起手”，这一节解决的就是“这个仓库能不能支撑 Agent 稳定工作”。`emotional_chat` 是一个很适合观察仓库 readiness 的例子。它有清晰的业务目标和丰富模块，但也有现实项目常见的复杂度：Python 依赖较多，前端单独管理依赖，RAG 和多模态能力可能需要额外环境，数据库和向量库文件也会影响本地运行。

这意味着第一次闭环不能默认整仓跑通，而要先找一条局部可验证路径。

### 5.2.1 readiness 不是漂亮，而是能执行

所谓仓库 readiness，不是指仓库“写得很漂亮”，而是指它至少满足一个外部执行者可以理解、可以修改、可以验证的最低条件。对 Codex 来说，这个最低条件比对人类新人还重要，因为它不会像团队老人那样自动补全上下文。

在 `emotional_chat` 中，可以先用五个维度判断仓库是否 ready。表 5-3 的第二列不是要求你立刻读完这些文件，而是告诉你应该去哪里确认事实；第三列列出第一次闭环最容易踩到的风险；第四列则给出最小补救动作。只要某一行的风险信号已经出现，就不要急着扩大任务范围，先把这一行补到可执行状态。

**表 5-3：`emotional_chat` 仓库 readiness 轻量检查表**

| 检查项 | 在本项目中看什么 | 风险信号 | 补救动作 |
| --- | --- | --- | --- |
| 依赖安装 | `requirements.txt`、`pyproject.toml`、`frontend/package.json` | Python、Node、系统依赖混在一起没人说明 | 先写清后端/前端分别怎么装 |
| 最小测试 | `backend/tests/unit/test_core.py`、根目录 `test_*.py` | 只能跑全量，且依赖外部服务 | 先选一个离线单元测试 |
| 运行入口 | `Makefile`、`run_backend.py`、`frontend/package.json` | 启动方式散落在 README 和脚本里 | 统一列出常用命令 |
| 目录说明 | `backend/`、`frontend/`、`knowledge_base/`、`alembic/` | 旧模块、新模块、生成物混在一起 | 给 Agent 一份目录边界说明 |
| 样例数据 / 环境样板 | `config.env.example`、`env_example.txt` | 真实密钥或本机路径被误用 | 只使用样板文件，不读秘密 |

这张表的价值不在于形式化，而在于它能帮你承认一件事：第一次失败，很多时候不是任务太难，而是仓库没有把执行前提交代清楚。

不少人第一次用 Codex，会把注意力放在怎么提问更厉害。真正更值得先做的，往往是把仓库里的关键脚本准备好。因为脚本是验证锚点，prompt 只是意图表达。代码清单 5-6 不是一份“全部都要跑”的清单，而是一次仓库勘察结果：它告诉你这个项目有哪些可能的运行和验证入口。

**代码清单 5-6：`emotional_chat` 中常见的运行与验证入口**

```bash
make help
make run
make quick-start
make rag-init
make rag-test
python -m pytest backend/tests/unit/test_core.py
cd frontend && npm test
cd frontend && npm run build
```

但第一次闭环不应该默认使用所有入口。`make run` 可能启动后端和知识库流程；`rag-test` 需要后端服务已经运行；前端测试可能进入 watch 模式；全量依赖安装也可能很慢。第一次闭环要做的，是从这些入口里选一条最小、离线、可重复的验证路径。

### 5.2.2 先选局部验证，再扩大工作面

对于后端核心工具函数，代码清单 5-7 是更合适的第一条验证命令：它不启动完整服务，不依赖前端，也不需要外部模型。

**代码清单 5-7：后端核心单元测试的最小验证命令**

```bash
python -m pytest backend/tests/unit/test_core.py
```

如果只是修改 `backend/core/utils/validators.py` 或 `backend/core/utils/formatters.py`，这条命令就比启动完整服务更合适。真实仓库里它也可能先暴露 readiness 问题，例如测试收集阶段发现某个导入路径已经断裂。这个结果并不是坏事，它说明你找到了第一条需要修通的验证链路。

进入 `emotional_chat` 后，可以先让 Codex 回答这些问题：后端入口在哪里，前端入口在哪里，最小单元测试在哪里，哪些根目录脚本是测试脚本，哪些路径不应该在第一次任务里碰。代码清单 5-8 把这些问题压成一段短块。它不是项目总说明，而是本轮任务的工作面说明。

**代码清单 5-8：第一次任务的 readiness 摘要块**

```markdown
仓库 readiness：
- 后端：FastAPI/Python，主要代码在 backend/
- 前端：React，主要代码在 frontend/src/
- 后端最小测试：python -m pytest backend/tests/unit/test_core.py
- 前端构建验证：cd frontend && npm run build
- 本轮不碰：chroma_db/、uploads/、emotional_chat.db、alembic/、Docker 配置
```

readiness 的重点不是把仓库整理成教科书示例，而是先建立一条最小可执行路径。对 `emotional_chat` 来说，第一次闭环完全可以只在一个局部面完成。代码清单 5-9 把局部 ready 的边界写成可直接交给 Codex 的任务约束，重点是把“只处理什么”和“不要处理什么”同时说清。

**代码清单 5-9：局部 ready 的任务边界示例**

```markdown
本轮任务范围：
- 只处理 backend/core/utils/formatters.py
- 只对照 backend/core/interfaces.py 确认类型定义
- 验证只跑 python -m pytest backend/tests/unit/test_core.py
- 不处理前端、数据库、RAG、多模态和部署配置
- 若发现需要跨模块修改，先停下来汇报
```

这种局部 ready 的策略看起来保守，却特别适合第一次闭环。因为它把原本分散在整个仓库里的不确定性，压缩到了一个能解释清楚的小区域里。

### 5.2.3 常见失败，先归因再加码

在 `emotional_chat` 这种仓库里，第一次失败很容易被误解成“模型不行”。但很多失败其实来自流程和环境：没先确认当前目录，导致命令在错误位置执行；一上来让 Codex 读完整个 `backend/` 和 `frontend/`，上下文过厚；修改后只看代码，没有跑对应测试；把 RAG、数据库、多模态能力一起卷进一个小任务；对前端任务默认跑交互式 `npm test`，结果卡在 watch 模式；把 `chroma_db/`、数据库文件、上传目录或日志当成普通源代码处理。

这些问题之所以致命，是因为它们会让 Agent 在“理解错”和“执行不了”之间来回摆动。你表面上看到的是结果不稳定，底层其实是验证链路没有建立。因此，第一次准备不追求完美仓库，只追求一条可解释的局部执行路径。

## 5.3 走完整条链路：一次最小变更的闭环

第一次工作流最容易犯的错，是把目标设成“做出一个看起来厉害的结果”。本节的建议刚好相反：故意选一个小题，把整个闭环走完整。在 `emotional_chat` 中，一个适合第一次闭环的小题是：先修通后端核心单元测试的收集错误，或者给核心工具函数补一个边界测试。它不需要启动完整聊天服务，不需要真实数据库，不需要外部模型，也不需要前端联调。

一个能稳定重复的小闭环，比一次偶然成功的大闭环更有价值。

### 5.3.1 先选小题，不要选大题

什么算合适的第一次任务？最好满足四个条件：问题边界清楚、影响范围局部、验证命令明确、失败后容易回退。放到 `emotional_chat` 中，合适的第一次任务通常有一个共同点：能在局部目录里完成，并且有明确的最小验证。表 5-4 列出几类可以作为第一次闭环起点的任务，用来帮助你判断“这个任务是不是足够小”。

**表 5-4：适合作为第一次闭环的 `emotional_chat` 任务**

| 任务 | 为什么适合第一次闭环 | 最小验证 |
| --- | --- | --- |
| 修复 `formatters.py` 的导入路径 | 测试收集阶段即可验证 | `python -m pytest backend/tests/unit/test_core.py` |
| 给 `validate_email` 补边界测试 | 只涉及验证器和单元测试 | `python -m pytest backend/tests/unit/test_core.py` |
| 给 `sanitize_text` 补控制字符测试 | 不需要外部服务 | `python -m pytest backend/tests/unit/test_core.py` |
| 调整 `format_error` 的测试覆盖 | 只涉及格式化函数 | `python -m pytest backend/tests/unit/test_core.py` |
| 修 README 中一条过期命令说明 | 风险低，diff 容易审查 | 人工审阅 + 命令核对 |

不适合作为第一次闭环的任务包括：重构 Agent Core、改造六层记忆系统、调整 RAG 检索策略、改数据库迁移、前后端一起改聊天主链路、升级多模态依赖。这些任务不是不能交给 Codex，而是不适合拿来验证第一次工作流。题目一旦过大，你就无法分辨失败到底来自任务定义、上下文不足、验证设计不清，还是模型真的推理错了。

示例 5-1 选用一个足够小的真实问题：`backend/core/utils/formatters.py` 里引用了错误的 `EmotionResult` 路径。这个例子的目的不是展示复杂能力，而是展示一个完整闭环怎样被收束到单文件修改、单条验证和可审查摘要里。

**示例 5-1：修复 `EmotionResult` 导入路径的最小闭环**

假设你第一次运行后端核心单元测试时，测试还没有真正执行就失败了。错误发生在收集阶段：`backend/core/utils/formatters.py` 里写了 `from .interfaces import EmotionResult`，但当前项目里实际存在的是 `backend/core/interfaces.py`，并没有 `backend/core/utils/interfaces.py`。这类问题很适合作为第一次闭环：范围小、原因清楚、验证明确，而且不需要启动完整服务。

代码清单 5-10 是示例 5-1 的任务交接块。它不是为了写得“正式”，而是为了把目标、范围、禁区、验证和输出格式一次性说清楚，让 Codex 不必在执行中猜你的边界。

**代码清单 5-10：修复导入路径问题的任务交接块**

```markdown
目标：
- 修复 `backend/core/utils/formatters.py` 中错误的 `EmotionResult` 导入路径，让后端核心单元测试可以正常收集。

范围：
- 优先查看 backend/core/utils/formatters.py
- 对照 backend/core/interfaces.py
- 必要时查看 backend/tests/unit/test_core.py

不要做：
- 不要改前端
- 不要改数据库、RAG、多模态模块
- 不要重构 formatters.py
- 不要调整无关测试

先做：
- 先确认 `EmotionResult` 实际定义位置
- 先汇报修改计划，再动手

验证：
- 运行 python -m pytest backend/tests/unit/test_core.py

输出：
- 改了什么
- 为什么这么改
- 已跑命令和结果
- 剩余风险
```

如果这个导入问题已经修好，再选第二个小任务，例如给 `validate_email` 补邮箱长度边界测试，也可以沿用同一套任务块。

### 5.3.2 用人机双泳道管理执行过程

可以把第一次闭环理解成一张很简单的双泳道图：人负责给边界、做判断、审结果；Agent 负责搜集上下文、执行修改、运行验证、整理说明。如果把这张流程图再往前推一步，第一次闭环真正应该产出的，不只是代码改动，而是一串可审查的中间产物。

表 5-5 把这条泳道落到 `emotional_chat` 的导入问题案例里：第一列说明闭环阶段，第二列说明这个阶段主要由谁推动，第三列把动作落到本项目的具体任务，第四列说明这个阶段结束时最好留下什么证据。读这张表时，不要只看“谁做什么”，更要看每一步有没有留下足够让人复盘和接手的产物。

**表 5-5：最小变更闭环的人机分工与产物**

| 阶段 | 谁主导 | 在 `emotional_chat` 中怎么做 | 最好留下什么产物 |
| --- | --- | --- | --- |
| 任务定义 | 人 | 指定只修一个后端核心导入问题 | 边界清楚的任务描述 |
| 上下文收集 | Agent | 读取 `formatters.py`、`interfaces.py` 和 `test_core.py` | 文件清单、修改计划 |
| 修改执行 | Agent | 只修正导入路径，不做顺手重构 | 最小 diff |
| 验证执行 | Agent | 跑目标 pytest 命令 | 命令与结果 |
| 提交前判断 | 人 | 看 diff 是否越界、验证是否对应目标 | 人工审查结论 |

图 5-1 把表 5-5 的关系画成双泳道：左侧是人的判断责任，右侧是 Agent 的执行责任。配套 drawio 文件为 `第5章_人机双泳道闭环.drawio`。

**图 5-1：第一次最小变更的人机双泳道闭环**

```text
人：定义目标 → 限制范围 → 审查 diff 和验证
                         ↘
Agent：            阅读相关代码和测试 → 给出修改计划 → 执行最小修改 → 运行验证 → 整理结果摘要
```

一个对应的最小命令链，通常也不需要很复杂。代码清单 5-11 把勘察、阅读、验证和 diff 审查串成一条线，每一步都服务于同一个目标：确认并修复 `EmotionResult` 的导入路径。

**代码清单 5-11：导入路径修复任务的最小命令链**

```bash
grep -R -nE "from \\.interfaces|EmotionResult" backend/core
sed -n '1,120p' backend/core/utils/formatters.py
sed -n '1,80p' backend/core/interfaces.py
python -m pytest backend/tests/unit/test_core.py
git diff -- backend/core/utils/formatters.py
```

这里每一步都对应一个明确目的：先锁文件，再读实现，再读测试，再跑最小验证，最后只审本次预期范围内的 diff。第一次闭环最怕的不是步骤少，而是每一步都没有明确目的。

### 5.3.3 提交前看链路，不只看代码量

很多人看到 Agent 改得少，会觉得“不够值”。第一次最重要的指标不是改了多少行，而是链路是否完整：你是否明确了需求、缩小了范围、看到了它如何理解上下文、拿到了真正跑过的验证和可用于提交说明的摘要。

在 `emotional_chat` 的第一次闭环里，一个合格的阶段性摘要应该包含表 5-6 里的四个字段。它们分别对应理解、范围、验证和风险，能让人及时判断 Agent 是否已经走偏。

**表 5-6：阶段性摘要的核心字段**

| 字段 | 要回答什么 | 示例 |
| --- | --- | --- |
| 当前理解 | 问题到底是什么 | `formatters.py` 引用了不存在的 `backend.core.utils.interfaces` |
| 拟改范围 | 计划改哪些文件，不改哪些文件 | 只改 `backend/core/utils/formatters.py` |
| 验证计划 | 改完准备跑什么 | `python -m pytest backend/tests/unit/test_core.py` |
| 预估风险 | 哪些点仍有不确定性 | 未跑整仓测试，不覆盖前端和 API |

即便是最小变更，也不应该把“修改完成”直接等同于“可以提交”。表 5-7 的作用是把“可以提交”从主观感觉变成可核对条件：范围没有越界，验证能对应目标，摘要能被别人接住，失败时也能回退。

**表 5-7：提交前的最小人工审查表**

| 检查项 | 通过标准 | 在本项目中的例子 |
| --- | --- | --- |
| 修改范围 | diff 只落在预期文件 | 只改 `backend/core/utils/formatters.py` |
| 验证覆盖 | 已跑命令能直接对应任务目标 | 跑过 `python -m pytest backend/tests/unit/test_core.py` |
| 摘要质量 | 没参与任务的人也能看懂 | 写清修的是 `EmotionResult` 导入路径 |
| 回退可行性 | 失败时知道怎么撤回 | 单文件导入改动，可直接回退 |

一个够用的提交前命令链通常是代码清单 5-12。它先看工作区状态，再看本次预期文件的 diff，最后重新跑目标验证，避免“刚才通过”与“当前状态”不一致。

**代码清单 5-12：提交前的最小复核命令**

```bash
git status --short
git diff -- backend/core/utils/formatters.py
python -m pytest backend/tests/unit/test_core.py
```

如果这些检查都成立，提交说明就不应该只是“update tests”。代码清单 5-13 展示的是一条可读的提交说明结构：标题说明动作，正文说明修改点和验证。

**代码清单 5-13：导入路径修复的提交说明示例**

```text
fix: correct EmotionResult import in core formatters

- import EmotionResult from backend.core.interfaces
- keep formatter behavior unchanged
- verified with python -m pytest backend/tests/unit/test_core.py
```

如果团队走 PR 流程，PR 摘要也可以沿用同一套字段。代码清单 5-14 把任务目标、改动范围、已跑验证和剩余风险拆开写，方便 reviewer 快速判断这次变更是否足够小、验证是否足够贴合。

**代码清单 5-14：最小变更的 PR 摘要示例**

```markdown
## 任务目标
修复 `backend/core/utils/formatters.py` 中错误的 `EmotionResult` 导入路径。

## 改动范围
- `backend/core/utils/formatters.py`

## 已跑验证
- `python -m pytest backend/tests/unit/test_core.py`

## 剩余风险
未跑整仓测试；本次只验证后端核心单元测试入口。
```

如果任务没有一次通过，也不应立刻把失败视为“第一次闭环失败”。表 5-8 用来帮助你判断下一步该修 prompt、修仓库说明、修验证链路，还是修实现本身。它的重点是“先归因，再动作”：同样是失败，找错文件通常说明上下文收集不够；测试没过才更可能是实现或测试理解问题；命令跑不动则往往是仓库 readiness 没准备好；diff 过大则说明任务边界没有收住。

**表 5-8：第一次闭环中的常见失败类型与处理动作**

| 失败类型 | 更像什么问题 | 下一步动作 |
| --- | --- | --- |
| 找错文件 | 上下文收集问题 | 缩范围，重读 `formatters.py`、`interfaces.py` 和测试文件 |
| 测试没过 | 实现或测试理解问题 | 看 pytest 报错，只修当前失败原因 |
| 跑不动命令 | 仓库 readiness 问题 | 先补依赖或测试说明 |
| diff 过大 | 任务边界问题 | 重新定义最小修改面 |

只有把失败拆类型，你才会真正知道下次该优化 prompt、优化仓库，还是优化验证链路。

## 5.4 从个人闭环到团队资产：复用、成本与模型选择

当第一次闭环跑通后，接下来要防止三类问题：个人经验无法复用，自动化和团队协作开始失控，成本与模型选择被误解成单纯的价格问题。个人会用 Codex，和团队能用 Codex，中间隔着的不是人数，而是沉淀程度。个人闭环之所以能跑通，往往依赖大量隐性经验：你知道该从哪个目录开始看，知道哪个脚本最值得先跑，知道什么问题不值得交给 Agent，知道改完后要怎样写摘要。这些经验如果只留在脑子里，团队就无法稳定复现你的成功。

### 5.4.1 把个人习惯沉淀成项目资产

从个人到团队，最先要抽出来的不是复杂平台，而是四类最小资产：固定任务模板、固定验证脚本、固定交付格式、固定审批点。表 5-9 的重点不是“团队马上平台化”，而是把个人脑子里的经验变成别人也能复用的项目资产。

**表 5-9：个人习惯到项目资产的转换**

| 资产类型 | 个人阶段通常长什么样 | 项目里应沉淀成什么 |
| --- | --- | --- |
| 任务模板 | 你知道修后端工具函数该先看哪里 | issue / prompt 模板写明目标、范围、验证 |
| 验证脚本 | 你记得要跑哪个 pytest 文件 | README 或 Makefile 增加 `test-core` |
| 交付格式 | 你会口头解释改了什么 | PR 模板固定“改动范围 / 已跑验证 / 剩余风险” |
| 审批点 | 你知道数据库和 RAG 别随便动 | 规则里写明迁移、依赖、数据文件必须先确认 |

两到五人的小团队已经值得做轻量流水线。你们真正需要的，通常只是把高频任务收束成几个稳定套路。表 5-10 把 `emotional_chat` 的高频任务、固定范围和固定验证放在一起，目的是让团队成员不再每次都重新猜“这类改动应该怎么验收”。

**表 5-10：`emotional_chat` 的三条轻量流水线**

| 高频任务 | 固定范围 | 固定验证 |
| --- | --- | --- |
| 后端核心工具函数 | `backend/core/utils/`、`backend/tests/unit/test_core.py` | `python -m pytest backend/tests/unit/test_core.py` |
| Runtime 模块导入检查 | `backend/runtime/`、`test_runtime_imports.py` | `python test_runtime_imports.py` |
| 前端 UI 小改 | `frontend/src/` | `cd frontend && npm run build` |

一个两到五人团队完全可以先从代码清单 5-15 这种最轻的 PR 模板开始。它的价值不在于字段多，而在于让每个人交付结果时都回答同一组问题：目标是什么、改了哪里、怎么验证、还剩什么风险。

**代码清单 5-15：小团队可直接启用的最小 PR 模板**

```markdown
## 任务目标

## 改动范围

## 已跑验证

## 剩余风险

## 是否需要继续人工接手
```

判断一项经验有没有资格进入团队流水线，可以看它是否满足两个条件：高频重复，且可被验证。表 5-11 给出的是优先级建议：先沉淀最常用、最容易验证、最能减少返工的资产。

**表 5-11：从个人习惯到项目资产的优先沉淀项**

| 优先沉淀什么 | 为什么先做这个 |
| --- | --- |
| 后端单元测试入口 | 让后端局部修改有快速验收 |
| 前端构建入口 | 让 UI 小改至少能证明可构建 |
| Runtime 导入检查 | 让模块拆分后不至于导入断裂 |
| 禁改路径说明 | 防止数据、日志、上传文件被误改 |

可以把团队沉淀的成熟度粗分成四级。表 5-12 不是考核表，而是帮助团队判断自己现在处在哪一层，以及下一步最该补什么。对大多数小团队来说，真正有价值的跃迁通常发生在 L1 到 L2：脚本已经有了，但还缺任务模板、交付格式和基本审批点。

**表 5-12：团队工作流沉淀成熟度分级**

| 级别 | 典型特征 | 在 `emotional_chat` 中的表现 |
| --- | --- | --- |
| L0 | 只有个人经验，没有固定脚本 | 每个人都问“改完跑什么” |
| L1 | 有脚本，但没有固定交付格式 | 测试能跑，PR 摘要不稳定 |
| L2 | 有脚本、有模板、有基本审批点 | 常规任务可复用 |
| L3 | 有自动化、交接协议、模块化说明书 | 多人并行和异步接力开始稳定 |

团队流水线的首要目标是稳定复用，而不是自动执行。自动化只是复用成熟后的放大器。对 `emotional_chat` 来说，第一条自动化不需要覆盖 RAG、多模态、数据库和前端全量测试。更现实的做法，是先覆盖低成本、高频、可重复的 smoke 检查。代码清单 5-16 展示的是一条克制的 GitHub Actions smoke check：只在后端核心目录或对应测试变化时触发，只跑后端核心单元测试。

**代码清单 5-16：后端核心 smoke check 的 GitHub Actions 示例**

```yaml
name: backend-core-smoke

on:
  pull_request:
    paths:
      - "backend/core/**"
      - "backend/tests/unit/test_core.py"

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.10"
      - run: pip install -r requirements.txt
      - run: python -m pytest backend/tests/unit/test_core.py
```

这一步也为后面几章铺路：第 6 章的 `AGENTS.md` 可以写清目录、验证入口和禁改路径；第 7 章的 Rules 和 Hooks 可以阻止误改数据库、密钥和生成物；第 8 章的 Skills 可以把“后端核心小改”“前端 UI 小改”“Runtime 导入检查”沉淀成可复用能力。到了这个阶段，Codex 才开始从个人助手变成团队资产。

### 5.4.2 成本不是价格表，而是消耗结构

只要 Codex 开始进入真实工作流，成本问题就一定会出现。但团队最需要的并不是记住一页价格表，而是看懂什么行为在消耗 credits。因为价格会变，套餐会调；唯独相对稳定的是消耗结构。

`emotional_chat` 这种仓库尤其适合说明成本问题：同样是“改一个小 bug”，如果任务边界锁在 `backend/core/utils`，上下文很轻；如果一上来让 Codex 理解后端、前端、RAG、Agent、记忆系统、多模态和部署脚本，上下文就会迅速变厚。表 5-13 不讨论具体价格，而是帮助你识别哪些任务设计会把消耗放大，以及最直接的压缩动作是什么。

**表 5-13：影响 Codex 消耗结构的主要维度**

| 维度 | 高消耗样式 | 更稳的做法 |
| --- | --- | --- |
| 触发次数 | 每个 PR 都跑全仓分析 | 只在相关路径触发 |
| 上下文长度 | 小任务也读完整个后端和前端 | 先锁目录和文件 |
| 输出与重试 | 测试入口不清，反复试命令 | 先给最小验证 |
| 自动化频率 | 定时任务持续扫描全仓 | 给自动化加门槛 |

最直观的成本驱动是上下文长度。一个只看 `validators.py` 和 `test_core.py` 的任务，输入负担很轻；一个同时要求理解 Agent Core、RAG、记忆系统、前端 UI 和部署配置的任务，输入负担会明显上升。这也是为什么仓库越大、任务越模糊，成本越容易失控。上下文不清时，Agent 会被迫去读更多文件、做更多搜索、保留更多中间信息。

输出和验证轮次也会抬高消耗。一次真正的闭环通常不止一轮生成，而是“生成一版修改，再跑验证，再根据结果修一轮，再输出摘要”。如果你没有告诉它最小验证命令，它可能会尝试根目录脚本测试、`backend/tests` 下的 pytest、启动后端服务、前端测试或构建、RAG 相关 curl 测试。更现实的目标不是“把验证变少”，而是把验证设计得更有层级：先跑目标单元测试，再决定是否扩大。

长时任务与自动化触发还有一层常被忽视的成本来源：频率。表 5-14 把 `emotional_chat` 的自动化触发按路径分层，避免所有 PR 都触发同一套重流程。

**表 5-14：按路径分层的自动化触发策略**

| 触发路径 | 适合跑什么 |
| --- | --- |
| `backend/core/**` | 后端核心单元测试 |
| `backend/runtime/**` | Runtime 导入检查 |
| `frontend/src/**` | 前端构建或相关测试 |
| `alembic/**`、`docker-compose.yml` | 先要求人工审查计划 |
| `chroma_db/**`、`uploads/**` | 默认不作为普通代码变更触发 |

不是所有 PR 都值得跑一整套深度分析，不是所有 issue 都值得自动发起长任务。只要触发策略粗放，累计成本就会很快抬升。一旦你能回答“本次花费主要是因为上下文厚、验证宽还是触发多”，成本问题就已经从情绪问题变成工程问题了。

### 5.4.3 模型选择先看风险，再看复杂度

模型选择不是购物问题，而是任务设计问题。真正应该问的不是“哪个模型最好”，而是“这类任务需要多强的理解和多稳的执行，值不值得为此付出更慢或更高的成本”。

一个简单但很实用的判断法是两步走。第一步先看风险：这次任务如果做错，会不会影响线上行为、共享接口、数据库状态、关键安全逻辑，或者让回滚变得困难。第二步再看复杂度：它需要跨多少文件、多少抽象层、多少隐含规则。表 5-15 把风险、复杂度和模型策略放在同一张表里，帮助你先判断任务性质，再决定是否升档。

**表 5-15：风险与复杂度驱动的模型选择矩阵**

| 任务类型 | 风险 | 复杂度 | 优先策略 | 升档信号 |
| --- | --- | --- | --- | --- |
| 读仓库、列目录、找测试 | 低 | 低 | 先快 | 找不到稳定入口 |
| 补 `backend/core/utils` 单元测试 | 低 | 低 | 先轻 | 发现实现和测试不一致 |
| 改前端组件样式或文案 | 低到中 | 中 | 先快后验证 | 涉及多个组件状态 |
| 改 Agent、Memory、RAG 主链路 | 中到高 | 高 | 先稳 | 需要跨模块解释取舍 |
| 改数据库迁移、密钥、部署配置 | 高 | 中到高 | 默认更稳并审批 | 任何误判都会放大成回滚成本 |

更快的模型或更轻的执行档位，通常适合探索、梳理和局部修补。在 `emotional_chat` 中，它适合列出目录结构、定位某个函数或测试文件、总结 `Makefile` 入口、给已有测试补明显边界、修改 README 中一条过期说明。这些任务的共同特征是：上下文相对短，失败代价低，验证路径也比较直接。

更强模型或更高 effort，真正值得用在那些需要较长推理链的任务上：理清 Agent Core、Runtime、Skills 三套结构之间的关系；判断 RAG、记忆系统和上下文组装的边界；设计前后端流式聊天改造方案；评估数据库迁移对已有模型和接口的影响；对多模态依赖、部署方式和性能成本做取舍。在这些场景里，更强模型的价值不一定体现为“写得更多”，而常常体现为“少走弯路”。

最需要避免的是用更贵的模型解决流程问题。不知道本轮到底改后端还是前端，没告诉 Codex 最小验证命令，把 RAG、数据库、前端和 Agent 改造塞进同一任务，仓库里哪些路径是生成物、数据文件、日志文件没有说清，测试失败后不看报错只让 Agent 反复重试，这些都不该靠更贵模型硬扛。

代码清单 5-17 是一条可以直接带走的实践策略。它可以作为每次开新任务前的短提示，帮助你把探索、修改、验证和升级条件固定下来。

**代码清单 5-17：`emotional_chat` 任务执行的默认策略**

```markdown
先让 Codex 只读定位相关文件和测试；
确认范围后，只改一个局部模块；
先跑目标测试；
如果通过，再决定是否扩大验证；
如果涉及数据库、部署、RAG 主链路或 Agent 主链路，先停在方案和风险说明。
```

最终，模型、速度与成本的权衡，从来不是单独做出的。它总是和任务范围、验证设计、团队风险承受能力一起决定。

## 本章小结

第 5 章真正想建立的，不是“我已经会操作 Codex”，而是“我已经能让一次任务沿着一条可解释、可验证、可复用的路径推进”。这两者的差别很大。前者只是工具熟悉度，后者才是工程执行能力。

以 `emotional_chat` 为例，第一次闭环的关键不是让 Codex 一口气理解整个情感陪伴系统，而是让它先在一个局部工作面里完成可靠交付：明确任务边界，读取相关文件，完成最小修改，运行目标验证，输出可交接摘要。

如果把本章压缩成三句话，第一句是：第一次闭环一定要小，而且要能验证。第二句是：仓库 readiness 比 prompt 技巧更重要，脚本和边界不清时，再强的模型也会被迫猜。第三句是：一条个人能稳定重复的最小闭环，才有资格继续沉淀成团队模板、自动化和共享资产。

因此，本章最值得带走的不是某一段命令，而是一个顺序感：先确认入口和目录，再把仓库整理到至少能验证，再选一个最小任务走完整条链路，最后才讨论团队复用、成本控制和模型权衡。

如果要把这一章的收获落实到下一次实际操作里，最简单的做法不是记住所有技巧，而是带着一张极短的 mental checklist 回到仓库：

- 这次任务是否够小？
- 验证入口是否清楚？
- 交付摘要是否能被别人接住？

只要这三项能稳定回答，第 5 章建立的工作流就已经开始起作用。等这三项开始变成习惯，工具本身就不再是主角；真正留下来的，是一条能够被复用的工程闭环。
