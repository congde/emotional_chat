# 第 7 章 Rules、Hooks 与 Approvals：先约束，再放权

如果说 AGENTS.md 解决的是“Agent 该知道什么”，那么 Rules、Hooks 与 Approvals 解决的就是“Agent 被允许做到哪一步、在哪些地方必须停”。在 `emotional_chat` 这样的项目里，这个问题尤其具体：Agent 可以读后端代码、改 React 组件、跑测试、初始化 RAG，也可能误改危机干预材料、重建向量库、碰数据库迁移、泄露本地配置。只要主动执行能力已经进来，护栏就不再是高级玩法，而是基础设施。

这套护栏可以理解成一组持续生效的控制层：Rules 定义长期约束，Hooks 在关键动作前后自动检查、格式化、审计或阻断，Approvals 把高风险跃迁重新交给人确认。再往下一层，还有沙箱、网络开关和环境权限，控制 Agent 技术上到底能执行到哪里。

本章会回答五个问题：

- 为什么工程环境里要先控制，再谈更强主动性。
- 自动格式化、测试、审计、阻断各自解决什么问题。
- 审批点为什么要放在风险跃迁处，而不是平均撒在流程里。
- 沙箱、网络开关、审批模式、分支保护和发布门禁分别控制什么。
- `emotional_chat` 这类带有情感支持、RAG、迁移和运行数据的项目，最小护栏模板应该长什么样。

> 注意：Agent 的主动性不应该直接撞上真实工程环境，而应该先穿过一层层可解释的护栏。越是想把 Codex 用到高频、多人、长任务和高风险仓库里，这条顺序就越不能反。

## 7.1 为什么智能之前要先控制

很多人第一次接触 Codex 的主动执行能力时，最自然的反应是继续追求“更聪明一点”。但进入真实工程环境后，优先级几乎总是反过来的。你真正先需要的，不是让它更会猜，而是让它先别乱猜。因为一旦 Agent 拥有读代码、改文件、跑命令、联网取信息的能力，任何一次默认假设错误，都会比普通聊天模型放大得更快。

以 `emotional_chat` 为例，你可能只想让 Agent 修一个情绪趋势接口的小 bug。它为了让测试通过，可能顺手改服务层；为了让服务能跑，可能调整环境配置；为了修 RAG 报错，可能尝试重建 `chroma_db/`；为了“优化回答质量”，甚至可能改动 `knowledge_base/clinical_guidelines/` 里的专业内容。单看每一步，它都像是在主动解决问题；放在一起看，已经偏离了原任务。问题不是 Agent 不够努力，而是它在缺少边界时，会自然把“把事情做通”理解成“我可以继续扩大动作面”。

### 7.1.1 风险不止是幻觉

很多人把风险只理解成“模型会不会胡说八道”。在工程任务里，更常见也更危险的，还有越界和误触。越界是指任务边界没锁住，Agent 顺着局部合理性把范围越做越大；误触是指动作本身不该直接执行，却因为缺少护栏被自动落地，比如动了危机干预内容、删了运行数据、改了数据库迁移、跑了高风险命令。

表 7-1 把这三类风险放在一起看。它的重点是提醒读者：有些问题可以靠更强模型降低概率，但有些问题应该被控制层直接挡掉。

**表 7-1：工程任务中的三类主要风险**

| 风险类型 | 在 `emotional_chat` 中的典型表现 | 单靠更强模型能解决吗 | 更直接的治理手段 |
| --- | --- | --- | --- |
| 幻觉 | 解释某个 RAG 或记忆逻辑时说得合理，但和真实调用链不符 | 只能降低概率，不能消失 | 验证、复查、对照真实仓库 |
| 越界 | 修接口时顺手改服务层、配置、前端调用和知识库内容 | 很难完全避免 | 明确范围、禁止顺手清理、审批点 |
| 误触 | 删除 `chroma_db/`、改旧迁移、修改危机干预材料 | 不能靠智力兜底 | 沙箱、命令阻断、权限门禁 |

这张表最重要的结论是：很多失败并不是“模型再聪明一点就能解决”，而是应该被更低层的控制机制直接消掉。把该靠制度和工具解决的问题继续交给模型临场判断，本身就是设计错误。

所以，Rules、Hooks、Approvals 并不是因为“不信任 Agent”才存在。更准确的说法是：护栏是在改默认路径。没有护栏时，默认路径往往是“能做就继续做”；有护栏时，默认路径变成“先在许可边界内推进，遇到高风险动作就停下解释”。

这和人类同事协作一样。你让新同事改 `emotional_chat` 的危机检测逻辑，不会只说“你自己看着办”，而会告诉他哪些知识材料需要复核、哪些测试必须跑、哪些目录不能直接改、哪些结论不能由模型生成后直接落地。Agent 进入仓库后也应该获得同样清楚的边界。

### 7.1.2 把控制理解成一条分层栈

AGENTS.md 位于说明层，Rules、Hooks、Approvals 则属于持续生效的控制层。图 7-1 把这条控制栈画成从软到硬的五层：说明层先让 Agent 知道项目，规则层定义长期约束，自动动作层做机械检查，放权层要求人类确认，环境层控制技术边界。

**图 7-1：Codex 在工程仓库中的最小控制栈**

```text
环境层      沙箱 / 网络开关 / 凭证边界
  ↑
放权层      Approvals / owner 审批 / 发布门禁
  ↑
自动动作层  Hooks / 脚本 / CI 检查
  ↑
规则层      Rules / 目录级规则 / 分支策略
  ↑
说明层      AGENTS.md / README / runbook
```

表 7-2 进一步解释这五层分别回答什么问题。后面几节会沿着这条栈展开。

**表 7-2：Codex 的最小控制栈**

| 层 | 回答什么问题 | 在 `emotional_chat` 中的典型载体 |
| --- | --- | --- |
| 说明层 | 这个项目是什么、先看哪里、哪些别动 | `AGENTS.md`、`README.md`、运行说明 |
| 规则层 | 哪些约束始终成立 | Rules、目录级规则、分支策略 |
| 自动动作层 | 某些动作前后自动做什么 | Hooks、pytest、npm test、敏感路径审计 |
| 放权层 | 哪些操作必须人工点头 | Approvals、owner 审批、发布门禁 |
| 环境层 | 技术上到底允许执行到哪一步 | 沙箱、网络开关、凭证边界 |

这五层不是可替代关系。AGENTS.md 可以告诉 Agent “不要直接删除 `chroma_db/`”，但如果命令层没有阻断，危险动作仍然可能发生。Approvals 可以要求“涉及危机干预内容必须人工确认”，但如果工作环境已经暴露了高权限凭证，风险仍然来得太早。说明、规则、自动动作、审批、环境必须一起看，才是完整控制面。

### 7.1.3 先保守，再逐步放权

没有控制时，主动性最容易把“局部合理”一路扩大。Agent 每多做一步，往往都有局部理由：测试没过，所以多改一个文件；命令跑不通，所以调整一下环境；上下文不够，所以继续搜索和联网。问题在于，这些局部理由如果没有外部边界约束，就会把原本很小的任务逐步推向更大的动作面。在日常实践里，失控通常沿着三个方向慢慢发生：

- 任务越做越大：本来修一个接口，最后顺带重构服务层、前端调用和样式。
- 命令越跑越深：本来只需读文件，最后开始装依赖、改配置、访问网络。
- 责任越来越模糊：最后没人说得清哪一步是允许的，哪一步本该人工确认。

表 7-3 把这三类失控方向和首选补救动作对应起来。这样设计护栏时，就不会停留在抽象安全感上。

**表 7-3：三类常见失控方向与首选补救动作**

| 失控方向 | 最先补什么 | 在 `emotional_chat` 中的例子 |
| --- | --- | --- |
| 任务越做越大 | 限定范围、禁止顺手清理、要求先报计划 | 只修 `emotion_analysis` 接口，不顺手改 RAG 和前端 |
| 命令越跑越深 | 沙箱、命令白名单、网络默认关闭 | 不允许自动重建向量库或联网安装依赖 |
| 责任越来越模糊 | 审批点、handoff 模板、操作审计 | 涉及危机干预、迁移、运行数据时必须升级 |

只要你开始这样映射，护栏就不再是“抽象安全感”，而是针对具体失败模式的工程回应。也正因为护栏对应的是失败模式，而不是团队规模，所以它并不只属于大团队。

有人会说，团队仓库当然需要护栏，个人项目没必要这么重。这个判断只对了一半。个人项目确实不需要团队级审批链，但仍然需要最小控制层。原因很简单：只有一个人使用，并不会让 Agent 自动知道任务范围、验证入口和危险目录。你一样会遇到改错目录、跑错命令、顺手重构太多的问题。

表 7-4 给出个人项目的最小护栏集合。它很轻，但足以防住很多第一次使用时的混乱。

**表 7-4：个人项目的最小护栏集合**

| 最小护栏 | 作用 | `emotional_chat` 中的写法 |
| --- | --- | --- |
| 任务范围说明 | 防止一边做一边扩任务 | “只改 `backend/routers/emotion_analysis.py` 和对应测试” |
| 最小验证入口 | 防止“我觉得可以”直接变提交 | “运行相关 pytest，必要时手动检查接口返回” |
| 危险目录禁区 | 防止把运行数据、迁移、密钥区当普通代码改 | “不动 `chroma_db/`、`uploads/`、`config.env`、旧迁移” |
| 先计划后修改 | 防止 Agent 直接在错误理解上开跑 | “先汇报调用链和修改计划，再动手” |

这四项不需要复杂平台，也不需要昂贵配置，只需要把“允许做什么”和“必须先停一下的地方”说清楚。

控制并不意味着永远把 Agent 锁死，但放权应该靠证据，而不是靠感觉。真正合理的顺序是：先用保守边界跑通稳定闭环，再根据任务类型逐步放权。比如一开始只允许 Agent 读代码、改工作区文件、跑相关单测；等几轮任务跑稳后，再开放更多脚本入口；再往后，如果高风险区域已经有成熟审批链和回滚方案，才考虑让自动化接触更深层动作。

表 7-5 给出逐步放权前应该观察到的信号。没有这些信号时，越早放权，越容易把偶然成功误当成可复制能力。

**表 7-5：考虑逐步放权前的信号**

| 可以考虑放权的信号 | 说明什么 |
| --- | --- |
| 同类任务已稳定多轮 | 任务边界和验证路径已比较清楚 |
| 失败类型大多可预测 | 风险已被识别，不再全靠临场猜 |
| 有清楚的回滚和审计 | 即使出错，也能追溯和纠正 |

因此，本节最重要的结论不是“Agent 很危险”，而是“主动性必须被工程化”。你不是先追求一个会自己冲刺的执行体，再想办法事后补救；更稳的路径是先把默认动作面收住，再让能力在边界内持续变强。

## 7.2 把护栏落到动作上：格式化、测试、审计、阻断

前一节讲的是为什么先要有控制，这一节讲的是控制到底落在哪里。很多团队第一次搭护栏时，最容易把所有动作混成一团：格式化也叫检查，测试也叫检查，敏感路径扫描也叫检查，真正应该阻断的动作最后反而没有明确位置。结果就是大家嘴上都说“有流程”，真正出问题时却不知道是哪一层漏了。

更实用的做法，是先把四类动作分开：格式化解决样式一致性，测试解决行为正确性，审计解决风险暴露，阻断负责在越界前把任务停住。它们看上去都像“检查”，但在工程流程里的职责完全不同。

### 7.2.1 四类动作各管什么

表 7-6 先把四类控制动作的差异摆平。后面所有 hook 和 CI 设计，都应该先能归入这四类之一。

**表 7-6：四类控制动作的职责差异**

| 动作类型 | 主要目标 | 适合自动执行吗 | 失败后该怎么处理 |
| --- | --- | --- | --- |
| 格式化 | 让输出风格一致 | 很适合 | 自动修正或要求重新提交 |
| 测试 | 证明改动没破坏目标行为 | 很适合 | 报告失败并停止继续推进 |
| 审计 | 发现敏感改动、风险路径、策略违背 | 适合自动跑，但通常不自动修正 | 输出风险说明，视级别决定是否阻断 |
| 阻断 | 在高风险动作真正发生前停下来 | 必须自动化 | 直接终止，等待人工确认 |

这四类动作最好不要互相代替。格式化通过，并不说明逻辑正确；测试通过，也不说明改动没有碰到敏感区域；审计识别到风险，如果没有阻断策略，也可能只是留下了一条无人处理的告警。

### 7.2.2 触发时机与最小链路

四类动作最常见的设计错误，是先写一堆脚本，再临时找触发点。更稳的顺序恰好相反：先想清楚它应该在哪个时刻触发，再去设计脚本内容。表 7-7 给出一个够用的时机分层。

**表 7-7：四类控制动作的触发时机**

| 时机 | 适合放什么 | `emotional_chat` 中的例子 |
| --- | --- | --- |
| 修改前 | 环境检查、危险命令阻断、只读分析 | 阻断删除 `chroma_db/`、读取调用链 |
| 修改后、本地提交前 | 格式化、最小测试、敏感文件审计 | 跑相关 pytest，审 `alembic/` 和 `knowledge_base/` |
| PR / review 阶段 | 更完整的测试、审计汇总、范围复查 | 前端构建、RAG 检索样例、迁移检查 |
| 合并 / 发布前 | 分支保护、发布门禁、人工审批 | owner 审批危机干预、迁移和生产配置 |

不是所有检查都要放在最早，也不是所有检查都能等到最后。太早会打断节奏，太晚会让错误已经扩散。真正该阻断的东西，应该尽量前置；耗时较长但价值高的检查，可以留到更靠后的阶段。沿着这个顺序看，最适合先自动化的通常是格式化，因为它最机械、最少争议，也最不应该占用人工判断。

格式化最适合自动化，因为它的目标通常最清晰，争议也最少。只要规则稳定，格式化就应该尽量做成低摩擦动作：要么自动修，要么快速报错，不要把它变成人工争论风格偏好的舞台。格式化如果长期靠 reviewer 口头指出，说明系统把最容易自动解决的问题留给了最贵的人力环节。

对 `emotional_chat` 来说，Python 侧可以沿用 `pyproject.toml` 里的 black、ruff 配置，前端则沿用 React 项目自己的 lint/test 入口。格式化规则不是为了显示“流程很严格”，而是让团队不用在空格、导入顺序和局部样式上反复耗费注意力。

格式化解决的是样式一致性，再往下一层才是测试。很多团队一谈自动测试，就会直接跳到“把能跑的都跑一遍”。这对护栏设计来说并不总是最优。测试真正重要的，不是数量，而是能否对应当前任务目标。修一个局部 bug，最先需要的是相关单测或 smoke test；动了共享接口，再扩大到集成测试；涉及发布路径，再补更高等级验证。验证范围应该跟风险走，而不是跟仓库里现成有什么脚本走。

表 7-8 把测试分成三个级别。它适合直接放进 AGENTS.md 或 PR 模板里，提醒 Agent 和 reviewer 不要把所有任务都推给“整仓测试”。

**表 7-8：按风险分层的测试级别**

| 级别 | 适合什么场景 | 目标 |
| --- | --- | --- |
| 最小测试 | 小 bug、小改动、本地快速反馈 | 证明当前任务确实被覆盖 |
| 相关回归 | 共享模块、跨文件改动 | 证明相关区域没有明显连带问题 |
| 发布门禁测试 | 合并前、发布前、高风险改动 | 证明主链路和高风险区仍然稳定 |

在 `emotional_chat` 中，改一个后端接口可以先跑相关 pytest；改聊天界面要检查前端测试或手动路径；改 RAG 或知识库要补最小检索样例；改迁移要说明 upgrade/downgrade 路径。这些验证都比一句“测试通过”更有信息量。

测试回答的是“功能有没有坏”，但它回答不了另一个同样重要的问题：这次改动有没有碰到本不该碰的地方。这就是审计要接住的部分。审计最容易被误解成“安全团队专属动作”。其实对 Codex 工作流来说，审计更广义。它可以是检查敏感目录是否被改动，可以是扫描是否触碰密钥、许可证文件、基础设施配置，也可以是判断这次 diff 是否越过了任务边界。

代码清单 7-1 给出一个面向 `emotional_chat` 的敏感路径审计脚本示例。它不追求覆盖所有风险，而是演示如何把“大家都知道这里敏感”变成机器可执行的检查。

**代码清单 7-1：`emotional_chat` 敏感路径审计与阻断脚本示例**

```bash
#!/usr/bin/env bash
set -euo pipefail

changed_files="$(git diff --name-only HEAD)"

echo "$changed_files" | rg -n '^(alembic/versions/|knowledge_base/clinical_guidelines/|knowledge_base/organization_policy/crisis_intervention_protocol.md|chroma_db/|context_storage/|uploads/|config.env$)' && {
  echo "Sensitive path touched. Human review required."
  exit 1
}

echo "$changed_files" | rg -n '^(pyproject.toml|requirements.txt|frontend/package-lock.json)$' >/dev/null && {
  echo "Dependency or lockfile changed. Explain why and run targeted verification."
}
```

这类脚本本身并不复杂，价值在于它把“这里不能靠猜”变成自动提醒或阻断。不过，审计发现风险之后，并不意味着每一次都要立刻拦停。四类动作里，阻断最重，也最需要克制。不是所有失败都该阻断，否则流程很快会变得僵硬。真正适合阻断的，一般是三类：动作一旦落地，回滚代价很高；风险不是靠后续 review 就能轻松兜住；团队已经明确共识，这类事必须先人工点头。

在 `emotional_chat` 里，典型阻断对象包括：删除运行数据目录、改历史迁移、改危机干预和临床建议内容、提交密钥、重建向量库、改生产配置、批量升级依赖。这些动作即使不是百分之百出错，也不值得拿自动推进去赌。

把格式化、测试、审计、阻断都拆开以后，就能更清楚地看见个人项目和团队项目的差别：它们不是要不要护栏的差别，而是护栏厚度的差别。表 7-9 把这层差别说清楚，方便团队从个人使用逐步升级。

**表 7-9：不同项目形态下的护栏厚度**

| 项目形态 | 格式化 | 测试 | 审计 | 阻断 |
| --- | --- | --- | --- | --- |
| 个人项目 | 本地自动修或快速报错 | 跑最小验证 | 审敏感目录和大 diff | 阻断明显危险命令 |
| 小团队项目 | 本地 + CI 双层覆盖 | PR 前跑相关回归 | 审敏感路径、凭证、共享接口 | 高风险改动要求审批 |
| 高风险团队项目 | 强制统一入口 | 分层门禁测试 | 审计日志、目录、依赖、权限 | 环境层和流程层双阻断 |

团队化不是突然创造出一套新动作，而是在同一套动作上加层级、加审计、加门槛。把这个思路落成流程时，最实用的做法不是先追求完整平台，而是先把格式化、测试、审计、阻断串成一条最小链路。代码清单 7-2 展示的是一种伪配置。它不是某个产品的精确语法，而是帮助读者看清控制顺序。

**代码清单 7-2：格式化、测试、审计、阻断的最小链路**

```yaml
before_edit:
  - block-dangerous-commands
  - require-plan-for-sensitive-paths

after_edit:
  - format-python-and-frontend
  - run-targeted-tests
  - audit-sensitive-paths

before_merge:
  - run-related-regression
  - require-review-on-sensitive-diff

before_release:
  - require-owner-approval
  - enforce-branch-protection
```

最关键的是：每一步都要回答“为什么在这里执行，而不是别处”。只要能回答清楚，具体用 shell、Git hook、CI 还是平台内建能力，反而是第二顺位。

### 7.2.3 别把护栏变成形式检查

护栏系统最容易失效的方式之一，是所有动作都还在跑，但已经没人相信它们真有意义。比如格式化规则形同虚设，测试永远跑整仓所以大家都跳过，审计只会刷一屏噪音，阻断规则又时常误伤正常改动。到这个阶段，系统表面上更严格，实际却更脆。

表 7-10 是一张护栏动作的失效自检表。它适合定期用来复盘 Rules 和 Hooks 有没有变成形式主义。

**表 7-10：护栏动作的失效自检表**

| 自检问题 | 如果答案是否定的，说明什么 |
| --- | --- |
| 这条动作是否真的对应某种失败模式 | 可能只是形式遗留 |
| 失败时能否明确知道谁来处理 | 可能只有告警，没有责任落点 |
| 正常任务会不会频繁被误伤 | 规则可能过粗 |
| 通过之后，信号是否真的提升可信度 | 检查可能已经失去价值 |

格式化、测试、审计、阻断不是为了“看起来有治理”，而是为了让系统在真正该停、该修、该提醒的地方做出稳定反应。

## 7.3 审批点：只在风险跃迁处打断

如果说 Rules 和 Hooks 解决的是“系统平时该怎么自动约束”，那么 Approvals 解决的是“哪些动作必须有人明确拍板”。审批点设计得好，流程会变得可控而不迟缓；设计得不好，就会在两个极端之间来回摇摆：要么什么都自动通过，最后把高风险动作也放进默认路径；要么什么都要点头，结果大家学会了绕开系统。

审批最常见的误区，是按形式设计，而不是按风险设计。比如“所有 PR 都要两个人批准”“所有改动都要 owner 看一眼”这类规则，听上去很严，实际并没有区分风险等级。一个改按钮文案的小改动和一个改危机干预策略的操作，被放进同一审批流程，本身就是流程设计偷懒。

### 7.3.1 审批点应该放在风险跃迁处

最稳的思路，是找到任务从低风险跃迁到高风险的那些节点，把审批放在那里。表 7-11 列出的是常见风险跃迁点。

**表 7-11：审批点应放置的风险跃迁点**

| 风险跃迁点 | 为什么值得审批 | `emotional_chat` 中的例子 |
| --- | --- | --- |
| 从只读分析转成真实写操作 | 动作开始产生持久影响 | 从分析 RAG 报错变成重建向量库 |
| 从普通目录进入敏感目录 | 风险面突然变大 | 从改接口进入 `knowledge_base/clinical_guidelines/` |
| 从本地验证进入合并 / 发布 | 影响范围从个人变成团队 | 从本地修复进入主分支 |
| 从开发环境进入生产环境 | 回滚和事故成本显著上升 | 改生产配置、密钥或部署脚本 |
| 从既有权限边界跨到更高权限 | 错误代价不再局限于当前任务 | 注入凭证、访问真实用户数据 |

审批点一旦跟这些跃迁挂钩，团队就比较容易接受，因为它不是形式负担，而是在真正要冒更大风险之前，要求人重新承担判断责任。

### 7.3.2 四级审批模型通常就够用了

很多团队的审批体系写得很复杂，但真正高频可用的，往往只需要四级。表 7-12 给出的模型足够覆盖个人、小团队和高风险仓库的大多数场景。

**表 7-12：四级审批模型**

| 级别 | 适用动作 | 处理方式 |
| --- | --- | --- |
| A0 自动通过 | 只读分析、低风险局部改动、文档修正 | 直接执行，保留日志 |
| A1 轻确认 | 改共享模块但风险可控、需要额外验证的改动 | 允许执行，但要求结果摘要和 reviewer 看一眼 |
| A2 方案先行 | 涉及敏感目录、配置、共享接口、迁移 | 先给计划，人工确认后再执行 |
| A3 双人或 owner 审批 | 数据删除、权限变更、发布门禁、生产写操作 | 没有明确批准就不执行 |

四级模型的价值，在于它让审批不再是二元的“要/不要”，而是能跟风险厚度对应起来。下一步就要把真实动作放进这个模型里，判断哪些可以自动通过，哪些必须先停。表 7-13 把 `emotional_chat` 常见动作映射到审批级别。它不是精确到每个团队都一样的标准，而是帮助团队形成共同语言。

**表 7-13：常见动作与审批级别映射**

| 动作 | 更适合什么审批级别 | 原因 |
| --- | --- | --- |
| 搜文件、读日志、生成修改计划 | A0 | 只读且可审计 |
| 改局部后端接口并跑相关 pytest | A0 或 A1 | 风险可控，回滚便宜 |
| 改共享服务、公共 schema、前端核心交互 | A1 或 A2 | 影响面扩大，需要额外判断 |
| 改数据库迁移、危机干预内容、密钥配置 | A2 或 A3 | 一旦误判，成本高 |
| 发布、回滚、生产写操作、删除运行数据 | A3 | 必须保留明确人工责任 |

只要大家能说“这次属于 A2，所以先给方案”，审批就从情绪化讨论变成了可执行协议。

### 7.3.3 审批链要能接住责任

审批链真正要接住的是责任，而不是制造更多按钮。审批过少的问题比较直观：高风险动作混入自动路径，最终把事故前置到系统里。审批过多的问题则更隐蔽：系统每一步都要求确认，用户会越来越不耐烦，最后要么机械点通过，要么绕过平台直接手动执行。两种结果都让审批失去意义。

表 7-14 可以用来判断审批系统是不是已经失衡。

**表 7-14：审批系统失衡信号**

| 失衡方向 | 常见信号 | 说明什么 |
| --- | --- | --- |
| 审批过少 | 高风险动作经常“已经做完才被发现” | 节点放得太后或根本没设 |
| 审批过多 | 低风险任务也反复弹窗，大家习惯性点过 | 风险分级太粗 |
| 审批过模糊 | 不同人对是否需要确认理解不一 | 规则没有落到动作级 |
| 审批过形式 | 审批通过后没人能解释为什么同意 | 责任没有真的被接住 |

好的审批系统，不会让人觉得“每次都在被打断”，而会让人清楚知道“只有到这里，才值得打断”。不过，审批如果只停留在 Agent 会话里，仍然不完整。很多团队把审批理解成会话里的一个按钮，但真正稳定的审批链，应该延伸到代码主线和发布路径。否则你在前面要求人工确认，后面却允许任何人直接合并或直接发布，前面的控制就会被旁路。

表 7-15 把审批链在不同层级的落点拆开。它提醒我们：Agent 会话里的审批只是内圈，仓库和发布系统也要接住同一条风险判断。

**表 7-15：审批链在不同层级的落点**

| 位置 | 控什么 |
| --- | --- |
| 会话 / Agent 层 | 某个动作能不能现在执行 |
| 仓库 / 分支层 | 某个结果能不能进入主线 |
| 发布 / 运行层 | 某个版本能不能触达真实环境 |

比如敏感目录改动要求 owner review；主分支要求状态检查全部通过；发布流水线要求额外 release approver。只有这样，审批责任才不会在链路中间消失。不过，审批链能把动作停下来，不代表人一定能做出好判断。确认人还需要知道要批准的动作是什么、为什么做、风险在哪里。审批最怕一句空话，比如“请确认后执行”。这类提示既没有说明风险，也没有说明确认人到底在确认什么。示例 7-1 给出一个更可执行的审批请求写法。

**示例 7-1：可执行的审批请求写法**

```markdown
动作：
- 计划修改 `knowledge_base/organization_policy/crisis_intervention_protocol.md` 中的危机干预升级说明

原因：
- 当前文案没有明确区分“情绪支持建议”和“紧急求助资源”

已知风险：
- 这部分内容会影响高风险用户场景，不能由 Agent 独立判断最终措辞

请确认：
- 是否允许继续提交修改草案
- 是否需要由具备心理健康经验的 reviewer 复核后再合并
```

只要把动作、原因、风险和确认问题补出来，审批就从“要不要点按钮”变成“能否基于事实做判断”。但审批变得可判断之后，还要避免另一个极端：把少数关键节点的确认，扩展成处处都要确认。小团队最常见的错误，是模仿大组织的形式，却没有大组织的配套机制。结果就是每个人都要审批，但其实没人真正负责。对两到五人的团队来说，更现实的做法通常是：把审批集中在少数高风险节点，明确 owner 和 reviewer 的职责，再用模板把确认信息留存下来。

表 7-16 是小团队最值得保留的必审动作清单。除此之外的多数日常改动，可以尽量靠规则、hooks 和 review 去解决，而不是一律上审批。

**表 7-16：小团队的必审动作清单**

| 必审动作 | 为什么 |
| --- | --- |
| 数据删除 / 数据迁移 | 回滚贵，误判成本高 |
| 危机干预 / 临床建议内容 | 影响用户安全和责任边界 |
| 生产配置 / 权限 / 密钥 | 影响范围大 |
| 发布 / 热修 / 紧急回滚 | 动作紧急，最容易跳过流程 |
| 跨模块共享接口改动 | 需要别的 owner 明确接住影响 |

本节可以压缩成一句话：审批点不是为了证明流程存在，而是为了把高风险跃迁重新交回给人做判断。

## 7.4 emotional_chat 的最小护栏模板

到了高风险仓库，护栏就不能再停留在“有一些规则”和“大家尽量小心”这个层级。所谓高风险，不一定只指金融、医疗或生产基础设施，也包括任何一类“动作代价明显高于普通应用代码”的仓库。`emotional_chat` 虽然是情感陪伴项目，但它涉及危机检测、心理健康知识、用户记忆、RAG 索引、数据库迁移和本地运行数据，因此也有不少高风险面。

很多团队的问题不是完全没有护栏，而是护栏太散。说明书写了一点，hook 配了一点，审批口头约定一点，但没人能拿出一份最小可执行模板，说明“这个仓库默认允许什么，不允许什么，遇到哪些动作必须停”。这一节要给的，就是这样一份最小骨架。

### 7.4.1 高风险仓库至少要同时控制四件事

如果只记一个框架，可以记表 7-17 里的四个控制面。缺任何一项，高风险仓库都容易出现假安全。

**表 7-17：高风险仓库的四个核心控制面**

| 控制面 | 你要控制什么 | `emotional_chat` 中的例子 |
| --- | --- | --- |
| 写权限 | 能改哪些文件，哪些目录默认禁写 | `alembic/`、`knowledge_base/`、运行数据目录 |
| 命令权限 | 能跑哪些脚本，哪些命令必须阻断 | 重建向量库、删除上传文件、迁移数据库 |
| 网络与凭证 | 是否允许联网，哪些凭证绝不暴露给默认任务 | `config.env`、模型 API key、数据库凭证 |
| 验证与放行 | 改完要跑什么，进入主线前谁确认 | pytest、前端测试、RAG 样例、owner review |

比如你限制了目录写权限，但允许任意联网和任意脚本执行；或者你有审批，但改完不要求任何针对性的验证。表面上都像“有控制”，实际上关键洞口还开着。确定这四个控制面之后，下一步就是给每个控制面设置保守默认值。表 7-18 给出一份够用的默认策略，它背后的原则很简单：高风险仓库不怕多一步解释，怕的是未经解释就自动深入。

**表 7-18：高风险仓库的保守默认策略**

| 维度 | 默认值 | 放宽条件 |
| --- | --- | --- |
| 文件系统 | 只允许工作区有限写入 | 任务边界清楚，且目标目录非敏感 |
| 敏感目录 | 默认禁写 | owner 明确批准后临时开放 |
| 网络 | 默认关闭 | 任务明确需要拉只读信息，且来源可信 |
| 凭证 | 默认不注入高权限凭证 | 经过审批且有操作审计 |
| 命令执行 | 只允许白名单脚本 | 额外脚本需先说明用途 |
| 合并 / 发布 | 必须经过 review 和门禁 | 仅低风险文档类改动可简化 |

### 7.4.2 护栏模板要同时落在文档、脚本和流程里

只写在文档里不够，只写在脚本里也不够。一个最小高风险模板，最好至少分成三块：项目说明书、自动检查脚本、审批 / 发布门禁。代码清单 7-3、代码清单 7-4 和代码清单 7-5 分别展示这三块在 `emotional_chat` 中可以怎样落地。

**代码清单 7-3：高风险项目的 `AGENTS.md` 骨架**

```markdown
# AGENTS.md

## 仓库性质
- 本仓库涉及情感支持、危机干预、用户记忆、RAG 知识库和数据库迁移，默认按高风险仓库处理。

## 默认工作方式
- 先读代码和脚本，先给计划，再执行修改。
- 默认只做最小变更，不顺手重构。

## 禁写区域
- `alembic/versions/`
- `knowledge_base/clinical_guidelines/`
- `knowledge_base/organization_policy/crisis_intervention_protocol.md`
- `chroma_db/`
- `context_storage/`
- `uploads/`
- `config.env`

## 必须审批的动作
- 数据删除或迁移
- 危机干预、临床建议、用户记忆逻辑改动
- 需要联网拉取外部依赖或访问真实环境
- 需要注入高权限凭证

## 最小验证
- 后端相关测试：`pytest backend/tests`
- RAG 相关检查：`make rag-init` 或最小检索样例
- 前端相关检查：`cd frontend && npm test` 或手动验证路径

## 交付格式
- 变更范围
- 已跑验证
- 风险说明
- 是否需要 owner 继续接手
```

代码清单 7-3 是文档层的高风险默认说明。它让 Agent 一开始就知道哪些区域不能靠猜。

**代码清单 7-4：高风险项目的阻断与审计脚本**

```bash
#!/usr/bin/env bash
set -euo pipefail

changed="$(git diff --name-only HEAD)"

echo "$changed" | rg -n '^(alembic/versions/|knowledge_base/clinical_guidelines/|knowledge_base/organization_policy/crisis_intervention_protocol.md|chroma_db/|context_storage/|uploads/|config.env$)' && {
  echo "Sensitive path touched. Approval required."
  exit 1
}

pytest backend/tests
```

代码清单 7-4 是脚本层的阻断与审计。它把高风险路径从“大家知道”变成“机器会拦”。

**代码清单 7-5：高风险项目的审批与门禁示意**

```yaml
required_reviews:
  sensitive_paths:
    - alembic/versions/**
    - knowledge_base/clinical_guidelines/**
    - knowledge_base/organization_policy/crisis_intervention_protocol.md
    - backend/runtime/**

required_approvals:
  - owner_for_crisis_intervention_change
  - owner_for_database_migration
  - release_approver_for_production

required_checks:
  - backend-tests
  - frontend-tests-or-manual-path
  - rag-minimal-retrieval-check
  - audit-sensitive-paths
```

代码清单 7-5 是流程层的审批与门禁示意。它强调高风险护栏必须跨层落地，不能只寄希望于某一层“大家会记得”。

### 7.4.3 外圈护栏与按任务放权

文档、脚本和门禁解决的是护栏如何落地；到了真正执行时，还要分清几个运行时开关分别管什么。沙箱、网络开关和审批模式经常被混在一起，所以表 7-19 单独拆开它们的分工。

**表 7-19：沙箱、网络和审批模式的分工**

| 开关 | 控制的是什么 | 不能替代什么 |
| --- | --- | --- |
| 沙箱 | 文件系统和进程动作边界 | 不能替代审批判断 |
| 网络开关 | 能否访问外部资源、接口、包源 | 不能替代凭证分级 |
| 审批模式 | 某类动作是否要人工点头 | 不能替代技术性阻断 |

比如网络默认关闭，可以避免 Agent 随手联网取未知信息；但如果高权限凭证已经在环境里暴露，风险仍然存在。再比如审批模式要求“改危机干预内容必须人工确认”，但如果沙箱本身允许任意写敏感目录，危险动作还是暴露得太早。三者一起看，才有完整意义。

不过，这些主要还是 Agent 执行入口的内圈控制。如果一个结果绕过 review、CI、分支保护或发布门禁，前面的护栏仍然可能失效。很多团队把护栏全压在 Agent 入口，忽略了仓库和发布系统本身也需要外圈门禁。更稳的做法是把控制做成同心圆。表 7-20 描述的就是这层结构。

**表 7-20：高风险仓库护栏的同心圆结构**

| 圈层 | 作用 |
| --- | --- |
| 内圈 | Agent 执行时的沙箱、命令限制、审批模式 |
| 中圈 | Review、CI、敏感路径审计 |
| 外圈 | 分支保护、发布门禁、运行环境权限 |

只要外圈缺失，中间任何一层被绕过后，结果都可能直接进主线。对高风险仓库来说，分支保护和发布门禁不是“平台同学的事”，而是护栏系统的一部分。

外圈护栏解决的是“结果能不能进入主线和真实环境”，但日常执行里还有一个更频繁的问题：这一次任务到底该给 Agent 多大动作面。高风险仓库的一个常见误区，是某个人熟悉系统后，就把整体权限一并放大。更稳的做法是按任务类型放权，而不是按“使用者已经很会用了”来放权。因为真正改变风险的，是任务性质，不是操作熟练度。

表 7-21 给出按任务类型切分的默认权限。这样系统不会因为“这次看起来只是个小问题”就偷偷越级。

**表 7-21：按任务类型切分的默认权限**

| 任务类型 | 默认权限 | 备注 |
| --- | --- | --- |
| 只读排查 | 允许读代码、读日志、生成计划 | 默认不联网、不写文件 |
| 局部修复 | 允许工作区修改和最小测试 | 敏感目录仍禁写 |
| 共享接口调整 | 先方案，后执行 | 需要 reviewer 或 owner 接住 |
| 知识库 / 危机干预调整 | 只允许准备草案和风险说明 | 真正落地必须人工复核 |
| 生产相关修复 | 只允许准备方案和验证路径 | 真正落地必须审批 |

按任务类型切分权限之后，护栏系统会稳定很多，但它仍然不可能保证零事故。高风险模板真正追求的，是让三件事同时发生：错误更不容易直接落地；即使落地也更容易被及时发现；一旦发现，责任边界和回滚路径更清楚。这三件事只要成立，团队面对 Agent 主动执行时的心理负担就会显著下降。

所谓“高风险仓库模板”并不神秘。它只是在告诉你：别把安全感建立在“模型大概会判断对”上，而要建立在“就算判断错了，系统也知道该在哪里拦、在哪里问、在哪里停”上。

## 本章小结

第 7 章真正想建立的，不是“要不要更严格”，而是“哪些风险该靠控制层直接消掉，哪些判断才值得继续交给人和 Agent 协作完成”。只要这个顺序建立起来，Rules、Hooks、Approvals 就不再是附加功能，而是 Agent 进入真实研发流程的前提。

如果把本章压缩成三句话，第一句是：工程环境里的主要风险，不只是幻觉，还有越界和误触，所以先要控制默认动作面。第二句是：格式化、测试、审计、阻断各管不同问题，不能被一锅端成“反正都是检查”。第三句是：审批点要放在风险跃迁处，高风险仓库则要把说明、脚本、审批和环境权限一起接成完整护栏。

这一章也在提醒读者一件很容易被忽略的事：真正好的护栏并不一定让系统看起来更重，而是让系统在真正该停的地方稳定地停住，在不该打断的地方尽量少打断。控制如果只是形式，会被绕开；控制如果能对上失败模式，才会被长期保留。
